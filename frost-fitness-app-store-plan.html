Read This First (Jesse)

This document is your execution plan. It is scoped to take Frost Fitness from TestFlight to public App Store launch. Nothing more. New features are out of scope — that's a separate roadmap after launch readiness lands.

Rules of engagement

  1. Work top-to-bottom by phase. Each phase has a gate. Do not start the next phase until the current one is reviewed and merged.
  2. One PR per task. Each task is sized to be a single, reviewable PR. If a task feels like it needs more than ~400 lines of changed code, stop and ask me.
  3. No scope creep. If you find a bug outside the scoped task, write it in projects/frost-fitness/JESSE-NOTES.md and keep moving. We triage later.
  4. When in doubt, ask. Surface the question to Andy or me. Do not invent answers to ambiguity — especially around product behavior. Cost of asking: 30 seconds. Cost of guessing wrong: a rewrite.
  5. Verify before you claim done. Every task has an acceptance checklist. Run through it before opening the PR. Paste results in the PR description.
  6. You own the test for your change. Phase 1 sets up the harness. From Phase 2 onward, every change ships with at least one test unless I explicitly waive it.

Read these before you start, in this order: the repo's README.md and CLAUDE.md; ARCHITECTURE-REVIEW.md (this folder); then this document, end to end, before writing any code.

Phase 1

Foundations (Safety Net)

Goal: Before we change behavior, install the things that will catch us when we break it.
Duration: 3–4 days of Jesse-time.
Exit gate: All four tasks merged, CI green, error boundary verified on a forced crash.

Task 1.1 — Set up the test harness

Why: You're an AI agent. Agent-written code without tests is where silent regressions accumulate. Before we touch anything else, we need a way to catch them.

What to do:

  1. Add vitest (or jest if it integrates more cleanly with Expo/RN — your call, but justify in the PR). Prefer vitest if both work.
  2. Add @testing-library/react-native for component tests.
  3. Create src/__tests__/.
  4. Write tests for the highest-value targets — the core data-layer functions:
    • saveMeal() — happy path + transaction recompute logic
    • recomputeMealTotals() — verifies low/mid/high sums correctly across multiple items
    • getDayTotals() — verifies aggregation across a day boundary
    • updateMealItem() — verifies meal total recomputes after item update
  5. Mock the SQLite layer with an in-memory shim. Do not call real SQLite in tests.
  6. Add npm script: npm test.
  7. Add CI workflow (.github/workflows/test.yml) running npm test on every PR.
Acceptance:
  • npm test runs locally and passes
  • At least 8 test cases across the 4 functions above
  • CI runs on PR and blocks merge on test failure
  • PR description includes test output

Out of scope: E2E tests, snapshot tests, UI tests. Just data-layer unit tests for now.

Task 1.2 — Wire up Sentry properly

Why: Sentry is in package.json but I want to verify it actually captures errors in production. Crash reporting is non-negotiable before public launch.

What to do:

  1. Verify Sentry is initialized at app startup with correct DSN (check app/_layout.tsx).
  2. Confirm enableNative: true so native crashes are captured, not just JS errors.
  3. Add a debug-only "Force Crash" button in Settings → Developer (visible only when __DEV__) that throws an error, and a second button that calls Sentry.captureException(). Use to verify end-to-end.
  4. Set release tagging from app.json version automatically.
  5. Set tracesSampleRate to 0.2 (20%) to start.
  6. Confirm source maps are uploaded on EAS build. If they aren't, fix it.
Acceptance:
  • Force a crash in a TestFlight build → appears in Sentry with symbolicated stack trace within 60s
  • Captured non-fatal exception also appears
  • Release version on Sentry event matches app version
  • Screenshot of Sentry event in PR description

Task 1.3 — Root-level error boundary

Why: Architecture review item #2. Without this, any uncaught React error leaves the user with a white screen and a force-quit.

What to do:

  1. Wrap the root <Stack> (in app/_layout.tsx) in a Sentry.ErrorBoundary.
  2. Fallback UI: app logo, "Something went wrong. We've been notified.", and a "Restart" button calling Updates.reloadAsync() from expo-updates.
  3. The fallback must NOT show a stack trace to the user. It goes to Sentry, not the UI.
  4. Unit test: mount the boundary with a child that throws, verify the fallback renders.
Acceptance:
  • Triggering the force-crash button shows the fallback UI, not a white screen
  • Sentry captures the error
  • Test passes in CI

Task 1.4 — Crash reporting baseline

Why: We need a baseline crash-free rate before changes start, so we can detect regressions.

What to do: Set up a Sentry saved view filtered to current production release showing crash-free user rate. Document baseline in JESSE-NOTES.md. We check against this after each phase.

Acceptance:
  • Baseline metric documented with date

Phase 2

Server-Side Proxy (The Security Blocker)

Goal: Eliminate the bundled API keys. The single most important pre-launch task.
Duration: 4–5 days.
Exit gate: App functions identically with all keys removed from the client bundle. Verified by building a release IPA and grepping for the key prefixes — must not appear.

Task 2.1 — Architecture decision: where the proxy lives

Decision (made by me): Use Cloudflare Workers.

Rationale:

  • We have no other backend yet, so we don't want to stand up a server.
  • Cloudflare Workers: free tier comfortably covers us, sub-50ms cold start, simple wrangler deploy, secrets management built in, streaming support (needed for Claude tool-use loops).
  • AWS Lambda: more ops overhead, cold starts can be rough.
  • Vercel Functions: fine, but we're not in a Next.js project and Vercel's pricing model is less predictable at scale.

If Andy has a strong preference for a different provider, that overrides me. Otherwise: Cloudflare Workers.

Repo layout: New folder proxy/ at repo root. Separate package.json, separate deploy. Documented in proxy/README.md.

Task 2.2 — Build the proxy

  1. Scaffold a Cloudflare Worker in proxy/ via wrangler init.
  2. Implement two routes:
    • POST /v1/vision — proxies to Gemini 2.5 Flash vision endpoint
    • POST /v1/chat — proxies to Claude Sonnet 4.6 messages endpoint (with streaming — the client tool-use loop requires it)
  3. Auth between client and proxy: client sends a shared bearer token (stored as a secret in the Worker, embedded as a public token in the client).
⚠️ Important nuance — read carefully: A shared client token still lives in the bundle. It is not a security primitive by itself. Its purpose: a kill-switch (rotate the token in the Worker → all old clients are locked out) and a rate-limiting handle. The real security comes from per-IP and per-token rate limits, not the token. This is the right pragmatic call for a v1 single-user → small-cohort launch. We revisit with proper user auth (Sign in with Apple) when we have actual user accounts. Document this in proxy/README.md.
  1. Add rate limiting in the Worker: max N requests/minute per IP, max M requests/day per token. Use Cloudflare's rate limit primitives or KV-backed counter. Conservative limits (e.g., 30/min, 1000/day) — tune later.
  2. Request logging (no payloads — timestamps, route, status, latency only).
  3. Forward Gemini and Claude error responses transparently — don't swallow them.
Acceptance:
  • wrangler dev runs locally
  • Both routes proxy correctly to upstream — verified with curl against deployed staging Worker
  • Auth rejects requests without bearer token (401)
  • Rate limits return 429 with Retry-After when exceeded
  • Streaming works end-to-end for /v1/chat
  • Logs in Cloudflare dashboard show test traffic

Task 2.3 — Migrate the client to use the proxy

  1. Update VisionService (Gemini) and ConversationService (Claude) to hit the proxy instead of upstream.
  2. Replace EXPO_PUBLIC_CLAUDE_API_KEY and EXPO_PUBLIC_GEMINI_API_KEY with EXPO_PUBLIC_PROXY_URL and EXPO_PUBLIC_PROXY_TOKEN.
  3. Update CLAUDE.md and README.md for the new env setup.
  4. Remove all upstream provider key references. grep -ri 'CLAUDE_API_KEY\|GEMINI_API_KEY' . returns zero hits outside historical docs.
  5. ai_call_log continues to work — client still measures latency/tokens. Proxy logs are separate.
Acceptance:
  • All existing features (meal photo → analysis → save, coach chat with tool use) work end-to-end against deployed proxy
  • Existing tests still pass
  • Build a release IPA. Unzip it. grep -r 'sk-ant\|AIza' . returns zero hits.
  • Integration-style test that mocks fetch verifies the client targets the proxy URL
  • PR description includes grep verification output

Task 2.4 — Rotate the now-leaked keys

Why: Once the proxy is shipped, keys in past TestFlight builds remain leaked in those binaries. We rotate them so they're permanently dead.

  1. After new client is on TestFlight and confirmed working, Andy rotates Anthropic and Gemini keys.
  2. New keys go into Cloudflare Worker secrets only.
  3. Old TestFlight builds will stop working — intentional and correct.
Acceptance:
  • Old keys rotated by Andy, confirmed
  • Worker secrets updated
  • Current TestFlight build works on new keys
  • Documented in JESSE-NOTES.md

Phase 3

Multi-User Productization

Goal: Make this an app a stranger can install. Remove "Andy"-specific assumptions. Add flows Apple requires for approval.
Duration: 4–5 days.
Exit gate: A fresh install on a new device walks a new user from "first launch" to "first meal logged" without any reference to Andy and no hardcoded personal values.

Task 3.1 — First-run onboarding flow

Why: Architecture review item #6. DEFAULT_GOALS hardcodes displayName: 'Andy' and targetWeightKg: 83.9. Cannot ship.

What to do:

  1. First-run onboarding triggered when app launches with no user_profile row.
  2. Screens (one decision per screen — don't make this a wizard from hell):
    • Welcome + value prop
    • Display name
    • Sex/age (for BMR calculation) — explain why we ask
    • Current weight + target weight
    • Daily calorie & macro targets — with "suggested defaults" option
    • HealthKit permission ask (with skip option — see Task 3.4)
    • Done
  3. Each screen skippable to "set this later" except display name. Completion tracked by onboarding_complete flag in user_profile.
  4. Existing Settings screen is the single source of truth — onboarding writes to the same fields Settings edits. No parallel state.
  5. Do not invent the calculation for suggested macros. Use Mifflin-St Jeor for BMR with 1.55 activity multiplier as default. Document formula in code comments and JESSE-NOTES.md.
Acceptance:
  • Fresh install → onboarding shows → completes → main app shows
  • All "Andy" defaults removed (grep -i 'andy' . outside docs returns zero hits)
  • All hardcoded targetWeightKg: 83.9-type personal values removed
  • Existing users (completed profile) do NOT see onboarding on app update — gated on onboarding_complete flag, with migration setting it true for any row with a non-empty displayName
  • Unit tests on BMR / macro suggestion functions

Task 3.2 — Account deletion flow

Why: Apple guideline 5.1.1(v): if an app supports account creation, it must support deletion. We technically don't have "accounts," but reviewers interpret this broadly — any persistent user data should be deletable from within the app. We do this defensively.

  1. Settings: "Delete all my data" with confirmation flow ("This will permanently delete all your meals, weights, and settings. Cannot be undone.").
  2. On confirm: DELETE FROM each user-data table in a single transaction. Reset to fresh-install state. Trigger onboarding on next launch.
  3. Log action in Sentry as a breadcrumb, not an error.
Acceptance:
  • Delete flow works, app returns to onboarding state
  • Confirmation prevents accidental deletion (two taps minimum)
  • Test covers deletion logic against in-memory SQLite mock

Task 3.3 — Privacy policy & Terms of Service

Why: Apple requires a public privacy policy URL in App Store Connect, and you need ToS for any app talking to a server.

This is Andy work — Jesse wires the links:

  1. Andy provides URLs for /privacy and /terms.
  2. Settings: "Privacy Policy" and "Terms of Service" rows open the URLs in the in-app browser.
  3. Onboarding final screen: "By continuing you agree to our Terms and Privacy Policy" with tappable links.
Acceptance:
  • Links open correctly
  • URLs configured via app config (updatable without rebuild)

Task 3.4 — HealthKit usage strings & permission flow review

Why: Apple's HealthKit review is a separate, stricter track. Common rejection: vague NSHealthShareUsageDescription / NSHealthUpdateUsageDescription strings.

  1. Audit current usage strings in app.json / Info.plist. Must be specific: which data types, what the app does with them, that data stays on-device.
  2. "Why we ask for Health access" explainer screen in onboarding before the system permission prompt.
  3. Audit read and write permission arrays in HealthKitService — trim anything unused.
  4. Confirm graceful degradation if HealthKit is denied — manual weight entry must work.
Acceptance:
  • Usage strings explicit and specific (cite exact strings in PR)
  • Permission denial path tested manually
  • PR description lists every HealthKit type we read and write, with one-line justification each

Phase 4

Reliability Hardening

Goal: Pay down the remaining debts from the architecture review.
Duration: 2–3 days.

Task 4.1 — Cryptographic UIDs

Replace every Math.random().toString(36).slice(2) + Date.now().toString(36) with crypto.randomUUID(). (RN 0.74+ has it natively; if Expo's runtime doesn't expose it, use expo-crypto's randomUUID.)

Acceptance:
  • grep -r 'Math.random' src/ returns no UID-generation hits
  • All tests still pass
  • Unit test verifies UID format is valid UUID v4

Task 4.2 — Migration robustness

  1. Wrap each migration in its own withTransactionAsync.
  2. Update db_version inside the same transaction as the migration DDL.
  3. If any step throws, transaction rolls back, db_version stays at old value, next launch retries.
  4. Test simulating a failing migration verifies db_version is unchanged.
Acceptance:
  • Per-migration transactional behavior verified by test
  • Documented in code comments

Task 4.3 — pendingImage cleanup

Always pendingImage.clear() on confirm.tsx mount. Move to a temp file path (expo-file-system's cacheDirectory) rather than in-memory base64. Prefer the temp file path — more robust on backgrounding.

Acceptance:
  • Backgrounding the app mid-analysis no longer pollutes the next capture
  • Manual test documented in PR description

Task 4.4 — Cold start performance baseline

  1. Measure cold-start time on iPhone XR (oldest supported device — confirm with Andy).
  2. Document in JESSE-NOTES.md. Goal: < 2 seconds to first interactive screen.
  3. If worse, identify largest contributors (likely SQLite migration, Sentry init, bundle size) and propose a fix for me to review before implementing.
Acceptance:
  • Baseline number documented
  • If > 2s, a proposal exists
Phase 4 gate: Arya reviews.

Out of Scope (Explicitly)

If these come up: write them down, don't do them.
  • New features (calisthenics, YouTube video integration) — deferred to post-launch roadmap
  • iCloud / CloudKit sync — explicit defer. iOS device backup covers data preservation acceptably for v1. Users will be told data is on-device. Revisit after real user feedback.
  • Sign in with Apple — deferred. No user accounts in v1. Proxy token + rate limits are our pragmatic stand-in. Revisit when we add real accounts post-launch.
  • Android — explicitly iOS-first; HealthKit is iOS-only
  • Onboarding A/B testing infrastructure — premature
  • Analytics beyond Sentry — premature, and Apple privacy labels get more complicated for every SDK we add
