EMPIRE-OS — Empire Operating System

> The long-term vision and north star for the Aspen Frost agentic empire. Andy + Aspen only. > For current org state, see EMPIRE.md (all agents).

Started 2026-05-23. Living document.

---

Vision

A full agentic product development pipeline — from idea to production — organized around three core functions: Design, Product, and Engineering. Andy acts as CPO, setting direction and staying in the loop until trust is established with each agent and flow.

This is simultaneously a real production system for building apps and an experiment in how agent teams should be structured.

Two Governance Ramps

Trust Ramp

How much autonomy agents get. See the Human-in-the-Loop section below.

Safety Ramp

What kinds of projects we take on. Independent of trust — you can have high-trust agents and still keep the safety posture conservative because the domain demands it.

StageCharacteristicsExamples
Low stakesNo PII, no personal data, no financial or health data. If something goes wrong, nobody gets hurt. User is Andy himself or a tiny trusted group.Frost Fitness, Frost Rocks, Ice Age Trail
Medium stakesReal users, real expectations, small blast radius. May involve friends & family with real data. Requires explicit conversation before starting.Curate Mental Health (note: mental health data is sensitive even at small scale — handle carefully)
Higher stakesExternal clients, real user bases, regulated data (health records, financial data, PII at scale). Each requires explicit Andy sign-off regardless of trust level.TBD

The rule: don't let excitement about a project skip the safety ramp conversation. The early empire builds in low-stakes territory intentionally — to build the process, the tooling, and the judgment before the stakes get real.

Client transparency principle: when outside people are involved, say the stakes-ramp position out loud before committing. A simple personal utility can be treated differently from a project with privacy, financial, or health implications. The promise is not "we can build anything now"; the promise is "we will be honest about what our operating maturity can safely support."

Two Axes of Evolution

Axis 1 — Breadth (stand up the functions)

Build out each of the three functions and establish how Andy interacts with each one. Key questions:

  • What does a good session with each role look like?
  • What artifacts does each role produce?
  • How do handoffs between functions work?

Axis 2 — Depth (adversarial review)

Introduce productive tension within and between roles. Examples:

  • Architect vs. Engineer — systems thinking vs. practical buildability
  • Designer vs. PM — user experience vs. scope and timeline
  • QA vs. Engineer — quality bar vs. shipping pressure
  • Strategist vs. PM — long-term vision vs. near-term roadmap

Human-in-the-Loop Philosophy

Andy stays in the loop on everything until trust is established, then gradually relaxes involvement in flows that feel safe.

Trust ramp (rough phases):

PhaseDescription
1Andy + one agent at a time. Andy is the integration layer between functions.
2Agents hand off to each other. Andy reviews the handoffs.
3Adversarial pairs run autonomously. Andy reviews outcomes, not process.
4Andy is CPO again — setting direction and reviewing exceptions only.

The goal of Phase 1 isn't just execution — it's building Andy's intuition for which decisions actually need him and which can be delegated.

Model Strategy

Different roles may use different models deliberately. Hypothesis: a model optimized for code generation isn't necessarily best for strategic challenge or long-horizon architecture reasoning. Experiment with model assignment per role rather than defaulting to "best available."

Candidate pairings to explore:

  • Strategist / thought partner — a model with a different reasoning style, slightly adversarial
  • Architect — strong at systems reasoning and constraint satisfaction
  • Engineer — strong at code generation and instruction-following
  • QA — strong at finding edge cases and failure modes

---

Checks & Balances Patterns

Patterns to explore as the system matures:

  • Adversarial review — QA and Engineer have explicitly different incentives in their prompts
  • Cross-functional sign-off — Designer and PM must agree before Engineering starts
  • Red team agent — dedicated agent whose only job is to find problems with everyone else's output
  • Confidence routing — agents flag low-confidence decisions for human review rather than proceeding

Token Economics & Model Strategy

Token Spend Tracking

  • Andy has ~$100 in Claude API credit — meaningful threshold between "hobby" and "real operation"
  • Aspen's responsibility: track and report on token spend across agents, surface burn rate, flag when we're approaching thresholds
  • Need to build a process for periodic spend updates (how much spent, by which agent, on what)
  • Arya's ai_call_log table in Frost Fitness is a good model — the empire needs something analogous at the infrastructure level

Backlog item: Build a token spend reporting process — periodic summary of API usage by agent, burn rate trend, and per-project cost attribution. This is a prerequisite for any token-cost-recovery client offer; account-level or agent-level burn is not enough once Andy asks someone else to cover spend.

Model Differentiation Strategy

Two reasons to run different models per agent (not just cost):

  • Cost optimization — Jesse and other execution agents should run cheaper models; strategic agents (Izzy, Arya, Petra) earn their token cost
  • Adversarial diversity — different model providers have different reasoning patterns, blind spots, and strengths. An empire where every agent runs the same model is an empire with correlated failure modes. Provider diversity = diversity of thought = better stress-testing.

Working hypothesis: Anthropic for strategic/design agents, experiment with other providers (OpenAI, Google, Mistral) for specific roles where their strengths align. Don't default to "best available" — match model to role deliberately.

Backlog item: As each new agent is built, make a conscious model assignment decision and document the rationale. Revisit periodically as models evolve.
