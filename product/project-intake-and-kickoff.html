Project Intake & Kickoff Template

> Reusable template for moving a project from fuzzy idea into a scoped, cross-functional project record. > Project-specific filled copies belong under projects/<project>/, not in this department folder.

Use this when Andy has enough signal to ask: should this become a real project, what are we committing to, and what questions have to be answered before design or engineering starts?

The name can flex by stage:

  • Project Intake when the question is still, "Should we do this?"
  • Project Kickoff when the answer is probably yes and the work is entering the formal pipeline.
  • Project Charter when the result becomes the durable beginning-of-project record.

When To Run It

Run intake after an initial idea, stakeholder conversation, discovery call, or rough opportunity becomes concrete enough to evaluate. Do not skip this because the project feels obviously good. The point is to make the assumptions visible while the cost of changing direction is still low.

Inputs can include:

  • Andy's initial thoughts
  • Stakeholder notes or discovery call recording
  • Existing product/app/repo links
  • Screenshots, app store listings, analytics, or user feedback
  • Known constraints, timelines, and non-goals

Outputs should include:

  • A filled PROJECT-INTAKE.md or README.md in projects/<project>/
  • An explicit stakes-ramp declaration
  • Open questions by function
  • A recommendation: proceed, pause, research more, or reject
  • The next gate in the process

Intake Agenda

  • Context dump - Andy provides the story, links, call notes, recordings, repo access, and initial instincts.
  • Clarify the opportunity - Petra and Andy pressure-test user, customer, market, and stakeholder value.
  • Map the current state - Document the existing product, workflow, data, code, ownership, and known pain points.
  • Declare the stakes - Aspen runs the 1-5 stakes rubric and records the initial safety posture.
  • Role questions - Petra, Izzy, Arya, Zev, and any relevant operators contribute questions before giving confident analysis.
  • Initial analysis - Each principal gives a short take: opportunity, risks, unknowns, and recommendation.
  • Scope the next move - Decide whether the next artifact is research, design, architecture review, prototype, or formal project charter.
  • Gate decision - Andy decides whether to proceed, pause, gather more signal, or drop it.

Stakes Ramp Rubric

Score each dimension from 1-5, then make a judgment call on the overall project level. The overall level is not an average; one high-risk dimension can set the project posture.

LevelLabelDescriptionExamples / Notes
1Low stakesPublic or generic data, no meaningful PII, no financial/health/legal impact, tiny blast radius, easy rollback, no serious harm if wrong.Trail information, hobby/reference apps, internal experiments, Frost Rocks-style learning tools with no sensitive data.
2Mild stakesReal users and expectations, light account data or preferences, inconvenience if wrong, but no sensitive domain and no serious dependency.Friends/family tools, small scheduling utilities, simple profile/preferences data.
3Moderate stakesExternal stakeholder, meaningful personal data, operational dependency, reputation risk, or decisions that affect real plans. Requires explicit scope and support expectations.Camp scheduling with family logistics, small business tools, apps people rely on weekly.
4High stakesSensitive personal data, health-adjacent context, financial implications, regulated workflows, or users depending on correctness for important decisions. Requires strong review, privacy posture, and reliability plan before commitment.Curate Mental Health-style work, anything involving health, finances, children, or sensitive records.
5Critical stakesRegulated production systems, clinical/legal/financial decision support, high-volume PII, safety-critical workflows, or material harm if wrong. Do not enter casually; requires mature operating controls and explicit Andy sign-off.HIPAA/clinical records, payment/financial systems, legal compliance systems, safety-critical tools.

Required Stakes Declaration

Every intake must include:

  • Proposed level: 1-5
  • Reasoning: the 2-4 facts that drive the score
  • Highest-risk dimension: even if the overall project is low
  • Mitigations: what keeps the project inside the declared level
  • Gate: what would cause the project to move up the stakes ramp
