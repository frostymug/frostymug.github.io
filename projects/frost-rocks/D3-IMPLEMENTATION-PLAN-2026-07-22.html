0. Feasibility answers (D3-Q5 – D3-Q7)

D3-Q5 — Vision transport: feasible, cheap, one new dep

- Gemini 2.5 Flash: ~$0.07/session - Gemini 2.5 Pro: ~$0.30/session - One session per evening — cost is a rounding error either way. Ten cents of nightly tutoring is not the constraint; trust is.

  • Format/size: downscale to 1280 px long edge, JPEG q≈0.7 via expo-image-manipulator (new dependency — photos are stored full-res; nothing in the app downscales today). That's ~200–400 KB/photo, ~300–550 KB base64. Sent as Gemini inline_data. Limits are nowhere near binding: Gemini inline-request cap is 20 MB, Workers request cap 100 MB. A 4-photo turn is ~1.5 MB.
  • Streaming + images concurrently: yes, cleanly. Images ride up in the request; streaming is the response (streamGenerateContent?alt=sse). The frost-fitness Worker already forwards upstream.body unbuffered — SSE flows through a transparent proxy for free. Client side: React Native's built-in fetch cannot stream response bodies — but expo/fetch (SDK 52+, we're on 54) can. That's the one client-side subtlety, and it's solved by an import.
  • Cost envelope (2–4 photos, 8–15 turns, full-context resend per D3-Q7): a 1280 px photo ≈ ~1K Gemini tokens. Worst-case session ≈ ~190K input / ~5K output tokens.
  • Latency: TTFT ~1.5–3.5 s (Flash) / ~3–8 s (Pro) with images; streaming makes both feel fine for a couch session. Per-turn photo upload on hotel wifi is the real latency, which is why the send-photo policy below matters.
  • Send-photo policy (tightens spec §6): send only the active agenda encounter's photos per turn (≤2/encounter), not all agenda photos every turn. Other encounters travel as text (note + place + residue so far). Cuts per-turn upload ~4× with zero credibility loss — the tutor only needs to see the rock under discussion.
  • Model — DECIDED (Andy, 2026-07-22): gemini-3.6-flash for /tutor (and /enrich). Newer-generation Flash; cost stays in the same trivial band as the envelope above. Kept env-switchable (TUTOR_MODEL Worker var) 🚪➡️🚪 — if the Strimpel-don register flattens in the first real sessions, the flip is one deploy, no client change. B2b's round-trip gate doubles as the first voice check.

D3-Q6 — Structured output: sentinel-delimited trailing block, validated app-side, repaired in background

Three candidate seams, one recommendation:

SeamVerdict
Gemini responseSchema (whole-body JSON)❌ Kills streamed prose feel (or forces streaming-partial-JSON parsing of a prose field — fiddly, fragile)
Function calling for the delta❌ Part-ordering with streamed text is not reliably prose-then-call; adds API surface for no reliability gain we can't get cheaper
Prose + sentinel + trailing JSON block✅ Recommended. One call per turn, deltas ride the same stream

Contract: model streams prose, then a line ---DELTA---, then the JSON block (§4 of the wedge spec). Client renders tokens until the sentinel, buffers the remainder, parses, validates against a schema.

Reliability comes from the applier, not the model (this is the important part):

  • Schema-validate the block (lightweight hand-rolled validator or zod — Jesse's call, it's ~40 lines either way).
  • The delta applier enforces the conservative-by-rule table structurally: dialogue-sourced evidence can only produce exposed/connected; only retrieval/field_application can produce solid; user_challenge moves either direction; unknown concept slugs (not in the bundled spine) are rejected and logged — the model doesn't get to invent curriculum at runtime by typo. Never trust the prompt to enforce what the applier can enforce in code.
  • Malformed block → prose still renders (spec rule: conversation never blocks on bookkeeping); a background repair call re-sends that assistant turn with "emit only the delta block, no prose" (non-streaming, ~1K tokens, pennies). One repair attempt, then park the delta as rejected with the raw text kept for manual replay. Idempotency: deltas apply once per (session_id, message_id).

D3-Q7 — Session state: stateless proxy, client resends full context. Not close.

  • Token cost of resend is trivial (the whole session is ~$0.07–0.30, above). The "tradeoff" the spec worries about is real but numerically irrelevant at one user × one session/evening.
  • Stateless keeps the Worker a transparent pipe (playbook pattern, already proven in frost-fitness), no session store to build/expire/secure, no state to leak. Privacy stays clean: photos transit per turn, nothing persists server-side — matching the spec's own privacy language.
  • Everything the prompt needs is already on the device: encounters, photos, Layer B rows, bundled spine, voice guide, transcript (tutor_messages). Prompt assembly is client-side pure functions — which also makes it unit-testable in Jest without a network.
  • 🚪➡️🚪: if a future moment (T3 over a whole trip) blows the context budget, we add server-side session memory then, for that route. Nothing in this design forecloses it.

-- concepts: bundled Layer A slice, upserted on spine_version change
CREATE TABLE IF NOT EXISTS concepts (
  id TEXT PRIMARY KEY,              -- slug, stable across spine versions
  name TEXT NOT NULL,
  family TEXT,
  diagnostic_features TEXT,         -- json
  confusion_pairs TEXT,             -- json: [{other_slug, resolving_feature}]
  place_packs TEXT,                 -- json: ["finland", ...]
  keywords TEXT,                    -- json: match terms for concept-implication (v1 keyword pass)
  spine_version TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS user_concept_state (
  concept_id TEXT PRIMARY KEY REFERENCES concepts(id),
  state TEXT NOT NULL CHECK (state IN ('exposed','connected','shaky','solid')),
  first_touched_at TEXT NOT NULL,
  last_evidence_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS evidence (            -- append-only; the provenance rail (A4 🚪🧱)
  id TEXT PRIMARY KEY,
  concept_id TEXT NOT NULL REFERENCES concepts(id),
  kind TEXT NOT NULL CHECK (kind IN ('exposure','self_explanation','retrieval',
                                     'field_application','user_challenge','user_mnemonic')),
  direction TEXT NOT NULL DEFAULT 'neutral' CHECK (direction IN ('up','down','neutral')),
  source_type TEXT NOT NULL CHECK (source_type IN ('encounter','dialogue','artifact')),
  source_id TEXT NOT NULL,
  excerpt TEXT,
  created_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_evidence_concept ON evidence(concept_id, created_at);

CREATE TABLE IF NOT EXISTS questions (
  id TEXT PRIMARY KEY,
  encounter_id TEXT REFERENCES encounters(id),   -- nullable
  trip_id TEXT NOT NULL REFERENCES trips(id),
  text TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'open' CHECK (status IN ('open','explored','resolved')),
  resolved_by_session_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_questions_status ON questions(trip_id, status);

CREATE TABLE IF NOT EXISTS tutor_sessions (
  id TEXT PRIMARY KEY,
  trip_id TEXT NOT NULL REFERENCES trips(id),
  moment TEXT NOT NULL CHECK (moment IN ('d3','t1','d1','t3')),
  started_at TEXT NOT NULL,
  ended_at TEXT,
  agenda TEXT,                                   -- json
  summary TEXT
);

CREATE TABLE IF NOT EXISTS tutor_messages (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL REFERENCES tutor_sessions(id),
  role TEXT NOT NULL CHECK (role IN ('user','tutor')),
  content TEXT NOT NULL,
  encounter_id TEXT REFERENCES encounters(id),   -- nullable
  created_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_tutor_messages_session ON tutor_messages(session_id, created_at);

-- delta bookkeeping (idempotency + repair queue)
CREATE TABLE IF NOT EXISTS tutor_deltas (
  message_id TEXT PRIMARY KEY REFERENCES tutor_messages(id),
  status TEXT NOT NULL CHECK (status IN ('applied','rejected','pending_repair')),
  raw_json TEXT,                                 -- kept for replay/audit
  applied_at TEXT
);

-- events rebuild: drop the type CHECK (fixes silent-drop bug, unblocks §7 instrumentation)
CREATE TABLE events_new (
  id TEXT PRIMARY KEY, ts TEXT NOT NULL, type TEXT NOT NULL, data_json TEXT
);
INSERT INTO events_new SELECT id, ts, type, data_json FROM events;
DROP TABLE events;
ALTER TABLE events_new RENAME TO events;
CREATE INDEX idx_events_ts ON events(ts);
CREATE INDEX idx_events_type ON events(type, ts);

4. Delta contract v1 + applier rules (B1)

Wire format per wedge spec §4, with one addition — encounter_tags for the two-pass concept-implication (spec §5, block 2):

{
  "encounter_tags": [{"encounter_id": "…", "concepts": ["jointing", "migmatite"]}],
  "deltas": [{"concept": "jointing", "state": "exposed->connected",
              "evidence": {"kind": "self_explanation", "source_type": "dialogue",
                           "source_id": "<message_id>", "excerpt": "…"}}],
  "questions": [{"op": "open" | "resolve", "id": "q_12?", "encounter_id": "…?", "text": "…?", "note": "…?"}],
  "learning_note": {"encounter_id": "…", "text": "…"},
  "mnemonics": [{"concept": "feldspar_vs_quartz", "text": "…", "verbatim": true}]
}

Applier rules (code, not prompt):

  • Unknown concept slug → reject that row, log delta_row_rejected{reason:"unknown_concept"}, apply the rest.
  • Transition legality by evidence kind: exposure|self_explanation (dialogue) → may set exposed/connected only · retrieval|field_application → may set solid · user_challenge → any direction · user_mnemonic → evidence row only, no state change.
  • Every applied delta writes the evidence row and upserts user_concept_state in one transaction.
  • learning_note writes via existing setAiLearningNote() — AI-owned field only; no code path from delta to any user field (audit: grep-able, testable).
  • Idempotent per message_id (tutor_deltas table).
  • Malformed top-level JSON → pending_repair → one background repair call → applied or rejected (raw kept).

5. Prompt assembly (client-side, pure) (B3)

src/services/tutor/promptAssembly.ts — pure functions, Jest-testable, no I/O. Blocks per spec §5:

  • System/persona ← bundled voice guide (versioned) + guardrails (§2 of spec, incl. ≤2-beat challenge rule, uncertainty grammar, turn budget).
  • Layer A slice ← concepts implicated by tonight's encounters: v1 = keyword match (concepts.keywords) over note text ∪ place-pack default cluster; plus any encounter_tags returned by the model's own agenda turn (two-pass). Include confusion pairs of every included concept.
  • Layer B ← compact render of user_concept_state + open questions + user_mnemonic excerpts.
  • Session objects ← today's encounters: notes verbatim, guess/confidence/uncertainty, place/time/event tag; photos per policy A7.
  • Task frame ← moment-specific job description: full loop (agenda→per-encounter→thread→close) or abbreviated (single-encounter, no agenda/close ceremony — doors 2 and 3).
  • Output contract ← prose + sentinel + delta JSON schema.

6. Transport (B2)

Worker (proxy/, scaffolded from frost-fitness):

  • GET /health (no auth) · POST /tutor · POST /enrich (finally give the dormant outbox its server — one-shot, env.ENRICH_MODEL, non-streaming; same shape the client already sends).
  • /tutor: transparent pipe to streamGenerateContent?alt=sse on env.TUTOR_MODEL (default gemini-3.6-flash, per A1); forwards upstream.body unbuffered (pattern already proven). Worker enforces maxOutputTokens cap (~1.5K — turn budget is 2–5 sentences + delta block) and strips/ignores any client-supplied generation config outside the allowlist.
  • Auth + dual rate limiters + metadata-only logging: copy frost-fitness verbatim. ai_call_log mirror rides the existing client table.
  • CI/CD: worker-ci.yml (typecheck, tests, secret scan, wrangler dry-run), worker-deploy.yml with worker-production environment gate. Deploy is Cassian's (SRE veto lane) — Jesse writes it, Cassian ships it.

Client (src/services/tutor/transport.ts):

  • expo/fetch streaming reader → token callback → sentinel splitter (---DELTA---) → delta pipeline (§4).
  • Downscale-and-encode helper (expo-image-manipulator + base64 read) with in-session memo so a photo is encoded once per session, not per turn.
  • Online-only: no outbox, no retry queue. Failure mid-turn = quiet inline "the guide lost the thread — tap to continue" that resends the turn (idempotent by message_id).
