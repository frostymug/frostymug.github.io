0. What this is

The first shipped tutor surface: the evening debrief — a tutor-guided review session over the day's encounters, entered from existing doors, producing durable residue on the notebook objects (learning notes, resolved questions, learner-model updates). D3 ships first per locked decision EMB-1; this spec covers only D3 plus the minimum shared substrate (knowledge tables, delta contract, tutor transport) that every later moment (T1/D1/T3) will reuse.

The bet being shipped (validated in WoZ pilot #1): a dialogue that starts from Andy's own noticing, teaches through it, and calls back concepts across encounters produces the "it knows what I know" feeling with a small curated spine + simple evidence rows — no embeddings, no knowledge tracing (H3), and the generative hunch-step feels satisfying, not like homework (H7 — supported by F2/F4).

1. Experience — STRUCTURE

Everything in §1 is layout/flow/navigation. None of it is paint. Build gates test these seams, not colors.

D3-S1 — Entry doors (3, all existing surfaces)

  • Revisit banner, debrief state (trigger per locked D3-Q1): the banner shifts to debrief framing — "4 encounters today — debrief with your guide?" — when (≥2 encounters captured today, at any hour) OR (≥1 encounter AND local time ≥ ~17:00). Rationale (Andy): two-plus encounters make a "review the day so far" worthwhile whenever; a single encounter waits for dinnertime. Tapping opens the D3 session. Otherwise the banner keeps current behavior. No new nav tab (canon: no chatbot-shaped destination).
  • Encounter detail affordance. A quiet "Explore with your guide" row on encounter detail opens a single-encounter D3 session scoped to that encounter — abbreviated loop (locked D3-Q2): the encounter rhythm of D3-S3 without agenda or close-out ceremony.
  • Review flow handoff. Inside the existing staged-reveal review, after the original-note reveal, a "talk it through" affordance escalates into dialogue for that encounter. (The staged-reveal flow is the chassis, per brief; dialogue slots in after the note reveal, exactly where the pilot's elaborative prompt sat.)
  • Bottom-nav Review → chooser page (locked, Andy 2026-07-22): Review no longer deep-links into an encounter review. It opens a small chooser: "Review Today's Encounters" (→ D3 session over today) and "Review the Trip So Far" — visible but stubbed/unclickable in v1; it is the reserved door for T3 trip consolidation and quietly advertises the coming loop. Reviewing a specific encounter is reached from the timeline or the map (its natural object doors), not from the nav.

Optionality is sacred: all three doors are invitations. No badge, no streak, no scolding for skipped evenings (guardrail; T3 catches what days miss).

D3-S2 — Session shape

OPEN      Tutor greets w/ context (day, place, count) — 1 short turn
AGENDA    Tutor proposes 2–4 items, user can reorder/skip/add
            priority: (1) encounters with explicit questions in notes
                      (2) photo-only encounters (enrichment targets)
                      (3) one cross-day consolidation thread (if material exists)
PER-ITEM  The encounter loop (D3-S3), one encounter at a time
THREAD    One session-spanning connection surfaced when real
            (pilot: "pink is the pigment, texture is the plot" — F3/F9)
CLOSE     Land the plane: 2–3 line recap, residue summary
            ("2 questions resolved, 1 quest opened, 3 notes written"),
            deltas applied. Target session: 5–15 min (guardrail).

User can bail at any point; residue for completed items persists; agenda leftovers stay later (no penalty).

D3-S3 — The per-encounter loop (the core rhythm, validated F9)

  • Context reinstatement. Tutor shows the photo + reads back the user's own note verbatim (quoted, honored — never paraphrased away). Pilot evidence: reading the note back was itself valued. If photo-only: tutor asks for the field memory ("why did you raise the phone?") — that is the reinstatement.
  • Elaborative prompt (hunch-first, INT-1). One question extending the user's own noticing — "tell me more about X", never pop-quiz. Visible "just tell me" escape (went unused in pilot — keep it anyway; it's the pressure valve that makes the prompt feel safe, F2). Rule from pilot: the prompt must be honest data-gathering — asked because the answer genuinely informs the reveal — not quiz theater (F2).
  • Reveal. Answer through what the user noticed. Error-correction sequence: validate what's right → correct via the single diagnostic feature → name it "productively wrong" energy, never gotcha (F5). Teach-the-term (KM-4): honor user's words, introduce curriculum vocabulary, adopt it forward. Honest uncertainty grammar mandatory: likely / can't rule out / what-would-settle-it (F6, H5-adjacent).
  • One look-again. Every encounter closes with exactly one field-executable check for next time ("tilt it in the light — what flashes is feldspar"). One, not three.
  • Residue write (D3-S4) + model delta (§4), then next agenda item.

Zero quiz energy in D3 (locked INT-2). Retrieval practice lives post-trip; the generative step here is the elaborative prompt only.

D3-S4 — Residue on objects (dialogue is means, notebook is the end)

ResidueWhere it lands
Learning noteencounters.ai_learning_note (existing field; AI-owned; never touches user fields)
Question opened/resolvedquestions row linked to encounter (new, §3)

The learning note is written during the session (user sees it land — "noted on the encounter") — trust through visible bookkeeping.

D3-S5 — Skin (separate, deliberately thin here)

Field-guide visual language per shipped v2.2. Tutor turns visually distinct from user notes (the notebook stays the user's). One skin-level flag from the pilot worth designing: the tutor showing its work on the photo (zoom/crop of the diagnostic region, F7) built trust in WoZ — spec'd as a v1-optional enhancement (D3-Q4), not a gate.

2. Interaction contract (prompt-enforced, tested in pilot)

  • Voice: the Strimpel don (locked INT-3) — warm, precise, British-register, Oxbridge-tutorial stance; inspiration not impersonation. Governed by the voice guide artifact (separate deliverable, in progress; F5/F6/F8 vocabulary folded in). Voice guide is versioned data, referenced by the system prompt — not hardcoded prose.
  • Turn budget: 2–5 sentences default, expandable on request. Reveal turns may run longer when earning it (pilot reveals ran ~2× budget and Andy engaged — watch, don't clamp yet; F8).
  • Grounded or silent: claims about this rock tie to visible photo evidence or the active place-pack; otherwise say what would settle it. No freelancing Finnish geology.
  • User challenge handling (KM-1): a pushback on any tutor claim or model state is welcomed, may be probed for ≤2 beats, then conceded/recorded as user_challenge evidence. Pilot proof: the ovoid-size challenge → recalibration increased trust (F6). A tutor that argues past two beats is a bug.
  • Never overwrite the user's words. AI writes only AI-owned fields. Architectural rule and prompt rule.
  • No unprompted homework; land the plane (session budget guardrail).

---

3. Data model — additive tables (D3-D1)

Encounter table untouched (the v0 bet held; keep it held). All new tables are the shared tutor substrate — designed so T1/D1/T3 add rows, not tables. Single-trip assumption stays, but every table carries trip_id so multi-trip doesn't force migration.

concepts               -- Layer A slice, BUNDLED DATA not user data; versioned like code
  id, slug, name, family, diagnostic_features (json),
  confusion_pairs (json: [{other_slug, resolving_feature}]),
  place_packs (json: ["finland","north_shore"]), spine_version

user_concept_state     -- Layer B state, one row per touched concept
  concept_id, state (exposed|connected|shaky|solid),
  first_touched_at, last_evidence_at

evidence               -- append-only; THE provenance rail (most important schema detail)
  id, concept_id, kind (exposure|self_explanation|retrieval|
       field_application|user_challenge|user_mnemonic),
  direction (up|down|neutral), source_type (encounter|dialogue|artifact),
  source_id, excerpt (text, e.g. verbatim mnemonic), created_at

questions              -- promoted from note text to first-class
  id, encounter_id (nullable), trip_id, text,
  status (open|explored|resolved), resolved_by_session_id, created_at

tutor_sessions         -- one per D3 session (schema moment-agnostic: `moment` field)
  id, trip_id, moment (d3|t1|d1|t3), started_at, ended_at,
  agenda (json), summary (text)

tutor_messages         -- transcript, backgrounded
  id, session_id, role, content, encounter_id (nullable), created_at

Notes:

  • **evidence.source_ provenance is load-bearing* — every future "remember when…" stands on it (brief §4.6). Do not ship without it.
  • Concept spine ships as bundled JSON with a spine_version; user rows reference concept ids stably across spine updates.
  • The pilot's Layer B deltas (WOZ-D3-PILOT-1 §2) are the seed fixture for testing these tables against real data.

4. Model-delta contract (D3-D2)

Every tutor response returns prose + a structured delta block; the app applies deltas — the dialogue never free-writes the model.

{
  "deltas": [
    {"concept": "jointing", "state": "exposed->connected",
     "evidence": {"kind": "self_explanation", "source_type": "dialogue",
                  "source_id": "<msg>", "excerpt": "cracks perpendicular to foliation"}}
  ],
  "questions": [{"op": "resolve", "id": "q_12", "note": "gaps = joints"}],
  "learning_note": {"encounter_id": "...", "text": "..."},
  "mnemonics": [{"concept": "feldspar_vs_quartz", "text": "flashy feldspar…", "verbatim": true}]
}

Conservative-by-rule (structural, not vibes):

  • Dialogue evidence can move state to exposed/connected only.
  • Only retrieval or field_application evidence can produce solid.
  • user_challenge can move state either direction, after ≤2-beat probe.
  • Malformed/rejected delta → prose still renders, delta queued for retry; the conversation never blocks on bookkeeping.

The inspectable "My geology" page (KM-1) is out of D3-v1 scope but the data above is exactly what it will render — nothing here precludes it (D3-Q3).

---

5. Prompt assembly (per session; brief §5.2 instantiated for D3)

  • System/persona ← voice guide (versioned) + guardrails (§2).
  • Domain grounding ← Layer A slice: concepts implicated by tonight's encounters + their confusion pairs + active place-pack. Retrieval by concept-tag match, not embeddings.
  • Learner state ← compact Layer B rendering + open questions + user vocabulary/mnemonics.
  • Session objects ← today's encounters: photos (vision — required, H5), notes verbatim, guess/confidence/uncertainty, place, time, event tag.
  • Task frame ← D3 job description: agenda → per-encounter loop → thread → close-out (§1).
  • Output contract ← prose + delta block (§4).

Concept-implication for block 2 (v1, honest and simple): match on user note keywords → concept slugs + place-pack default cluster; the tutor model itself proposes concept_tags per encounter in its first agenda turn, which then pull grounding for subsequent turns (two-pass, no embeddings).

---

6. Transport (D3-D3)

  • Generalize the dormant enrichment outbox pattern → /tutor route on the existing Cloudflare Worker proxy: streaming responses, multi-turn (session id), accepts encounter photos (the biggest new technical requirement — vision is necessary for credibility, H5; a tutor doing stand-up about a photo it never saw breaks trust in one session).
  • Existing one-shot /enrich remains the offline-queued fallback; D3 itself is online-only by design (couch/hotel moment — offline-critical D2 is untouched, the architectures never fight).
  • Privacy per playbook: keys server-side, model/token caps, photos transit for the session, proxy logs metadata only.
  • Send-photo policy: agenda encounters' photos only, downscaled (long edge ~1280px) — enough for texture verdicts per pilot experience (zoomed crops of a 3MP frame were decisive, F7).

7. Instrumentation (rides the existing events rail)

debrief_invite_shown{door}, debrief_start{door, encounter_count}, encounter_discussed, hunch_prompt_shown, hunch_answered | hunch_skipped("just tell me"), question_resolved, delta_applied{n}, mnemonic_captured, challenge_raised, debrief_complete{duration, items_done, items_skipped}, debrief_abandoned{at}.

These are H1/H7's measurement: invitation→start rate, hunch answer-vs-skip rate, completion time vs. 15-min budget.

8. Content dependency — Layer A Finland slice (blocking, mine)

D3-v1 needs only the Finland place-pack slice of the spine (~20–30 concepts), not the full 60–100. Pilot's seed cluster: jointing, mineral cleavage, foliation, migmatite, gneiss, granite, K-feldspar, quartz, feldspar-vs-quartz discrimination, rapakivi (+ ovoid diagnostic), glacial erratic/unloading, plus the confusion pairs the pilot surfaced (cleavage↔jointing, quartz↔feldspar, granite↔gneiss↔rapakivi). I own this build (Petra review pass); it ships as data, versioned.

9. Build path (sequenced; structure before skin; each step gated)

  • B1 — Substrate: tables (§3) + delta applier + fixture test using pilot deltas. Gate: pilot session log replays cleanly into the tables.
  • B2 — Transport: /tutor proxy route w/ vision + streaming; app session client. Gate: round-trip a real encounter photo + note, get grounded prose + valid delta.
  • B3 — Session structure: doors (D3-S1), agenda, per-encounter loop, close-out, residue writes (D3-S4). Gate: full session on Finland export data end-to-end on device; residue visible on encounters.
  • B4 — Skin: field-guide dress, tutor-turn styling, polish. (Only after B3 gate — structure is not paint.)
  • B5 — Instrumentation + TestFlight.

Arya feasibility questions attached in D3-Q5–Q7.

10. Acceptance (what "working" means)

  • A real evening on trip data: invitation → agenda → ≥2 encounters discussed → residue on objects → done in ≤15 min (H1).
  • Tutor references at least one prior-session/prior-encounter concept correctly from Layer B, unprompted (F3 — the magic-moment mechanic, this is the wedge's soul).
  • Andy answers the hunch prompt more often than he skips it across a week (H7).
  • Learning notes read like the pilot's (§1 of pilot log) — specific, diagnostic-feature-centered, one look-again — not generic rock facts.
  • No user-field ever modified by AI (audit via events + schema perms).

11. Out of scope for D3-v1

T1/D1/T3 surfaces (schema-ready, not built) · "My geology" inspectable page (data-ready; D3-Q3) · quiz/retrieval practice & FSRS · question-marking in field capture · dynamic user themes · multi-trip IA · audio · in-field AI.
