Product Note: First Trip Export Read

Petra | 2026-07-21 | Source: Andy shared Frost Rocks "share data" and "share events" JSON exports from Finland 2026.

Raw sources preserved at:

  • source-material/trip-export-2026-07-21.json
  • source-material/events-export-2026-07-21.json
  • source-material/keyboard-flow-bug-2026-07-21/ screenshots showing the alternate keyboard / Wispr Flow path into a blank encounter screen.

What The Export Contains

  • Format: frost-rocks-export, version 2.
  • Trip: Finland 2026.
  • Encounters: 14.
  • Encounters with notes: 9.
  • Photo-only encounters: 5.
  • Encounters with GPS coordinates: 9.
  • Encounters missing GPS coordinates: 5.
  • Encounters with event tags: 3.
  • Encounters with initial guesses: 2.
  • Encounters with learned_later: 1.
  • Encounters with non-empty themes: 0.
  • Encounters not in review_state: later: 1.

Behavioral Read

Capture is happening. That is the first and most important signal.

The app is already serving as a low-friction field record: Andy has 14 encounters across three days, including several photo-only saves. That supports the camera-first bet. People do not always write in the field; sometimes the win is simply that the thing got caught before the moment moved on.

The structured learning fields are barely being used yet. Themes are empty, guesses are rare, confidence appears only twice, and learned_later appears once. That does not mean those ideas are wrong, but it does mean the current field behavior is still mostly noticing, with only light remembering and almost no explicit connecting inside the data model yet.

This makes Andy's shorthand useful: noticing, remembering, connecting. It maps better to the observed export than a pure rock-ID workflow does.

What The Events Add

The events export contains 99 total events:

  • Capture events: 61.
  • Review opens: 19.
  • Revisit completions: 6.
  • Save events: 13.

The live trip slice from 2026-07-19 onward is more important than the earlier test/debug usage:

  • Live capture opens: 14.
  • Live shutters: 14.
  • Live enrich screens opened: 14.
  • Live save events: 9.
  • Live review opens: 8.
  • Live revisit completions: 4.
  • Live review doors: map 5, timeline 2, nav 1.
  • Median open-to-shutter time: about 7 seconds.
  • Median open-to-save time among saved/enriched flows: about 47 seconds.

This is stronger than the static export alone. The capture funnel is real: every live capture open became a shutter, and every shutter opened the enrichment screen. That supports the two-tap camera-first pattern.

The event data also exposes an instrumentation mismatch. The trip export has 14 live encounters, and the live event log has 14 shutters, but only 9 live save events. Since the encounters exist, this looks more like a logging gap or save-event placement issue than lost data. Still worth fixing before we lean on event metrics too hard.

Andy has a concrete hypothesis for this mismatch: the missing live save events may be related to the alternate keyboard / Wispr Flow glitch. The attached screenshots show this sequence:

  • The saved encounter enrichment screen is open with the standard keyboard focused in the "What did you notice?" field.
  • The keyboard switches into Wispr Flow mode.
  • Wispr Flow opens its own "Flow is on" system/instruction screen.
  • On return, the app shows the "New encounter" sheet header but the body is blank; Andy has to swipe down to escape and navigate away.

That makes the save-event gap more concerning than a simple analytics naming issue. The encounter may be persisted before enrichment, while the later save event depends on a normal enrichment-screen completion path that this keyboard flow interrupts. If so, the app is preserving the rock record but losing the behavioral truth about how the user exited the flow.

The review-door signal is also interesting. During live trip use, map beats timeline and nav: 5 map opens, 2 timeline opens, 1 nav open. That does not prove the map is the main product surface, but it does make the map-zoom request more credible. Andy is not just admiring the map as scenery; he is trying to use it.

What Andy Is Actually Capturing

The note language clusters around:

  • Pink / pinkish color.
  • Cleavage / cleaved rocks / gaps / angles.
  • Banding and foliation.
  • Quartz or quartz-like inclusions.
  • Rapakivi as a tentative remembered label.
  • Place-memory cues: near Maren's apartment, edge of the park, under a tree root, popping out of the trail.
  • Questions: "Why do those gaps appear in those patterns and those angles?"

This is not clean taxonomy usage. It is field cognition in progress: noticing visual features, reaching for partial vocabulary, connecting to prior places, and preserving uncertainty.

Product Implications

Five encounters have no note. Do not shame this pattern. The product should make later enrichment easy and obvious.

  • Photo-only capture is a feature, not failure.

Themes are empty across all 14 encounters. This reinforces the backlog item on dynamic themes and user-defined keywords.

  • The current themes model has no evidence of pull.

At least one note is explicitly question-shaped, and several others are implicit uncertainty. Review should be able to say: "You wondered about cleavage angles here. Want to explore that?"

  • Questions should become first-class review material.

The JSON contains photo filenames and local camera roll asset IDs, but not the image files. For human backup this may be acceptable. For an agent or AI review workflow, it is not enough: the receiving agent cannot inspect the rocks.

  • The export is not yet agent-usable for image-based review.

Five encounters are missing coordinates. Some may be expected if captured offline, permission-limited, or before location resolved. But if place is part of the memory loop, the app needs graceful recovery: "add approximate place later" may matter.

  • GPS gaps need diagnosis.

Almost everything is review_state: later. That may be right during a trip, but the product needs a strong post-field moment where these saved observations become learning fuel.

  • The real review queue is waiting to be designed.

The live capture funnel lines up well through shutter/enrich, but save events are missing for five encounters that appear in the trip export. Before using events as product truth, we need to tighten the logging contract.

  • Event instrumentation is useful but not yet trustworthy enough for decisions.

Five of eight live review opens came through the map door. That reinforces map zoom/pan as more than cosmetic polish.

  • Map review has earned attention.

This is not a polish edge case. If a dictation keyboard can leave the enrichment screen blank and force a swipe-away escape, then the app needs to handle scene/app lifecycle restoration, keyboard focus, and partially completed encounter state more deliberately.

  • Keyboard-extension recovery is part of capture reliability.

Open Product Questions

  • Is "share data" meant for backup, human review, AI review, handoff to agents, or all of those?
  • Should the export include photos, a zip bundle, or stable local retrieval instructions?
  • Should review group encounters by recurring noticing patterns such as pink color, cleavage, banding, quartz, and Rapakivi?
  • Should the app offer "add what you learned later" as a lightweight post-field ritual rather than a hidden field?
  • Should missing GPS trigger a quiet later recovery prompt when the encounter has an event tag or location-like note?
  • Why do five live encounters appear without corresponding save events?
  • Can we reproduce the blank "New encounter" screen by switching to Wispr Flow from the enrichment note field, starting dictation, and returning?
  • Does the app emit save when the encounter is initially persisted, when the enrichment screen completes, or only on a specific button/action path?
  • Should enrichment have an explicit recoverable draft state so app/keyboard interruptions return to the partially filled form instead of a blank sheet?
  • Are enriched and noticed the right event properties, or are they confusing product shorthand leaking into analytics?
  • Should the events export include a lightweight schema legend so another agent can interpret it without reading app code?
