Frost Rocks — Finland Build Readiness Questions (v2)

Arya (Architect) | 2026-06-28 | Status: Superseded reference after Build 1

> Superseded by Finland One-Shot Sync and Finland Architecture. Andy later authorized Arya's autonomous Build 1, so this file should be read as answered pre-build context, not the active plan or gate.

What changed in v2

My v1 asked questions the org already answered in production with Frost Fitness — build target, backend-or-not, Apple Developer account, CI/CD, AI key handling. That was a context-sharing failure on my part. Frost Fitness is a shipped Expo/EAS app, live on the App Store via TestFlight, with a Cloudflare Worker AI proxy, CI/CD, and a rollback runbook. I've now read that repo and its docs and generalized the whole thing into a reusable Aspen Mobile App Playbook.

So the platform is settled by default, not by debate:

  • Expo (RN) + TypeScript, expo-router, expo-sqlite, local-first — the house stack.
  • AI keys go in a Cloudflare Worker proxy, never the client bundle — the proxy pattern is already built and generalized.
  • EAS Build → TestFlight → App Store — account, signing, provisioning, aspen-frost EAS owner already exist.
  • CI/CD + rollback — Cassian's pipeline copies in.

None of that needs an Andy decision. It's inherited. That deletes v1's Q1, Q2, Q9, and most of Q6/Q10.

What's left is the small set of questions that are actually specific to Frost Rocks. Five of them.

---

The Five Frost-Rocks-Specific Questions

Q1. Offline AI behavior — confirm the deferred-queue contract for this app.

The playbook sets the default: capture never depends on the network; anything needing a call is queued needs-context and drained when online. For Frost Rocks this is more central than it was for Frost Fitness, because Andy named bad signal as the #1 trip constraint and the whole app is used in the field.

The contract I'd build: photo + note + guess + confidence + location + timestamp + event tag all save instantly, always, offline. "Ask the guide" with no signal → queued, with honest copy ("Saved. I'll add geology context when you're back online"), drained over hotel wifi that evening. The instructional design wants this delay — attempt-first means Andy observes before the app explains.

Decision needed: confirm this is the behavior you want. My strong lean: yes, and we lean into "the guide catches up tonight" rather than apologizing for it. (This is the one genuinely architecture-shaping answer.)

Q2. Encounter data model — confirm the minimal-knowledge-model cut.

The design doc gestures at a hidden "user knowledge model" (concepts, confusion pairs, diagnostic features, FSRS scheduling). For a one-user trip prototype, building that graph is premature — same "fine for personal, not yet" judgment the Frost Fitness architecture review made repeatedly.

My lean: store the rich encounter record, but the knowledge layer is just theme tags + uncertainty status (unknown/possible/likely/identified) + simple review state (soon/later/know-it). Defer the confusion graph and FSRS to a real MVP, after we know review even happens. Decision needed: agree we defer the graph + FSRS for Finland?

Q3. Content pack — who writes it, what format, by when?

This is the real critical-path risk, and it's not code. The app is hollow without the Finland starter pack (Petra scoped 8–12 themes, 20–30 cards, 10–15 prompts, glossary). It must exist before you fly. Format is settled by the playbook (Markdown/JSON bundled at build time).

Decision needed: confirm Petra owns authorship (Izzy reviews) and set a hard internal deadline ~1 week before departure so Jesse can bundle it. This is the thing most likely to silently slip.

Q4. Frozen v0 scope — and is AI in the first TestFlight build or a fast-follow?

The platform being free doesn't make the timeline free — Finland is ~1 month out. I want an explicit cut line.

Build-it: Finland trip, photo capture (camera + roll), auto location/timestamp, note + guess + confidence + event tag, offline save, timeline + simple map, encounter detail, manual learning note, uncertainty status, basic review, local export.

**My recommended cut for the first build: ship capture + review bulletproof** first; AI learning notes as the very next build once the proxy route + content are proven. Also defer: prompt-of-the-day, compare-two-encounters, heavy analytics, end-of-day reflection as a separate surface.

Decision needed: agree with the cut line, and specifically — AI in first TestFlight build, or fast-follow? (My lean: fast-follow.)

Q5. Durability for a trip where data loss is the unforgivable failure.

Local-first means the device is the only copy. The playbook's cheap mitigation: write captured photos to the device camera roll (survives in iCloud Photos even if the app dies) + a one-tap "export my trip." A few hours of work; no backend.

Decision needed: include this safety net? (My lean: clearly yes — the brief calls data loss the one thing we can't do to your vacation.)

---

Arya's Bottom Line (v2)

The platform questions are gone — inherited from Frost Fitness via the new playbook. What remains is five small decisions, and only Q1 (offline-AI contract) actually shapes the architecture. The rest are scope and ownership calls.

At the time, these five answers were intended to feed FINLAND-ARCHITECTURE.md and the standard pipeline. That path has since been superseded by Andy's one-shot Build 1 authorization and the active post-TestFlight reconciliation path.

Timeline reality is unchanged: ~4 weeks is tight, and the way we make it is shipping capture + review rock-solid first with AI as the immediate fast-follow. A notebook that never drops a single Finland encounter beats a clever app that loses your data on a ferry. Reliability beats cleverness.

The Frost Fitness build already mapped this territory. I should have read the map before asking for directions. Map's now generalized so the next app — and the one after — starts from it.
