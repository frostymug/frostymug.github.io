1. Feasibility answers (spec §8)

Q1 — D1 capture inversion clean? Yes, with one gotcha. capture.tsx is registered as a modal (presentation: 'modal' in _layout.tsx). Today FAB does router.push('/capture'). New flow: FAB handler calls takePhoto() directly, then on a real photo creates a photo-only encounter and routes to /capture?id=<id>&mode=enrich. The gotcha is lifecycle, not routing: takePhoto() is async and fires a permission prompt on first run — the FAB must show a pending state and be re-entrancy guarded (ignore taps while a capture is in flight) so a double-tap can't create two encounters. Two-way door 🚪➡️🚪.

Q2 — GPS sequencing (the trap Izzy flagged) — lifecycle issue kicking off from FAB? No lifecycle issue, and this is the sharpest part of the plan. Today location acquisition starts on capture.tsx mount. In the inverted flow the camera opens first, so we start it earlier — but not naively from the FAB tap. The subtlety the spec undersells: takePhoto() currently only requests camera permission; getCurrentLocation() requests location permission separately. If we fire both at the exact same instant on first run we stack two OS permission dialogs and the shutter races the fix. Solution in §3 (D1): kick getCurrentLocation() off from the FAB handler in parallel with takePhoto(), stash the promise, and await (with the existing 8s timeout) at encounter-create time — which is already after the shutter, so there's natural slack. Null fix = un-pinned encounter, save never blocks (offline contract holds). Two-way door 🚪➡️🚪.

Q3 — reuse the current capture form as the post-shutter enrich view? Yes — it's ~90% reusable and reuse is the right call. capture.tsx becomes two modes driven by a mode/id param: (a) enrich (has id, post-shutter, common path) loads the just-created encounter and its save becomes an updateEncounter(); (b) manual (no id, the library/back-out fallback) keeps today's create-on-save behavior. The photo-row stays as the escape hatch (add-more / library). No reason not to. Two-way door 🚪➡️🚪.

Q4 — events table + export addition — disturb the migration/test surface? No. The migration system is clean and versioned (db_version table, atomic per-version transactions). D3 is a version: 2 entry appended to MIGRATIONS — additive, never touches v1. Adds one table (events), one insert helper (logEvent), and an events.json builder that rides the existing pure-export seam in src/db/export.ts. Bumps EXPORT_VERSION 1→2. Migration idempotency + a round-trip logEvent/export test cover it. Two-way door 🚪➡️🚪 (see §4 for the one 🚪🧱 sub-decision: event vocabulary is a soft contract).

Q5 — D5 offline map, the big one — lightest Expo-compatible offline approach? Decided in §5. react-native-maps with Apple Maps (iOS) is wrong for offline Finland; the right answer given this repo's constraints is @maplibre/maplibre-react-native with a bundled offline style + a small MBTiles/PMTiles pack of Finland, OR — if we want to ship this week with zero native-map risk — an interim "poor-man's map": a static bundled map image with GPS pins projected on top. Recommendation: ship the interim static-map for the Finland TestFlight, land MapLibre as the fast-follow. Rationale + trade-off in §5. This is the one 🚪🧱 in the delta.

4. D3 — Events (do this second; D1 instrumentation depends on it)

  • Migration v2 appended to MIGRATIONS: CREATE TABLE events (id TEXT PRIMARY KEY, ts TEXT NOT NULL, type TEXT NOT NULL, data_json TEXT). Additive; v1 untouched.
  • logEvent(type, data) helper — id via uid() (playbook rule; never Math.random()), ts = nowIso(), data_json = JSON.stringify(data). Offline-only, no network, fire-and-forget (must never throw into the capture path — wrap and swallow like the camera-roll net).
  • Export: add events.json to the bundle; bump EXPORT_VERSION 1→2 in src/db/export.ts. Pure-function seam stays pure (pass events in, don't read the DB inside the builder).
  • Event vocabulary is a soft contract 🚪🧱-ish: once Finland data is collected under a set of type strings, renaming them fragments the dataset. So we lock the vocabulary from Izzy's §4.2 table now and don't churn it mid-trip. Not database-irreversible, but analysis-irreversible — treat the names as a contract.
  • Gate #1 evidence: migration-idempotency test (run migrations twice, assert v2 applied once) + logEvent→export round-trip test (log 3 events, export, assert events.json shape).

5. D5 — Offline map (my call, the one 🚪🧱)

**Constraint that drives everything: this is Expo managed workflow (CNG — no ios//android/ dirs). Any map library must ship as a config plugin and survive expo prebuild. And Finland means no signal** — tiles cannot be fetched live.

Options I considered:

OptionOffline-capableManaged-workflow fitEffortRisk
react-native-maps (Apple/Google)❌ Apple/Google tiles are online-only; offline tile support is poor/nonePlugin existsMedWrong tool — can't do offline Finland
@maplibre/maplibre-react-native + bundled MBTiles/PMTiles✅ True offline vector/rasterConfig plugin, works with prebuildHigh — tile pack prep, style JSON, plugin config, real native buildMedium — the honest long-term answer
Interim: static bundled map image + projected GPS pins✅ Trivially offline (it's an asset)✅ Zero native depsLowLow — "simple/pixelated" is exactly Andy's stated bar

Decision 🚪🧱: ship the interim static-map for the Finland TestFlight; land MapLibre as the fast-follow.

  • Why: Andy explicitly said "even if it's a simple pixelated whatever, a real map of real places" and "put a pin on the map." A static raster of the Finland region (Helsinki + national-park areas) with encounter pins projected via a fixed lat/lng→pixel transform fully satisfies the stated intent for the trip, ships this week, and carries zero native-build risk on the timeline that matters. The pin logic is identical either way (encounters already store lat/lng) — so the interim is not throwaway; it's the pin layer we'll reuse.
  • Why not go straight to MapLibre: it's the correct destination, but it's a real native dependency landing days before a trip. That's the exact "plausible-but-unrun native slice fails on device the night before" risk. We de-risk by shipping the sure thing and treating MapLibre as the earned upgrade — same discipline as the histogram (§5.0 of the spec): ship the honest-simple version, earn the heavy one.
  • The 🚪🧱 part: the decision to add a native map library is the irreversible/expensive one (bundle size, prebuild, tile licensing). By deferring it we keep that door closed until after Finland tells us the map door (D2 Door A) actually gets used. If Finland telemetry shows nobody opens the map, we never pay the MapLibre cost.
  • Tile licensing note for whoever builds MapLibre later: OpenStreetMap-derived tiles (via a provider like Protomaps/PMTiles or a self-rendered MBTiles pack) — attribution required, check the ODbL terms. Flagged so it's not a surprise at prebuild time.

  • Asset DELIVERED & verified (Izzy, 2026-07-10): assets/map/finland-map.png (1024×1208, field-guide styled) + assets/map/README.md with the load-bearing bounds contract. Corner contract: NW = 64.0°N/23.0°E → px(0,0); SE = 59.5°N/31.0°E → px(1024,1208). Arya ran the 7 fixture points through the projection: all pass — projection math and stated bounds agree exactly (gate #1 for the seam, verified without a device). Jesse: bundle the PNG, mirror projectCoord from the asset README verbatim, use those exact fixtures as the test.
  • MapCard (compact, on timeline home) + full MapScreen (new route app/map.tsx, register in _layout.tsx).
  • Project each encounter's lat/lng → pixel via linear interpolation between the known corner coords. Pin tap → /encounter/[id] or /review?door=map.
  • Gate #1 evidence: a pure projectCoord(lat, lng, bounds) → {x, y} function with a test asserting known coords land at known pixels (corners + center). This is the load-bearing seam and it's trivially testable without a device.
