2. D1 — Camera-first two-tap capture (the real work)

2.1 What v0 does today (the gap)

app/capture.tsx is form-first, not camera-first. On open you see a scrolling form; the photo is one field among many, offered as two equal buttons — "Camera" and "Library" (addPhoto('camera') vs addPhoto('library')). The camera only launches after you choose it. Then you fill fields, then tap Save. That's roughly 3+ taps to a photo and a decision (camera vs library) in the fast path — exactly the friction Andy wanted gone.

src/services/photos.ts::takePhoto() already launches the device camera directly via ImagePicker.launchCameraAsync — so the camera launch primitive exists; it's just not the entry point.

3. D2 — Three review doors

3.1 What v0 does today

One door. Home (app/index.tsx) has a review-count stat that pushes /review (testID="button-open-review"). app/review.tsx takes no source/door param. There is no map door and no per-encounter "revisit this" door. (Encounter detail exists at app/encounter/[id].tsx but doesn't launch review.)

3.2 Target

Three distinct entry points into review, each tagged as a separate event (§4):

Pins come from encounter lat/lng.

  • Door A — map pin (spatial recall). Now buildable — Andy approved a real map (D5).
  • Door B — per-entry "↻ Revisit this one" from encounter/[id].tsx (temporal recall).
  • Door C — nav/home review stat (deliberate practice). Already exists — just instrument it.

3.3 Implementation notes

(§4), otherwise identical destination. No data change.

  • app/review.tsx: accept a door param (map|timeline|nav) via useLocalSearchParams, log it
  • Door B: add a "Revisit this one" action on encounter/[id].tsx → router.push('/review?door=timeline&id=…').
  • Door C: annotate the existing home stat press with door=nav.

3.4 Design note — these are three intents, not one feature with three doors

Map/timeline = serendipitous re-encounter; nav = deliberate practice. This matters downstream: deliberate practice (Door C) is the natural home for discrimination-first confusion-pair interleaving (train the eye, not quiz the memory) when Andy greenlights that fork. Not in this delta — flagged so we don't design the nav door in a way that blocks it later.

**The point of shipping all three is to measure which Andy uses (§4), not to decide now.**

5. D4 — Field-guide skin + D5 map (cosmetic + one new dependency)

5.0 Histogram population — the sharp question Andy surfaced (design-owned)

Andy on FR-V2-05: "leave the bars as-is, but the question is how do those bars get populated?" This is not a UI decision — it is a measurement decision, and it is the door back into the confusion-pair / discrimination fork parked 2026-06-30.

The bars claim to show understanding of granite / gneiss / glaciation. Signal options:

and it is the confusion-pair engine ("train the eye, not quiz the memory").

  • Count-based (# encounters logged per theme) — cheap, dishonest (logging ≠ understanding).
  • Self-report (existing confidence + review_state) — better, but users are poorly calibrated.
  • Discrimination-based (can Andy tell granite from gneiss side-by-side?) — the honest signal,

The honest version of this histogram requires the discrimination machinery we deliberately deferred. So this bar is a fork trigger, not a small feature.

Recommendation for Finland (Izzy): populate the bars from the honest-but-simple signal we already have (self-report confidence + review outcomes), label them truthfully as "your sense of it," NOT a hard mastery score. Then use Finland telemetry (does review actually happen? — D3 review_open / revisit_done) to earn the confusion-pair fork afterward. If review has oxygen on the trip, that's the evidence that justifies building the real discrimination engine. If it doesn't, we saved ourselves from over-building. No new schema needed for the interim version.

> DECISION (Andy, 2026-07-10): interim simple signal approved for Finland. Ship the > honest-but-simple bars now; do not build the discrimination engine before the trip.**
