Frost Rocks — Field Fixes Implementation Plan (Build 2 follow-ups)

Arya (Architect) | 2026-07-21 | For: Jesse (implementation) Sources: Petra's BACKLOG.md (2026-07-21), Andy's scope answers (2026-07-21, msg thread), Andy's Wispr Flow screenshots, code inspection of aspen-frost/frost-rocks@main. Status: APPROVED SCOPE — Andy confirmed 2026-07-21 (static map pan/zoom; photo zoom on both surfaces; Wispr bug evidence attached). Ready for implementation.

> Door tags: 🚪➡️🚪 = two-way door (cheap to reverse; default chosen, move on). 🚪🧱 = one-way door (expensive/irreversible; trade-offs laid out, Andy's call).

---

0. TL;DR

Three field-driven fixes from Andy's Finland TestFlight usage, in priority order:

  • Capture keyboard bug (P1 reliability): tall third-party keyboards (Wispr Flow dictation) collapse the capture screen to a blank view, and dismissal silently discards typed notes. Fix the keyboard layout AND add a dirty-state guard. Two bugs, one PR.
  • Photo pan/zoom (both surfaces): one reusable full-screen photo viewer (pinch/pan/double-tap/dismiss), wired into review flow and encounter detail.
  • Map pan/zoom (static approach): pinch/pan on the existing bundled map in the full map view, pins stay anchored, higher-res asset to buy zoom range. Home compact card unchanged. MapLibre explicitly deferred.

All gesture work shares one zoom/pan core (two real consumers arrive simultaneously — the abstraction earns its place under the rule of three's second-consumer trigger). Three PRs, sequenced. Every PR passes the two hard review gates: evidence the changed seam ran and diff matches the story.

---

1. What we inherit (do not re-litigate)

  • FINLAND-ARCHITECTURE.md and the Aspen Mobile App Playbook govern where this doc is silent.
  • Stack: Expo SDK 54 / RN 0.81 / expo-router v6. react-native-gesture-handler (~2.28) and react-native-reanimated (~4.1) are already dependencies — all gesture work uses these. No new libraries without a named reason.
  • Petra's backlog items are the product spec (problem statements, ACs, event names). This doc is the technical plan against them; where they conflict, flag it, don't guess.
  • Testing doctrine from frost-fitness applies: Jest is not the runtime. Runtime-shape concerns (keyboard heights, gesture handlers) need device evidence, not just green unit tests.
  • Spec-conformance lane: capture, mapProjection, and photos are protected paths. Every PR body declares scope claims (does-not-touch) honestly.

---

2. Key decisions

- Decision: add pinch/pan to the existing bundled PNG map. Ship a higher-resolution asset to make zoom useful. Do NOT add a tile-based map engine this pass. - Constraint (the why): the current asset is 1024px wide over ~450km of Finland (~440 m/px). Static zoom gives regional orientation ("which side of the lake"), and Andy confirmed that is the job for this pass. Cluster-level separation (~50 m) is physically impossible on a static country map at any sane asset size — acknowledged and accepted. - Revisit-trigger: field evidence that distinguishing near-adjacent encounters matters (Petra's pin-cluster follow-up signal), or a trip outside Finland (the bundled asset is Finland-only). Either triggers the MapLibre/expo-maps roadmap item.

- Decision: extract usePinchPanZoom (gesture composition + clamp math) used by both the photo viewer and the map. Clamp math lives in pure functions (src/utils/zoomMath.ts) so it's unit-testable without a device. - Constraint (the why): two real consumers exist simultaneously — this is the second-consumer trigger, not premature abstraction. And isolating gesture/viewport math from FinlandMap internals is exactly what Petra's AC demands for a clean MapLibre swap later. - Revisit-trigger: if photo and map semantics diverge hard during implementation (different clamping, different dismiss gestures), split into two hooks sharing only zoomMath — don't bend one hook to serve two masters.

- Decision: dismissing the enrich sheet (swipe-down or "Skip for now") with unsaved typed text applies the note to the encounter (it already exists; the user's words are strictly better kept than deleted — and Petra's note-edit backlog item provides the recovery UI if they want to change it). Dismissing a new capture with typed content prompts (Save / Discard) since silent save would create an encounter. - Constraint (the why): Andy lost a field note to a swipe-down. The field-notebook thesis is "forgiving in the field"; silent discard is the least forgiving possible behavior. - Revisit-trigger: Petra's UX review of this PR, or field evidence that auto-applied notes surprise users.

  • Static map pan/zoom, not MapLibre (🚪➡️🚪 — Andy's call, 2026-07-21)
  • One shared zoom/pan gesture core, two thin wrappers (🚪➡️🚪)
  • Dirty-state guard: auto-save in enrich mode, prompt in new-capture mode (🚪➡️🚪 — Petra may override the UX)
  • Keyboard fix approach: native insets first, KAV only if evidence demands (🚪➡️🚪) — see §4.1 implementation notes.

3. Work Item 1 — Capture screen: tall-keyboard collapse + silent note discard

Backlog ref: "P1 Bug — Blank screen after Wispr Flow keyboard handoff." Priority: first. This is a field-reliability bug in the core loop.

3.1 What the evidence actually shows (re-read of the bug)

Andy's screenshots (2026-07-21) revise Petra's initial app-switch framing:

  • The 7:01 screenshot shows the "New encounter" header still rendered with Wispr Flow's dictation panel (a custom keyboard, ~55% of screen height) active. The app never backgrounded at that moment and never crashed. The content area is blank cream.
  • The blank screen is not an app-lifecycle/restoration failure. It is a layout collapse: app/capture.tsx wraps the screen in KeyboardAvoidingView behavior="padding" (iOS) with no keyboardVerticalOffset. A keyboard reporting ~600–700px of height leaves the inner ScrollView with ≈0 rendered height. Nothing but styles.flex background remains visible.
  • Petra's instinct was right: not Wispr-specific. Any tall custom keyboard (dictation panels, some ML keyboards) triggers it. Wispr is just the first one Andy used in anger.
  • Second failure, same incident: Andy swiped the sheet down to escape; his note state lived only in component useState and was discarded. Skip for now (router.back()) has the same hole. The encounter saved with no note.
  • (The separate screenshot of the Wispr app onboarding — "swipe right to continue" — is the one genuine app-switch in the flow. Returning from it was not the blank-screen moment; deprioritize the lifecycle theory but keep the foreground-transition logging below as cheap insurance.)

3.2 Required behaviors

  • B1 — No collapse: with any keyboard height up to ~75% of screen height, the capture screen's content remains rendered and scrollable. The screen must never present as blank while the app is foregrounded.
  • B2 — Focused field stays reachable: while typing/dictating into the note or guess field, the focused field scrolls into the visible region above the keyboard (standard keyboard-aware behavior).
  • B3 — No silent discard: no dismissal path (swipe-down, Skip, back) may throw away non-empty typed text. Enrich mode: auto-apply per §2. New-capture mode: prompt.
  • B4 — Typed text survives keyboard switching: switching keyboards (globe key → Wispr → back) must not reset field state. (Expected free — state is in React — but assert it in the test so a regression is loud.)

3.3 Implementation notes (guidance, not gospel)

  • Primary path: remove the KeyboardAvoidingView; use the ScrollView's native automaticallyAdjustKeyboardInsets (iOS, supported in RN 0.81) plus keyboardDismissMode="interactive". This delegates inset math to iOS, which handles nonstandard keyboard heights (exactly our failure) far better than JS-side padding.
  • Fallback path (if the native prop misbehaves in the modal presentation): keep KAV but with useHeaderHeight()-derived keyboardVerticalOffset, and add a minimum-content-height guard so padding can never crush the ScrollView below a floor.
  • Decide by evidence: build a dev client, install/enable a tall custom keyboard, and watch it. Log Keyboard.addListener('keyboardWillShow') heights during diagnosis (temporary logging; strip or downgrade before merge).
  • Dirty guard: track a dirty flag (any of note/guess/confidence/event/themes changed since load). Intercept dismissal with navigation.addListener('beforeRemove', ...) (react-navigation under expo-router): enrich → apply note via existing updateEncounter() path, then allow dismissal; new-capture → Alert with Save / Discard. Instrument capture_dismiss_dirty_saved / capture_dismiss_dirty_discarded.
  • Keep the diff surgical. capture.tsx is a protected path; the PR story must be "keyboard layout + dismissal guard," and the diff must contain exactly that.

3.4 Acceptance criteria

  • With Wispr Flow dictation active on device, the capture/enrich screen shows its content (Petra's AC, restated against the real cause). No blank state.
  • Focused note field visible above the keyboard while typing with (a) system keyboard, (b) a tall custom keyboard.
  • Swipe-down dismissal of the enrich sheet with typed note → note is on the encounter (verify in encounter detail).
  • "Skip for now" with typed note → same guard behavior.
  • New-capture dismissal with typed content → Save/Discard prompt; both branches behave.
  • Existing capture tests stay green; no changes to save/DB semantics beyond the guard's updateEncounter call.
  • Foreground/background transition logging around the capture screen (Petra's AC) — cheap insurance for the lifecycle theory we're deprioritizing.

3.5 Evidence gate (what the PR must show)

  • Jest/component: simulated keyboardWillShow with heights {300, 450, 700} → content container height stays above floor; dirty-guard unit tests for both modes (this seam is fully testable — beforeRemove handler is a function you can invoke).
  • Maestro: extend save-note-only-encounter.yml lineage — type note → swipe dismiss → reopen encounter → note text present.
  • Device evidence: screen recording with an actual tall keyboard on a dev build, attached to the PR. Jest cannot prove this fix; the recording is the "evidence it ran" gate for the changed seam.
  • Final acceptance: Andy repeats his exact Wispr flow on the next TestFlight build. That's the only gate that fully closes the ledger.

---

4. Shared foundation — zoom/pan core (lands inside PR 2)

  • src/utils/zoomMath.ts: pure functions — clampScale(scale, min, max), clampTranslation(tx, ty, scale, content, viewport), doubleTapTarget(current). Fully unit-tested; this is where the gesture math correctness lives, because Jest can't drive a pinch.
  • src/hooks/usePinchPanZoom.ts: composes Gesture.Pinch() + Gesture.Pan() + Gesture.Tap(2) via gesture-handler v2 API with reanimated shared values; exposes an animated transform + reset(). Config knobs: min/max scale, bounds mode, double-tap levels.
  • Verify GestureHandlerRootView wraps the app root (expo-router usually provides it; check, don't assume — if absent, add it in _layout.tsx, one line).
  • Photos and map consume this with different knobs. If their semantics fight, split per §2's revisit-trigger.

---

5. Work Item 2 — Photo pan/zoom, both surfaces

Backlog ref: "P1 — Zoom into encounter images during review." Andy widened scope 2026-07-21: review flow AND encounter detail.

5.1 Design

  • New src/components/PhotoViewer.tsx: full-screen modal overlay (RN Modal, dark backdrop) — not a navigation push, so underlying screen state (review progress, scroll position) is trivially preserved, which satisfies Petra's "return without losing progress" AC by construction.
  • Behaviors: pinch zoom (1x–6x), pan while zoomed (clamped to image bounds), double-tap toggle (1x ↔ ~2.5x at tap point if cheap, center otherwise), swipe-down dismiss only at 1x (avoid gesture fights), explicit close button (44pt target — field gloves).
  • Image inspection, not gallery (Petra's product note): no filters, no share sheet, no chrome beyond close.
  • URIs through resolveLocalPhotoUri() — same rebasing rules as everywhere else. Fully offline.
  • Multi-photo encounters (explicit, per Petra's AC): tapping a thumbnail opens that photo. No in-viewer paging/carousel in v1 — see §8.

5.2 Wiring

  • app/review.tsx: encounter image tappable → viewer. Dismiss returns to the same card/prompt state (assert, don't assume). Events: review_image_zoom_open / review_image_zoom_close.
  • app/encounter/[id].tsx: each gallery thumbnail tappable → viewer on that photo. Events: encounter_image_zoom_open / encounter_image_zoom_close.

5.3 Acceptance criteria

Petra's review-zoom ACs, plus: works identically from encounter detail; one-photo and multi-photo encounters both behave (multi = per-thumbnail open, no paging); review card state identical after open/close; offline with local photos.

5.4 Evidence gate

  • zoomMath unit tests (clamping, double-tap targets — the correctness core).
  • Component test: viewer opens with resolved URI, close restores prior tree.
  • Maestro: review flow → tap image → viewer visible → close → assert same review card content (this is automatable and is the state-preservation proof). Second flow for encounter detail.
  • Device recording of pinch/pan/dismiss on real photos in the PR.

---

6. Work Item 3 — Map pan/zoom (static, full map view only)

Backlog ref: "P1 — Zoom and pan the map view." Andy confirmed static approach 2026-07-21.

6.1 Design

  • app/map.tsx full-view FinlandMap gains pinch/pan via the shared core: scale 1x–4x, pan clamped so the map never detaches from the viewport, double-tap toggles 1x ↔ 2.5x, and a small reset control ("fit trip" — Petra's "return to default without getting lost" AC; don't rely on gesture-only reset).
  • Transform strategy: image + pins live in ONE animated container so pins stay anchored by construction (no per-pin math on gesture frames). Counter-scale the pin visual (inverse scale on the pin core) so pins don't balloon at 4x; effective hit target stays ≥44pt at all zoom levels.
  • Pin taps after zoom/pan must still route to the map-review door (door:'map') — regression-assert this; it's the map's actual job.
  • Isolation for MapLibre (Petra's AC, load-bearing): mapProjection.ts untouched. Gesture/viewport state lives in the shared hook + a thin MapViewport wrapper inside FinlandMap. The later native-map upgrade replaces FinlandMap internals; callers and projection tests don't move.
  • Home compact card: unchanged. Glanceable per Izzy's v2.2 compress note. Full map view is the only zoom surface.
  • Events: map_zoom_start, map_zoom_end (+ max scale if lightweight), optional pin-tap-after-zoom.

6.2 Higher-res asset (parallel task, not a blocker)

  • Current: assets/map/finland-map.png 1024×1208. Target: 2048×2417 re-export (bundle-size delta noted in the PR; expect low single-digit MB for a stylized map — acceptable).
  • Source lives in projects/frost-rocks/source-material/ or with Izzy — Izzy owns the re-export (visual style must survive upscaling; do NOT naive-upscale the PNG, that's decoration pretending to be resolution).
  • Zoom ships regardless; without the new asset it just goes soft past ~2x. Constants (FINLAND_MAP_WIDTH/HEIGHT) update with the asset — projection math is resolution-relative already; existing mapProjection.test.ts must stay green un-edited (if the asset change forces test edits, the projection wasn't resolution-independent and that's a finding, not a nit).

6.3 Acceptance criteria

Petra's map ACs verbatim, plus: pins visually constant-size across zoom; reset control returns to trip-level view; compact card behavior untouched; offline unaffected (bundled asset).

6.4 Evidence gate

  • zoomMath clamp tests cover map-bounds mode (shared with §5 — one suite, two consumers).
  • Component test: pins remain at projected positions under a mocked transform state.
  • Maestro: open map → (gesture support permitting) zoom → tap pin → map-review opens. If Maestro's pinch support proves flaky on iOS, the pin-tap-after-transform assertion moves to a component test + device recording — don't burn days fighting the harness; say so in the PR instead.
  • Device recording: pinch, pan, double-tap, reset, pin tap at 4x.
