2. What's different about Frost Rocks (the five decisions)

2.1 Offline AI contract — the one architecture-shaping call (Q1: confirmed)

The whole app is used in the field, and Andy named bad signal as the #1 constraint. So the offline contract isn't a nice-to-have edge case — it's the spine. See §3 for the concrete design. Headline: capture is a synchronous local write that can never fail on connectivity; AI enrichment is an asynchronous queued job. UI copy is honest and calm ("Saved. I'll add geology context when you're back online"), never a spinner or an error. The instructional design wants this gap — attempt-first means Andy observes and guesses before the guide explains.

2.2 No knowledge model for v0 (Q2: defer)

The design doc gestures at a hidden user-knowledge model (concepts, confusion pairs, diagnostic features, FSRS). Deferred. For a one-user trip prototype, that graph is premature — we don't yet know if review even happens (it's a core learning goal to test, not assume). The v0 knowledge layer is deliberately flat:

2.3 Content pack → thin offline scaffold (Q3: Andy challenged — here's the reconciliation)

Andy's question was the right one: "if we have robust AI, why a content pack? Can't the user just capture a photo and dialogue with the AI?" The honest answer is that a big curated pack is not needed, but a thin offline scaffold is — for two reasons the "just dialogue with AI" model can't cover:

Decision: thin offline scaffold (≈6–8 observation prompts + small glossary) = REQUIRED in v0. Full curated 20–30-card pack = deferred to should-have. This also removes the biggest non-code critical-path risk before the trip — nobody has to author a big pack to ship. Format per playbook: Markdown/JSON bundled at build time, no runtime fetch. Ownership: Petra authors the thin scaffold, Izzy reviews; small enough to not gate the build.

2.4 AI is a fast-follow (Q4: confirmed, with Andy's caveat)

First TestFlight build = capture + review, bulletproof and AI-free. The AI learning-note path (proxy route + enrichment worker + drain loop) is the immediate next build. Andy will try to land it before Finland; architecture supports that by building the proxy route early and keeping it off the release-blocking path. Because AI enrichment is already an async queued job (§3), adding it later is additive — no refactor of the capture path. That decoupling is exactly what makes the fast-follow cheap.

Also deferred out of the first build (should/could, per Petra's ladder): prompt-of-the-day, compare-two-encounters, heavy analytics, and a separate end-of-day-reflection surface (the reflection field exists on the encounter; the dedicated surface is later).

3. The offline enrichment queue (the concrete design)

This is the only genuinely novel mechanism in the app, so it gets specced properly.

Two independent write paths:

location + timestamp + event tag + theme tags → one INSERT into encounters in local SQLite. No network in this path at all. This is the promise: capture cannot fail because of signal.

enrichment_jobs row referencing the encounter, status queued. We do not block the UI.

  • Enrichment (asynchronous, queued): if Andy asks for AI help ("ask the guide"), we enqueue an

The drain loop:

where status ∈ {queued, failed and attempts < N}, oldest first.

scaffold context (§2.3), carrying an idempotency key** = the job's UUID.

failure (network/5xx/429): leave queued/failed, increment attempts, honor Retry-After; never retry-storm. On permanent failure (4xx that isn't 429): mark dead, surface a quiet "couldn't add context — tap to retry."

  • A background task (on app foreground + on connectivity-regained event) selects enrichment_jobs
  • For each, it calls the Worker proxy /enrich route with the encounter payload **+ the grounding
  • On success: write the returned learning note to the encounter, set job done. On transient

Why the idempotency key matters here: the drain loop can fire on a flaky connection that succeeds server-side but drops the response before the client sees it. Without idempotency, the next drain double-charges the AI call and can double-write the note. The Worker dedupes on the key (short-lived cache is enough at single-user scale). This is the correctness subtlety of the whole app, which is why the playbook's UID rule (§1) is load-bearing, not ceremony: the idempotency key is a key that could one day sync, so it must never come from Math.random().

Ordering / conflict: single user, single device for v0 → no merge/CRDT concerns. If Andy edits his own note while a job is in flight, the enrichment writes only to the AI-note field, never Andy's note field — the two never collide by construction.

4. Data model (v0)

Deliberately rich record, thin knowledge layer — so the deferred FSRS/graph can land later without a schema rewrite.

trips
  id (uuid) | title | start_date | end_date | created_at

encounters
  id (uuid) | trip_id -> trips.id
  captured_at | lat | lng | location_label
  event_tag                       -- "Helsinki walk", "ferry day", etc.
  note                            -- Andy's "what did you notice?" (his words, never AI-written)
  initial_guess                   -- attempt-first
  confidence                      -- sure | kind_of | guessing
  uncertainty_status              -- unknown | possible | likely | identified
  ai_learning_note                -- filled by enrichment; nullable; never overwrites `note`
  learned_later                   -- Andy's "what I learned later" reflection
  review_state                    -- soon | later | know_it
  created_at | updated_at

encounter_photos
  id (uuid) | encounter_id -> encounters.id
  local_uri | camera_roll_asset_id   -- proves the camera-roll safety net (§2.5)
  width | height | created_at

encounter_theme_tags
  encounter_id -> encounters.id | theme   -- glacial | bedrock | shoreline | erratic | built_stone | ecology

enrichment_jobs                    -- the outbox (§3)
  id (uuid)                        -- doubles as the AI idempotency key
  encounter_id -> encounters.id
  status                           -- queued | done | failed | dead
  attempts | last_error | retry_after_ts | created_at | updated_at

ai_call_log                        -- per playbook cost instrumentation (proxy-side mirror optional)
  id | route | model | tokens_in | tokens_out | latency_ms | status | ts

Explicitly NOT in v0 (deferred to MVP): concepts, confusion_pairs, diagnostic_features, FSRS scheduling columns (due_at, stability, difficulty, reps, lapses). The 3-state review_state stands in for the scheduler.

6. What we are explicitly NOT building for Finland

  • Full AI rock identification / a complete rock database.
  • The user knowledge graph, confusion matrix, FSRS scheduling.
  • Public community, social feed, gamified streaks, heavy onboarding curriculum.
  • Multi-user auth, cross-device sync, cloud backup (Rung 3+).
  • Diarium integration (we ship a clean export; integration is a fast-follow).
  • Natural-history/POI content as primary (Andy Q4: geology-focused; other context stays background).
  • Prompt-of-the-day, compare-two-encounters, heavy analytics dashboards (should/could, post-v0).

7. Timeline Reality

Finland is close enough that the architecture has already been tested by a real build. On 2026-07-02, Arya implemented Build 1 and pushed it through the EAS/TestFlight lane. Build 1 covered capture, review, camera-roll safety, export, and a dormant enrichment outbox. At handoff, the repo had 40 tests green, TypeScript clean, and Expo lint clean.

The timeline question is now what Build 2 should be, not whether Build 1 can exist. My lean: make Build 2 a tight trip-readiness update driven by Andy's TestFlight feedback, then enable AI enrichment only if it does not threaten reliability.

Known Build 1 deltas from the original architecture:

  • No Sentry yet; Arya used a simpler error boundary because org credentials were not available during the autonomous run.
  • No map tiles; the app stays closer to list/timeline than map-led exploration.
  • Export is Markdown journal + JSON manifest + photo copies, shared as artifacts.
  • Day reflection is folded into reviewing today's encounters, not a separate ceremony.
  • Scaffold content exists, but Arya authored it as a placeholder. Petra/Izzy should approve or replace it.

Biggest residual risks: (a) the enrichment queue's idempotency/retry correctness once live AI is enabled; (b) whether the current capture/review flow feels good enough on Andy's phone; (c) the thin scaffold's product voice and factual quality.
