Frost Rocks PRD v1.1 — Foundation PRs 16-20

Status: Draft for Andy/Arya/Jesse review Author: Petra Date: 2026-07-22 Scope: Five prioritized backlog items after completed field fixes PRs #12-15 and active D3 tutor work Sources: BACKLOG.md, BACKLOG-PRIORITY-2026-07-22.md, D3-WEDGE-SPEC.md, D3-IMPLEMENTATION-PLAN-2026-07-22.md, live Finland export/review notes, repo aspen-frost/frost-rocks at 249fabd

---

0. Product Context

AI tutor / D3 evening debrief is the active next major product track. Field fixes are already implemented on origin/main: Wispr/tall-keyboard capture recovery, photo zoom, and static map zoom. The remaining product risk is the evidence substrate underneath the tutor and review loop.

This PRD defines the next five backlog items as implementation slices PR #16-20:

PRFeatureSizePrimary Job
16Export encounter data with linked photo filesMMake session review/export self-contained.
17Preserve native-quality photo metadataMProtect place/time/camera context through capture and export.
18Edit note on existing encounterSAdd small recovery for incomplete captures.
19Capture user questions for later AI explorationMPromote curiosity into first-class tutor/review material.
20Dynamic themes and user-defined keywordsMReplace fixed prototype taxonomy with user-shaped tags.

Product Principle

The tutor is only as good as the field record underneath it. Photos, metadata, notes, questions, and user vocabulary are not admin details. They are the raw material of learning.

---

1. Goals

  • Make exports usable by humans and agents without manually reconstructing encounter/photo relationships.
  • Preserve the same practical capture context users expect from native iPhone photos where technically available and privacy-appropriate.
  • Keep the camera-first capture loop forgiving by letting users add/edit their own notes after the moment.
  • Treat user-authored questions as durable learning objects that can feed D3 and later tutor moments.
  • Let user tags/keywords reflect what the user actually notices, instead of forcing prototype categories.

2. Non-Goals

  • Do not redesign D3 tutor interactions in this PRD.
  • Do not build a public/social sharing format.
  • Do not add MapLibre, multi-trip, or live-map infrastructure.
  • Do not build an AI-generated taxonomy in PR #20.
  • Do not make encounter editing a full CMS in PR #18.
  • Do not use EXIF as the only source of product truth; structured app metadata remains canonical.

3. Cross-Cutting Requirements

  • All features must work with the local-first/offline app model unless explicitly online-gated.
  • Existing encounter IDs, photo IDs, and trip IDs must remain stable.
  • Exports must be readable outside the app.
  • AI-owned fields must never overwrite user-authored fields.
  • New instrumentation must pass the D3 implementation plan's event-table fix: the database cannot silently drop new event types.
  • Each PR must include a small verification artifact in its PR body: test output, sample export, screenshot, or device evidence depending on the seam.

PR 16 — Export Encounter Data With Linked Photo Files

Problem

Andy and Izzy currently have to stitch encounter JSON back together with photo files manually. That makes research/design review slower, increases mismatch risk, and blocks agent review of real field sessions.

User Story

As someone reviewing field usage, I want one export package containing encounter data and the corresponding photo files with stable links, so I can inspect the session without rebuilding the record by hand.

Requirements

IDRequirement
PR16-R1Add an export option that packages encounter data and associated photo files together.
PR16-R2Export package must include manifest.json, encounters.json, and a photos/ directory or equivalent zip/package structure.
PR16-R3manifest.json must map each encounter ID to photo file paths, photo IDs, capture timestamps when available, and metadata availability flags.
PR16-R4Photo filenames must be deterministic and collision-resistant, preferably including encounter ID and photo ID.
PR16-R5Exported encounter records must reference packaged photo paths, not only original camera-roll/local app URIs.
PR16-R6Existing JSON-only export remains available if lightweight export is still useful.
PR16-R7Package must open outside the app without access to the app database, app sandbox, or camera roll.
PR16-R8Export should preserve original/local photo files where possible; thumbnails are optional and not a substitute.
PR16-R9If a photo file cannot be included, the manifest must mark it missing with a reason instead of silently omitting it.
PR16-R10Instrument export start, export success, export failure, and export package shape/version.

Acceptance Criteria

  • Andy/Izzy can review a sample export without manually matching images to encounter rows.
  • A reviewer can open manifest.json, find an encounter, and locate all associated photos.
  • At least one sample export is attached to the PR or saved as a test fixture.
  • Export gracefully reports missing/unavailable photos.

Open Questions

  • Should the package be a zip immediately, or is a folder/share bundle enough for the next internal diagnostic pass?
  • Should export package version start at export_schema_version: 1?

PR 17 — Preserve Native-Quality Photo Metadata

Problem

Photos taken through Frost Rocks and viewed in Google Photos appear to lack expected metadata such as location, time zone context, and camera/device info. Native iPhone Camera photos preserve richer metadata by default. Frost Rocks should not silently downgrade the field record.

User Story

As a field user, I want photos captured through Frost Rocks to preserve practical native-photo metadata where available, so my encounters keep their place, time, and capture context across review and export.

Requirements

IDRequirement
PR17-R1Inspect current Frost Rocks photo files to determine which metadata exists at capture and where metadata is lost.
PR17-R2Preserve available EXIF/TIFF metadata through local save and export paths.
PR17-R3Attach GPS coordinates to the saved image file when location permission is granted and a location fix is available.
PR17-R4Preserve capture timestamp in the image metadata and encounter record.
PR17-R5Store local time zone explicitly in the encounter record/export, even if not represented cleanly in EXIF.
PR17-R6Preserve or write device/camera/lens metadata when available from the iOS capture pipeline.
PR17-R7Distinguish missing metadata because of permission/unavailable data from metadata lost during processing.
PR17-R8Export package manifest from PR #16 must expose metadata presence/absence at the photo level.
PR17-R9Do not rely on image metadata as the only source of product truth; structured encounter metadata remains canonical.
PR17-R10Privacy behavior must be explicit: GPS metadata depends on location permission and should not be fabricated.

Acceptance Criteria

  • A Frost Rocks photo and a native iPhone Camera photo taken at the same place/time show comparable practical metadata in Apple Photos or Google Photos, allowing for fields unavailable to third-party apps.
  • Raw-file inspection confirms EXIF/TIFF/GPS preservation before and after export.
  • If metadata cannot be preserved because of framework limitations, the PR documents the limitation and stores the equivalent structured metadata in the encounter/export.

Open Questions

  • Minimum parity: GPS + timestamp only, or GPS + timestamp + camera/device/lens fields?
  • Does the current Expo photo/camera path expose enough metadata, or does this require lower-level capture work?

PR 18 — Edit Note on Existing Encounter

Problem

The app supports fast capture, but if a user saves photo-only or incomplete notes, there is no obvious narrow recovery path to add/edit their personal note later.

User Story

As a field user, I want to open an existing encounter and edit or add my note, so I can enrich an encounter after the moment without creating duplicates.

Requirements

IDRequirement
PR18-R1Add an edit/add-note affordance on encounter detail.
PR18-R2If the encounter has no user note, label the action as adding a note, not correcting a mistake.
PR18-R3Editing must update only the user's note field and updated_at.
PR18-R4Editing must not overwrite ai_learning_note, photos, GPS, guess, confidence, event tag, themes, questions, review state, or tutor residue.
PR18-R5Cancel/back exits edit mode without changing the note.
PR18-R6Works offline and persists locally.
PR18-R7The path is reachable in two taps or fewer from encounter detail.
PR18-R8Instrument encounter_note_edit_start, encounter_note_edit_save, and encounter_note_edit_cancel.

Acceptance Criteria

  • User can add a note to a photo-only encounter.
  • User can edit an existing note.
  • Cancel leaves the original note unchanged.
  • AI-owned fields remain unchanged after edit.

Open Questions

  • Should note editing use an inline field, bottom sheet, or route/modal? Petra lean: narrow modal/sheet, not full encounter edit.

PR 19 — Capture User Questions for Later AI Exploration

Problem

Andy is often capturing questions, not just observations or guesses: "Why is it like that?" "Why is that boulder alone?" Today those questions are buried in notes/guess text instead of becoming durable material for review and D3 tutor exploration.

User Story

As a field user, I want to save questions that occur to me while looking at rocks, landforms, or places, so I can revisit and explore them with the tutor later.

Requirements

IDRequirement
PR19-R1Add a first-class local question object aligned with the D3 questions table direction where possible.
PR19-R2User can mark or enter a captured note/guess as a question during capture or enrichment.
PR19-R3Questions preserve the user's exact wording.
PR19-R4Questions can be linked to an encounter and trip; encounter linkage should be nullable only if future non-encounter questions need it.
PR19-R5Questions appear on encounter detail distinctly from notes, tags, guesses, and AI learning notes.
PR19-R6Review/D3 surfaces can query open questions as candidate prompts.
PR19-R7Question status supports at least open, explored, and resolved.
PR19-R8AI/tutor responses must not overwrite the original question text.
PR19-R9Export includes questions in a stable structure distinct from notes and tags.
PR19-R10Instrument question_capture, question_review_open, question_ai_explore_start, and question_resolved if D3 does not already own the last event.

Acceptance Criteria

  • User can capture a question and see it later on encounter detail.
  • Export includes the question with exact text and status.
  • D3/review can select open questions without text-scraping notes.
  • Resolving/exploring a question updates status while preserving original wording.

Open Questions

  • Should PR #19 ship inside the D3 implementation sequence, or immediately after D3 as a capture/review model change?
  • Should question detection be manual-only at first, or should question-like note text be suggested for promotion?

PR 20 — Dynamic Themes and User-Defined Keywords

Problem

The current fixed themes/keywords are not proving useful in real field usage. Andy is noticing things the prototype taxonomy did not anticipate, and the app should not force user observations through static categories that have weak evidence.

User Story

As a field user, I want to add and reuse my own tags/keywords on encounters, so Frost Rocks reflects what I am actually noticing and lets that vocabulary become useful later.

Requirements

IDRequirement
PR20-R1User can add a custom tag/keyword during capture or enrichment.
PR20-R2User can reuse previous tags, with recent/frequent tags surfaced ahead of static suggestions.
PR20-R3Pre-filled prototype themes become optional suggestions, not a closed list.
PR20-R4Existing theme tags continue to display and export correctly.
PR20-R5Custom tags persist offline locally.
PR20-R6Export includes custom tags in a stable structure.
PR20-R7System distinguishes user-authored tags from app-suggested tags if cheap to preserve.
PR20-R8Avoid AI-generated/normalized taxonomy in this PR unless a small suggestion can be clearly separated from user-authored truth.
PR20-R9Instrument tag_add_custom, tag_select_existing, and tag_suggestion_used.

Acceptance Criteria

  • User can create a new tag and reuse it on another encounter.
  • Existing theme-tag display does not regress.
  • Export shows custom tags clearly.
  • Recent/frequent tags are visible without making capture feel like a filing chore.

Open Questions

  • Should tags be called "keywords" in the UI to feel more notebook-like and less taxonomy-heavy?
  • When the tutor later maps user tags to curriculum concepts, where should that soft mapping live?

4. Suggested PR Order

PRWhy This Order
16Export package is immediately useful for Andy/Izzy/Petra/Arya review and future agent analysis.
17Metadata preservation pairs naturally with export and prevents context loss before tutor reasoning.
18Note editing is small, useful, and reduces recovery friction after capture reliability is fixed.
19Questions are strategically important for D3/tutor, but should respect the D3 data model rather than invent a competing shape.
20Dynamic tags matter, but they should follow questions because the tutor is more likely to be anchored by questions first.

5. Risks

RiskMitigation
Export packaging gets overdesigned as public sharing.Keep PR #16 diagnostic/internal first.
Metadata preservation is blocked by Expo/camera framework limits.Start PR #17 with raw-file inspection; preserve equivalent structured metadata where EXIF cannot.
Note editing balloons into full encounter editing.PR #18 edits user note only.
Questions duplicate D3 schema.Align PR #19 with D3 questions table and delta model.
Custom tags create messy vocabulary.PR #20 starts with manual add/reuse; normalization waits for evidence.

6. Evidence Gates

  • PR #16: sample export package reviewed outside the app.
  • PR #17: raw metadata comparison against native iPhone photo.
  • PR #18: before/after encounter detail proof and field-level non-regression check.
  • PR #19: saved question appears on encounter detail and export; D3/review can query it.
  • PR #20: custom tag created, reused, displayed, and exported.

7. Product Read

This is foundation work, but not plumbing in the dismissive sense. It is the product learning how to preserve the evidence it will later reason over. The AI tutor can be charming, clever, even a little professorial. Fine. But if the field record is incomplete, unlinked, or forced through the wrong vocabulary, the tutor will inherit bad raw material.

The sequence is therefore deliberately sober: export the evidence, preserve the context, recover user notes, promote questions, then let vocabulary grow from use. Very dull. Very necessary.
