Frost Rocks v1.1 Foundation — Implementation Plan (PRs 16–20)

Status: ARYA BUILD PLAN — Jesse-ready; answers Petra's open questions inline Author: Arya (Architecture) · 2026-07-22 Inputs: FROST-ROCKS-PRD-V1.1-PRS-16-20.md (Petra) · repo aspen-frost/frost-rocks @ 249fabd (= origin/main, source-verified today) · D3-WEDGE-SPEC.md · D3-IMPLEMENTATION-PLAN-2026-07-22.md · BACKLOG.md

0. What the repo read changes about the PRD

Petra specced against the product surface; the code is further along (and in one case further behind) than the PRD assumes. Three findings reshape scope:

  • PR 16 is ~60% built. buildExportBundle (src/services/exporter.ts) already writes a self-contained folder — journal.md, trip.json, events.json, and photos/ with deterministic <encounterId>-<n>.<ext> filenames — and trip.json already maps encounters → photo filenames. What's actually missing: the folder can't leave the device as one artifact (the share sheet shares single files; Andy has only ever shipped the JSON), missing-photo entries are a silent counter instead of manifest rows, no photo IDs/flags, no instrumentation. PR 16 = zip + manifest upgrade, not a new export system.
  • PR 18 mostly exists. capture.tsx has a full enrich mode (/capture?id=X&mode=enrich) that loads an existing encounter into the form and saves note/guess/confidence/event-tag/themes via updateEncounter — with dirty-state guards and tests. It's only reachable from the home-timeline card. PR 18 = add the door from encounter detail + label polish, unless Petra insists on strict note-only editing (see Q-P1).
  • PR 17 root cause is two lines. src/services/photos.ts calls the camera and library pickers with exif: false and quality: 0.85 — metadata is never requested, and the sub-1.0 quality forces a re-encode that strips whatever survived. Also: GPS was never going to be in third-party camera EXIF on iOS — the app's own expo-location fix (already captured per encounter) is the source; we have to write it into the file ourselves. And captured_at is stored as UTC ISO (toISOString()) — local timezone is currently lost entirely, which PRD R5 correctly demands we fix.

Plus one standing item from the D3 plan that lands here first: the events.type CHECK constraint silently drops every event type added since migration v2 (all v2.2 zoom events were never recorded). Every PR in this PRD adds event types, so the fix ships in PR 16 (first in sequence), satisfying the PRD's cross-cutting requirement.

1. Cross-cutting decisions

#DecisionDoor
C1Migration ladder: v3 = events rebuild (PR 16) · v4 = tz_offset_min column (PR 17) · v5 = questions table (PR 19) · v6 = encounter_keywords (PR 20). D3's B1 migration renumbers to v7 and drops questions + the events rebuild from its scope (they land here). D3 plan updated with a cross-reference note.🚪➡️🚪
C2export_schema_version continues the existing counter → 3 (current EXPORT_VERSION = 2), answering Petra's open question. One version authority, no parallel numbering.🚪➡️🚪
C3Zip via fflate (pure JS, no native module, no EAS rebuild): JPEGs stored uncompressed (they're already compressed — level 0 is fast), JSON/markdown deflated. ~40MB trip zips fine in memory at this scale.🚪➡️🚪
C4EXIF read/write via piexifjs (pure JS, JPEG-only). HEIC files (library picks) that can't be stamped get flagged in the manifest, never silently skipped and never fabricated (PRD R7/R10).🚪➡️🚪
C5PR 19's questions table is byte-identical to the D3 plan §3 schema. It doesn't "align with" the D3 direction — it is the D3 table, landing early. Zero drift allowed; D3's delta applier consumes it as-is.🚪🧱 (shared substrate — same reasoning as the evidence rail)
C6PR 20 unifies tagging going forward: legacy theme rows are copied into encounter_keywords (source: 'theme') at migration; encounter_theme_tags stays in place (nothing dropped, old exports reproducible) but stops receiving writes. One write path forward, one read path (union not needed after migration).🚪➡️🚪 — reversible because the legacy table is preserved
C7Every PR body: evidence gate #1 (test/device output exercising the changed seam) + gate #2 (body matches diff). Petra's §6 evidence gates adopted verbatim as the per-PR artifacts.—

PR 16 — Export package (zip + manifest upgrade) — S/M

Migration v3 (ships here): rebuild events without the type CHECK (create-copy-drop-rename + reindex; SQL already written in D3 plan §3). Regression test: insert a review_image_zoom_open row post-migration and read it back — that's the live bug, caught.

Files: src/db/export.ts (manifest builders — pure, tested), src/services/exporter.ts, app/export.tsx, new src/services/zipBundle.ts. Dep: fflate.

Package structure (zip, frost-rocks-export-<stamp>.zip):

manifest.json      — export_schema_version: 3, trip, generated_at,
                     per-encounter photo map: [{photo_id, path, width, height,
                       photo_created_at, status: 'included'|'missing',
                       missing_reason?, metadata: {gps, timestamp, tz, device}  ← flags land in PR 17, keys reserved now}]
encounters.json    — full encounter records (note/guess/confidence/uncertainty/ai_note/
                     learned_later/review_state/themes→keywords/questions when PR 19–20 land),
                     photo refs by packaged path, NOT device URIs (R5)
journal.md         — unchanged human journal
events.json        — unchanged
photos/<encounterId>/<photoId>.<ext>   — deterministic, collision-proof (R4)

Implementation notes:

  • Reuse buildExportBundle's folder write, then zip the folder (read files → fflate.zipSync with per-file level: 0 for photos/, 6 for text) → single shareFile(zipUri, 'application/zip').
  • Missing photo → manifest row status: 'missing', missing_reason: 'file_not_found' (replaces the silent photoFailures counter; keep the counter for the UI line).
  • Existing three share buttons remain (R6); "Share full package" becomes the primary action.
  • Events: export_start, export_success{encounters, photos, missing, schema_version, bytes}, export_failure{stage}.

Gate: unit tests for manifest builder (incl. missing-photo row); a real sample zip from Finland data attached to the PR, opened cold on a laptop: find an encounter in manifest.json, locate its photos by path (Petra's acceptance, literally).

Answers Petra: zip immediately (a folder can't ride the iOS share sheet as one artifact — zip is the feature); version = 3 per C2.

PR 17 — Native-quality photo metadata — M (the only PR with genuine unknowns)

Step 0 is mandatory and comes first (PRD R1 is right): pull one Frost Rocks photo (app dir + camera-roll copy) and one native Camera photo off the device, run both through exiftool, and put the comparison table in the PR body. Everything below is the expected result of that inspection; if reality disagrees, the inspection wins and the plan adjusts.

Root cause (verified in source): exif: false, quality: 0.85 on both pickers + re-encode. GPS additionally requires app-side stamping (iOS never gives third-party picker captures location EXIF).

Changes (src/services/photos.ts, src/utils/dates.ts, migration v4):

- Plumbing: persistAsset needs the location fix — pass locationRef.current from capture.tsx through takePhoto(). GPS starts from the home FAB before the camera opens, so the fix is normally resolved; if not, stamp without GPS and flag (R10: no fabrication — a missing fix is a missing fix). - HEIC (library only): piexif can't write it → skip stamp, flag metadata.write_skipped: 'heic'.

  • Camera path: exif: true (harvest the dict iOS provides — DateTimeOriginal, orientation, some device fields). Keep quality: 0.85 — the re-encode is acceptable because we stamp metadata after it.
  • Library path: quality: 1 + exif: true — avoid transcoding; original bytes (and their EXIF, incl. GPS if the user's photo has it) survive the copy.
  • Stamp step in persistAsset, before the camera-roll safety-net save (ordering matters — the roll copy must carry the metadata too): merge picker EXIF dict + app GPS fix + DateTimeOriginal/OffsetTimeOriginal from device clock + Make/Model via expo-device → write with piexifjs → then createAssetAsync.
  • Migration v4: ALTER TABLE encounters ADD COLUMN tz_offset_min INTEGER — populated at capture from device timezone (R5). Exported in encounters.json. (Side note, not this PR: captured_at is UTC, so day-grouping in the timeline files a 1 a.m. Finland capture under the previous day. The column makes fixing that possible later.)
  • Manifest metadata flags (keys reserved in PR 16) go live: {gps: bool, timestamp: bool, tz: bool, device: bool, write_skipped?: reason} — distinguishing never available from lost in processing (R7).

Answers Petra: minimum parity = GPS + timestamp + tz offset guaranteed; Make/Model best-effort; lens/aperture fields are not exposed by the Expo picker pipeline — that's the documented framework limitation the acceptance criteria anticipates. Lower-level capture (expo-camera / native module) is not warranted for this; structured encounter metadata stays canonical (R9).

Gate: before/after exiftool dumps + the native-photo parity table, on device.

It's a day of extra UI for a narrower door — my view: the wider door is the better product (a wrong guess is as worth fixing as a missing note), and it's all user-authored truth. Petra's call; the plan works either way. 🚪➡️🚪
  • Events: encounter_note_edit_start/save/cancel (persist thanks to migration v3).
  • One real gap either way: enrich mode is currently invisible from detail — that's the actual bug being fixed.

  • Events: question_capture{door: capture|promote}, question_review_open; question_resolved stays owned by the D3 delta applier (R10's conditional resolved correctly — D3 owns it).
  • Status transitions to explored/resolved are D3-session-driven; this PR ships the object, the doors in, and the reads. No manual resolve UI in v1 (residue comes from dialogue, per the wedge).

Answers Petra: ship it now, in foundation order — it's D3 substrate landing early, not a competing shape (C5 makes that structural). Manual-only detection first.

PR 20 — Dynamic keywords — M (the real design-touching one)

Migration v6:

CREATE TABLE encounter_keywords (
  encounter_id TEXT NOT NULL REFERENCES encounters(id),
  text TEXT NOT NULL,                -- stored as typed; matched case-insensitively
  source TEXT NOT NULL DEFAULT 'user' CHECK (source IN ('user','suggested','theme')),
  created_at TEXT NOT NULL,
  PRIMARY KEY (encounter_id, text COLLATE NOCASE)
);
-- backfill: copy encounter_theme_tags rows in as source='theme' (label text, e.g. 'Built stone')

encounter_theme_tags stays (nothing dropped, old exports reproducible) but receives no new writes (C6).

Files: new src/db/keywords.ts (add/reuse, recentKeywords(tripId, limit) by GROUP BY normalized text ordered by recency+frequency), app/capture.tsx, src/db/export.ts, display components.

  • Capture UI: the fixed theme-chip row becomes a keyword row: recent/frequent user keywords first (R2), the 6 legacy themes trail as suggestions (R3 — suggestions, not a closed list), plus a small free-text add. Off-path, optional, below the fold — capture rhythm untouched.
  • UI copy: "Keywords" (answers Petra — agreed, notebook-register, not taxonomy-register; the export field is keywords too).
  • source column preserved per R7 (it's one INSERT param — cheap).
  • Display/export: detail + cards render keywords (legacy theme rows included via the backfill, so R4 holds); encounters.json gains keywords: [{text, source}], old themes array kept in the export for one version for downstream compatibility, marked deprecated in the manifest.
  • Enrichment/scaffold compatibility: promptsForThemes and the /enrich payload key off ThemeTag — map keywords to theme slugs case-insensitively where they match, else pass none. (The tutor-side soft mapping of keywords → Layer A concepts lives in D3 prompt assembly block 3, not in tag storage — answers Petra's last open question. Tags stay the user's words; the mapping is the tutor's problem.)
  • No AI normalization/suggestion of any kind (R8) — source:'suggested' exists for the legacy chips, not for model output.
  • Events: tag_add_custom, tag_select_existing, tag_suggestion_used.

Gate: create → reuse on second encounter → both visible on detail → export shows keywords with sources; legacy-theme display regression test; recents query unit test.
