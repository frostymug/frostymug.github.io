Frost Rocks - Finland One-Shot Sync

Date: 2026-07-08 Status: Active synthesis after Arya's 2026-07-02 autonomous build Repo: https://github.com/aspen-frost/frost-rocks TestFlight app: Frost Rocks, bundle com.frostrocks.app, ASC app id 6786727379

What Changed

The old project docs described Frost Rocks as waiting for Andy + Izzy architecture sign-off before stories and build. That is no longer true.

On 2026-07-02, Andy authorized Arya to do a one-shot autonomous build for the Finland prototype. Arya implemented Build 1 in a new private GitHub repo, created the EAS project, built iOS successfully, submitted to App Store Connect, and got the app into the TestFlight path. Andy then tried the app and has feedback.

This does not invalidate the existing strategy, design, or architecture docs. It changes their role. They are now the product canon to reconcile against a real prototype, not the gate before a prototype exists.

What Build 1 Appears To Implement

Build 1 tracks the architecture doc more closely than a random spike would, but it also makes a few pragmatic solo-run calls:

  • Local-first capture and review exist as the usable spine.
  • Camera-roll safety and export are present as the data-loss safety net.
  • The enrichment queue/outbox shape exists, but the proxy is unconfigured and the guide affordances stay hidden until Build 2.
  • Map tiles were cut; the prototype stays closer to timeline/list than a map-led trip view.
  • Sentry was cut because the autonomous run did not have org credentials; the repo uses a simpler error boundary.
  • Scaffold content was authored by Arya as a placeholder; Petra should replace or approve it before treating it as product voice.

What Is Still Canon

The durable Frost Rocks thesis remains: a field encounter seeds later learning, memory, and place-linked understanding. The Finland prototype should keep geology and stone as the spine while testing whether a broader landscape-learning journal is more motivating than commodity rock identification.

The most useful canon docs are now:

  • DESIGN.md for the durable product thesis and learning loop.
  • FINLAND-PROTOTYPE-BRIEF.md for why the Finland trip is the right user-zero test.
  • FINLAND-PROTOTYPE-EXPERIENCE-SKETCH.md for the desired trip/review feel.
  • FINLAND-ARCHITECTURE.md for the intended technical baseline, now updated as the implemented baseline rather than an unapproved pre-build gate.
  • This sync note for the bridge between the docs and the actual TestFlight prototype.

Older question docs and product notes remain useful source material, but they should not be read as the active next-step plan.

Open Questions After The One-Shot

These are the questions that matter now, before another build:

  • What did Andy's TestFlight feedback reveal about the core loop: capture, review, export, and the emotional feel of using it on a trip?
  • Which parts of Build 1 should be accepted as the new baseline, and which were just one-shot shortcuts?
  • Does the app feel like Frost Rocks, or has the Finland use case pulled it toward a broader place-learning journal?
  • Should Build 2 focus on polish and trip-readiness, or should AI enrichment be brought online before Finland?
  • Who owns the replacement/approval of the thin offline scaffold: Petra, Izzy, or Arya with review?
  • What minimum reliability/observability pass does Cassian need before Andy depends on it during travel?

Recommended Next Path

  • Capture Andy's feedback as a structured review note. Split it into bugs, UX friction, product-direction signals, and nice-to-have ideas.
  • Run a reconciliation pass against the docs. Compare Build 1 to the design, experience sketch, and architecture; decide what becomes baseline, what gets changed, and what stays deferred.
  • Cut a Build 2 scope. Prefer a tight TestFlight update: fix feedback, improve trip reliability, add missing instrumentation, and only enable AI if it is low-risk enough for the trip.
  • Assign the work with the current org, not imaginary Podrick. Until Podrick exists, Aspen/Arya can write the build brief; Jesse can implement if Andy wants a normal engineering pass; Cassian can own EAS/release verification.
  • Keep the old docs, but downgrade them to reference material where appropriate. The README should be the living map so the directory stops feeling like a paper avalanche.
