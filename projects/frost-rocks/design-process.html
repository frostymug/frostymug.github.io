Frost Rocks Design Process

Owner: Izzy. Status: Active protocol. Added 2026-06-27.

Purpose

Frost Rocks needs visual and experience design artifacts that Andy can review on his phone. Telegram is the conversation layer, not the design surface. The design surface is a clickable, responsive HTML prototype published through the Aspen Frost docs site.

When Andy asks Izzy for "the prototype," that means Izzy should produce the full prototype package described here unless Andy narrows the request.

Design Artifact Levels

LevelArtifactPurposeOwner
Experience sketchMarkdown in the shared project folderFlow, screens, states, learning loop, emotional targetIzzy
Clickable HTML prototypeStandalone responsive HTML under published-html/Let Andy feel the interaction and visual hierarchy on mobileIzzy
Review recordReview notes beside the prototypeCapture what is being tested, feedback anchors, decisions, and open questionsIzzy

Prototype Location

Project-owned prototypes live under:

/home/ubuntu/.openclaw/workspace/projects/frost-rocks/published-html/

Use one folder per prototype version:

projects/frost-rocks/published-html/
  finland-v0/
    index.html
    styles.css
    script.js
    review.html

Simple prototypes may keep CSS and JavaScript inline in index.html. Larger prototypes may use local styles.css and script.js. Do not use remote assets unless the prototype clearly needs them and the dependency is documented.

Published URL

After publishing, the prototype is available at:

https://docs.aspenfrost.com/projects/frost-rocks/published-html/<prototype-folder>/

For example:

https://docs.aspenfrost.com/projects/frost-rocks/published-html/finland-v0/

The docs site is published through GitHub Pages and served behind Cloudflare Access. It is the durable review surface Andy can open on mobile.

Required Prototype Package

Each prototype version should include:

- the prototype question, - what Andy should react to, - what is intentionally fake, - known rough edges, - decisions made, - open questions.

  • A mobile-first clickable index.html.
  • Enough fake data to make the experience feel real.
  • Clear screen/state coverage for the design question being tested.
  • Visible review anchors such as FR-V0-01, FR-V0-02, and FR-V0-03.
  • A review page or review section explaining:

Review anchors are important. Andy should be able to send feedback in Telegram like:

FR-V0-03: capture still feels too heavy. I want photo -> note -> save, with everything else secondary.

Fidelity Rules

Use the lowest fidelity that can answer the current question.

  • Use an experience sketch when the flow or learning loop is still unsettled.
  • Use clickable HTML when layout, hierarchy, mobile feel, or screen-to-screen transitions matter.
  • Use native Expo/TestFlight only when the question depends on real camera, location permissions, offline storage, gestures, push notifications, or device APIs.

Do not jump to native app work just to review visual direction. HTML is the design prototype lane.

Izzy's Responsibility

Izzy owns the design prototype. She should not require Jesse or another implementation agent to create early clickable prototypes.

When asked for a prototype, Izzy should:

  • Identify the design question.
  • Create or update the experience sketch if needed.
  • Build the responsive HTML prototype in projects/frost-rocks/published-html/<version>/.
  • Add review anchors and a review page/section.
  • Publish through the docs pipeline or ask Aspen to publish if she cannot publish directly.
  • Return the published URL and the specific feedback prompts for Andy.

Publishing

Manual publish command:

cd /home/ubuntu/.openclaw/workspace/project-viewer
./scripts/publish.sh

Before publishing, the docs pipeline runs:

node scripts/build.js
node scripts/check-links.js

Do not run the local docs server. The docs viewer is GitHub Pages only.

First Frost Rocks Use Case

The first expected prototype is the Finland user-zero prototype.

Primary design question:

Does this feel like a travel field notebook with memory, or does it feel like homework?

Required surfaces:

  • Trip Notebook
  • Quick Capture
  • Encounter Detail
  • Review At Home

The prototype should make Andy react to the product's feel, not just its feature list.
