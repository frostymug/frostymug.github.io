Frost-Rocks — Design Document

Project: Rock identification & learning app Company: Aspen Frost Consulting Design lead: Izzy Last updated: 2026-05-23

---

LOCKED DOWN

Audience

  • Rock hobbyists — birder energy
  • Intrinsic motivation, life-list mentality, real-world collection
  • Not students, no credentials, no formal curriculum

Core Loop

  • In the field: photograph rock → attempt ID → log to location
  • At home: flashcard practice seeded by field encounters
  • These feed each other — field experience shapes the practice deck
  • Geographic/experiential context is the primary motivational scaffold — flashcards should feel like memories, not drills

Instructional Design Principles

  • Fading scaffolding: novices get perceptual + vocabulary + elimination help; seasoned users get minimal to none
  • Perceptual scaffolding fades last — noticing the right features IS the skill
  • Attempt first, always — generation effect; scaffolding is a reward for the attempt, not a prerequisite
  • Error correction sequence: validate reasoning → isolate one distinguishing feature → make them look again → resurface soon
  • Confident-wrong = highest priority re-teach event
  • Confidence rating before answer revealed (Sure / Kind of / Guessing) — feeds algorithm AND builds metacognitive calibration

Spaced Repetition: FSRS (py-fsrs)

- Guessing + Wrong → Again - Sure + Wrong → Again (flag: high-priority re-teach) - Guessing + Right → Hard - Kind of + Right → Good - Sure + Right → Easy

  • Repo: https://github.com/open-spaced-repetition/py-fsrs
  • Algorithm: FSRS — models stability (how long you'll remember) + difficulty (how hard this item is for you) separately
  • User never sees the FSRS machinery — confidence + outcome maps to FSRS ratings behind the scenes:
  • Mastery proxy: FSRS stability value — days until forgetting. Continuous signal, not a graduation ceremony.

Domain Knowledge Model (framework, details TBD)

The knowledge model has four layers:

  • Taxonomy — igneous/sedimentary/metamorphic → subtypes → individual rocks; structured for learnability, not just geological correctness
  • Identifying features per rock — which features are diagnostic for a given rock, not just a full attribute list
  • Confusion graph — which rocks get confused with which, and which features resolve each specific confusion; this IS the curriculum sequence
  • Feature salience by photo — which features are observable from an image vs. requiring physical examination; the app must be honest about this ceiling

OPEN QUESTIONS (rough priority order)

- How do we model the taxonomy in detail? - What attributes does each rock entry need? - Who/what maintains the knowledge base — curated, crowdsourced, or AI-generated?

  • Content / domain knowledge model

- Which specific rocks get confused with which? - Which features resolve each specific confusion? - What's learnable from a photo vs. requiring physical examination?

  • The confusion graph

- Where do flashcard/reference photos come from? - Multiple photos per rock? (different lighting, orientations, weathering states) - Do user-submitted field photos feed back into the shared corpus?

  • Photo sourcing

- What does "knowing a rock" mean in this app? (FSRS stability threshold? Explicit graduation? Visible level system?) - Do we surface mastery to the user, and if so how?

  • Mastery model

- For field use: does the app use AI vision to suggest an ID, or does the user guess cold? - If AI assist exists, how do we prevent it from short-circuiting the retrieval practice?

  • AI identification assist

- How does the app know which rocks to surface? (user location, stated interests, travel history?) - Cold start: what do we show a brand new user with no field history?

  • Personalization / relevance engine

- Is there any structured sense of progression or is it purely self-directed? - How do we show the user they're getting better over time?

  • Assessment (if any)
