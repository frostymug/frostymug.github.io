Decision Frame

Andy is already working with Izzy and Arya on AI tutor interactions, so this document treats the AI tutor as the active next major product track. The purpose here is to organize everything else in the live backlog, assign rough size, and make the next product/engineering sequence easier to reason about.

Recommended Sequence

0. Continue Active AI Tutor Design

Keep AI tutor interaction design moving as the next major track, especially the D3 evening debrief wedge and wizard-of-oz pilots. This is the strategic product bet.

1. Make Exports Reviewable

Build an export package that includes structured encounter data plus the corresponding photo files, with deterministic filenames and a manifest. This directly helps Andy, Izzy, Petra, Arya, and future agents review real sessions.

2. Preserve Metadata

Add native-quality photo metadata preservation through capture/save/export paths and explicit structured encounter metadata for fields like time zone. This should be tested against a native iPhone photo taken at the same place/time.

3. Add Small Capture Recovery

Add note editing for existing encounters. This is a good small follow-up once the larger capture reliability problem is closed.

4. Shape the Knowledge Inputs

Fold first-class questions into the AI tutor design, then build the minimal question object needed for review/exploration. After that, revisit dynamic tags/custom keywords as the broader knowledge-model cleanup.

Size Notes

Blank screen after Wispr Flow keyboard handoff — M

Implemented on origin/main in PR #12. Keep the size for historical estimate/reference.

Export encounter data with linked photo files — M

Requires package shape, manifest design, file copying/export mechanics, deterministic names, and regression evidence that the exported package can be reviewed outside the app. Not conceptually hard, but it touches data, filesystem, and handoff workflows.

Preserve native-quality photo metadata at capture/export — M

Could become L if the current camera library cannot expose or preserve the needed metadata cleanly. Start as M with a spike-like verification step: inspect raw Frost Rocks image metadata, identify where metadata is lost, then preserve/write it through the final file.

Zoom into encounter images during review — M

Implemented on origin/main in PR #13 plus follow-up gesture fixes. Keep the size for historical estimate/reference.

Zoom and pan the map view — M

Implemented on origin/main in PR #14. MapLibre remains out of scope.

Edit note on an existing encounter — S

Narrow UI and data update. Reuse existing encounter update path. Keep it to personal notes only.

Capture user questions for later AI exploration — M

The first UI could be small, but making questions first-class touches capture, encounter detail, review, export, and tutor dialogue. It should wait until the tutor object model has settled enough to avoid building the wrong shape.

Dynamic themes and user-defined keywords — M

Manual custom tags plus recent/frequent reuse is moderate work across capture, persistence, display, and export. Do not jump to AI taxonomy yet.

Priority Rationale

The order is intentionally not "shiniest first." AI tutor is the active strategic track, yes. The first field-fix layer is now done: capture reliability, photo inspection, and map inspection have landed. The remaining risk is the evidence substrate around export and metadata. Otherwise we will be asking a very clever tutor to reason over a partial field record. That is how expensive hobbies happen. We are trying to avoid those. Stoically.

Open Product Questions

  • Should export-with-photos be built immediately as a zip/share artifact, or is a local diagnostic bundle enough for the next TestFlight review?
  • For metadata, what is the minimum acceptable parity with native iPhone photos: GPS + timestamp only, or GPS + timestamp + camera/device/lens fields?
  • Should first-class questions ship inside the D3 tutor wedge, or as a capture/review model change immediately after D3?
  • Should dynamic tags remain user-authored only for the first pass, or should the tutor suggest normalized concepts after the user writes in their own words?
