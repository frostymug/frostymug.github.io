Frost Fitness — Architectural Review

Author: Arya (Architect, Aspen Frost Empire) Date: 2026-05-24 Repo: frostymug/frost-fitness Status: First review — living document

---

What We're Working With

Expo + React Native (SDK 54, React 19). SQLite-first local storage. Two AI backends: Gemini 2.5 Flash for meal photo vision, Claude Sonnet 4.6 for conversational coaching. iOS-only currently (EAS build config, HealthKit dependency). Single user. In production via TestFlight, version 2.0.0.

---

✅ What's Strong

1. The schema design is genuinely good

The low/mid/high range model for all macro values is the right call. Calorie estimation from a photo is inherently uncertain — preserving that uncertainty all the way to the UI instead of collapsing it to a fake-precise number is architecturally honest. It also pays dividends in the coach layer: the AI can reason about ranges ("you're probably between 400–600 kcal today") rather than pretending a photo analysis is precise to the gram. This is the kind of decision that's much harder to add retroactively. Good call.

2. Clean service interface pattern

VisionService, ConversationService, and HealthKitService are defined as TypeScript interfaces with concrete implementations behind them. The singletons (visionService, conversationService, healthKitService) export the interface type, not the implementation. Swapping model providers or adding a mock for testing is a one-file change. This is what makes the codebase sustainable as the product evolves.

3. The tool-use loop is solid

The Claude conversation service handles multi-turn tool use correctly:

  • Appends assistant + tool_result turns to the in-memory message list without persisting intermediate steps to the DB
  • Caps at 5 iterations and fails loudly if it hits the cap (rather than silently returning empty text)
  • Correctly identifies the terminal condition (stop_reason !== 'tool_use')

A lot of implementations get this wrong. This one doesn't.

4. Local-first architecture with rich context injection

buildContextSummary() assembles a structured Markdown block — goals, today's progress, 7-day daily history, 30-day weight trend, recent foods — and injects it into every chat system prompt. The coach only calls database tools for data outside that pre-loaded window. This keeps the common case fast and the uncommon case correct. Good design.

5. Transaction discipline

saveMeal() and updateMealItem() are wrapped in withTransactionAsync. Denormalized meal totals are recomputed atomically within the same transaction. The FK ON DELETE CASCADE on meal_items.meal_id is set correctly. This is foundational and a lot of apps get it wrong.

6. AI cost instrumentation exists

ai_call_log records token counts, latency, and success/failure for every API call. Most personal apps don't instrument this at all. Having it means cost spikes are visible before they show up on the credit card bill.

---

⚠️ What's Weak

1. API keys are in the client bundle — the biggest risk

EXPO_PUBLIC_CLAUDE_API_KEY and EXPO_PUBLIC_GEMINI_API_KEY are embedded in the compiled JS bundle. The CLAUDE.md acknowledges this as "acceptable for a single-user personal app distributed via TestFlight." That's true today — but TestFlight IPAs can be extracted and reverse-engineered. The Anthropic key in that bundle has write access to the account.

If this stays personal: manageable, monitor the AI cost logs for unexpected usage. If anyone else ever installs it: becomes a liability immediately.

Fix: a thin server-side proxy — a few lines of serverless function (Vercel, Cloudflare Workers, AWS Lambda) that holds the keys and forwards requests. Client sends data, proxy adds auth, forwards to AI provider. One afternoon of work, eliminates the risk permanently.

Priority: High if productizing. Monitor if staying personal.

---

2. No error boundaries on screens

Any unhandled exception in a React component tree kills the entire screen with no recovery path — user sees a blank white screen and has to force-quit. Sentry is already in the deps (@sentry/react-native) but there's no root-level <ErrorBoundary> wrapping the screen stack.

Fix: Wrap the root <Stack> in a Sentry error boundary. 20 lines of code.

Priority: Medium — important before real users.

---

3. The migration system is fragile

runMigrations() checks the current schema version once and applies pending migrations sequentially. The fragility: if a migration fails partway through, the db_version table won't be updated, but some DDL may have already executed. SQLite DDL inside transactions has quirks — not all DDL is transactional in the same way DML is. For a solo app this is survivable (wipe and reinstall). For a multi-user product with real user data, a failed migration is a support nightmare.

Fix: Execute each migration in its own transaction with rollback on failure and explicit version bookkeeping.

Priority: Low now, High before multi-user.

---

4. pendingImage is a global mutable singleton

pendingImage is a module-level variable used to pass base64 image data between camera.tsx and confirm.tsx, because Expo Router can't serialize large base64 strings in route params. The workaround is pragmatic but has a failure mode: if the user navigates away from confirm.tsx mid-analysis (force-quit, background app), the stale image persists in memory and may be incorrectly associated with the next camera capture.

Fix: Always call pendingImage.clear() on confirm.tsx mount regardless of whether a valid image is found. Could also move to a more principled async store (e.g., a temp file path rather than in-memory base64).

Priority: Low — minor edge case today.

---

5. UID generation must be runtime-compatible, not just random

Math.random().toString(36).slice(2) + Date.now().toString(36)

Was used everywhere for primary keys. Math.random() is seeded, deterministic in some environments, and has non-negligible collision probability under rapid generation. This is a dormant landmine — harmless for a single-user local-only app, meaningful if records ever sync to a backend or get merged across devices.

Fix: use the strongest UUID source the deployed runtime actually provides. crypto.randomUUID() may exist in Node/Jest while being absent in the Expo/Hermes runtime. The current implementation should prefer crypto.randomUUID() when available, fall back to UUID v4 bytes from crypto.getRandomValues, and only then fall back to Math.random() as a last resort.

Priority: Low now, High before sync/multi-user.

Lesson captured 2026-06-06: the production photo-submit failure was not a pure UID bug; it was a test-boundary bug. Jest proved the utility worked in Node. It did not prove the utility worked in the app runtime.

---

6. Default settings hardcode personal values

DEFAULT_GOALS in settings.ts has displayName: 'Andy' and targetWeightKg: 83.9 baked in. Minor for a personal app. Meaningful the moment this becomes a product.

Fix: Replace hardcoded defaults with a first-run onboarding flow that collects these values.

Priority: Productization prerequisite.

---

7. No test coverage

Zero test files visible in the repo. The core of this app is: (a) write to SQLite correctly, and (b) call AI APIs correctly — both have fairly deterministic inputs and outputs. A small suite of unit tests on saveMeal, recomputeMealTotals, getDayTotals, and the AI service adapters would catch regressions cheaply and give Eddie (when he arrives) a safety net.

Priority: Medium — important before agent-assisted development at scale.

---

Testing Doctrine — Runtime Compatibility Is a First-Class Boundary

Frost Fitness is an Expo app, so Node/Jest is not the runtime. Tests that pass in Jest can still hide missing or partial APIs in Expo/Hermes, especially around platform and browser-shaped capabilities.

Any code touching runtime-provided APIs needs compatibility coverage that models the deployed shape of the API, not just the convenient test shape. This includes:

  • crypto, especially partial availability such as getRandomValues present while randomUUID is absent
  • FileSystem, cache paths, and temp-file cleanup
  • camera/media/image manipulation APIs
  • storage and SQLite adapters
  • route params and navigation lifecycle
  • browser-shaped globals such as Blob, File, URL, atob, and fetch

Testing rules for this project:

  • Prefer real local utilities in screen tests unless mocking is necessary for determinism.
  • Mock external services, network calls, permissions, and device-only integrations at the boundary.
  • Do not mock below the failure boundary when the behavior is cheap and stable enough to exercise directly.
  • For platform utilities, test multiple runtime shapes: full API exists, partial API exists, API missing, and fallback behavior.
  • Every production screenshot/error should become a regression test that recreates the exact missing capability or failing state.

The goal is not simply more tests. The goal is less flattering tests: a fake universe that resembles the deployed app closely enough to catch compatibility mistakes before Andy does QA with a real phone.

---

❓ Open Questions

These can't be answered from the repo. They determine which debts above get paid first.

The current architecture is explicitly calibrated for personal use. If this becomes a product — even a small beta cohort — several of the "acceptable for personal use" decisions become simultaneous blockers. Need to know the horizon to prioritize.

  • What's the growth ambition?

All data lives in SQLite on-device. iCloud device backup covers it implicitly, but that's implicit. If the phone is lost or wiped, nutrition history is gone. Is that acceptable? If not: iCloud Drive, CloudKit, or a backend sync are the options, each with meaningfully different complexity.

  • Is there a sync or backup strategy?

If Eddie is an AI agent (low-cost model), the absence of tests is a real problem — agent-written code without a test harness is where silent regressions accumulate. If Eddie is a human, the code is readable enough to onboard quickly. The answer shapes the urgency of item 7 above.

  • Who is Eddie, and how does he work?

The schema has a comment: -- Future: add fasting_windows table for v2 fasting timer. The checkInTime setting (morning/evening/off) suggests push notification infrastructure. app.json is at 2.0.0. Understanding what's coming next would change prioritization — some debts above matter a lot more in some directions than others.

  • What's the v2 feature roadmap?

If it stays personal, most of the above is informational. If it hits the App Store: the key security issue becomes urgent, the hardcoded personal values become embarrassing, and the missing error boundaries become 1-star reviews waiting to happen.

  • Any plans to publish beyond personal use?

---

Summary Verdict

This is a well-built personal app that took its data model seriously. The macro range system, service abstractions, and coach context injection are all above-average decisions for a project of this scope. The weaknesses are almost entirely in the "fine for personal use, not ready to scale" category — which is the right set of weaknesses to have at this stage.

Most actionable item before this touches anyone other than Andy: move the API keys to a server-side proxy. Everything else is debt that can be carried.
