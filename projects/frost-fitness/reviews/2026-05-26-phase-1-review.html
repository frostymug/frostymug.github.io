---

Overall

Solid work. The test harness is genuinely good — the in-memory SQLite mock is clever and pragmatic. Error boundary is the right shape. Sentry config is reasonable. One real problem and several smaller things to fix before this merges. Let me go in order of priority.

---

🔴 Blocking — must fix before merge

B1. PRs were never opened (process violation)

Per APP-STORE-PLAN.md rule of engagement #2: "One PR per task." You pushed four branches but opened zero PRs. I can review branches directly (and did, today), but the plan is for tasks to merge through PRs so we get:

  • CI on the PR (not just on master)
  • A reviewable diff with context
  • An audit trail in GitHub

Fix: Open four PRs — one per branch — targeting master. Paste the acceptance checklist results from APP-STORE-PLAN.md into each PR description. Do this before anything else.

---

B2. Branches 1-2 and 1-3 will collide on merge

Both phase-1/task-1-2-sentry and phase-1/task-1-3-error-boundary contain their own copy of Sentry.init(...) in app/_layout.tsx. Whichever lands second will either (a) hit a merge conflict, or (b) silently duplicate Sentry.init calls — which Sentry tolerates but is sloppy and means the second init's config wins, hiding bugs.

This is a real scoping problem with the way the work was split: Task 1.3 has a hard dependency on Task 1.2 (the error boundary uses Sentry.ErrorBoundary, which requires Sentry to be initialized). They should not be two independent branches off master.

Fix: Rebase phase-1/task-1-3-error-boundary onto phase-1/task-1-2-sentry, not master. Open Task 1.2's PR first; once it merges, rebase 1.3 onto the new master. The 1.3 diff should then contain only the error-boundary changes, not a second Sentry.init.

---

B3. CI workflow only runs on PRs targeting master

.github/workflows/test.yml:

on:
  pull_request:
    branches:
      - master

This means the workflow will not run on PRs between feature branches (e.g., a PR from phase-1/task-1-3 targeting phase-1/task-1-2). For Phase 1 you don't notice; the moment Phase 2 stacks PRs we lose CI coverage on the stacked ones.

Fix: Either remove the branches filter (run on all PRs) or extend it to include feature branches. I prefer running on all PRs — pull_request: with no filter — it's simpler and safer.

Also: add push: { branches: [master] } so we get a green/red signal on the default branch over time.

---

B4. ErrorBoundary.test.tsx doesn't actually test the production fallback

This is the most important code-quality issue and the one I want you to internalize, because it'll come up again.

The test file declares a BombChildFallback component inline in the test file, with a comment saying "The real _layout.tsx ErrorFallback is identical in structure." It then asserts the inline copy renders correctly when the boundary catches an error.

This is a tautology. The test verifies that the test file's own inline fallback works. It does not verify the actual ErrorFallback in app/_layout.tsx. If someone breaks the real ErrorFallback tomorrow (changes the copy, removes the button, throws inside it), this test will still pass. That is the opposite of what a regression test is for.

The reason you ended up here was real — app/_layout.tsx is hard to import in a unit test because it has top-level side effects (SplashScreen.preventAutoHideAsync(), Sentry.init()). The right fix is not to duplicate the component; it's to extract the testable piece.

Fix:

  • Extract ErrorFallback from app/_layout.tsx into its own file: src/components/ErrorFallback.tsx. Default export, no side effects, just the component.
  • Update app/_layout.tsx to import { ErrorFallback } from '@/components/ErrorFallback'.
  • Update the test to import the real ErrorFallback and render it inside the mocked boundary. Delete the inline BombChildFallback duplicate.

This is also good architecture independent of testability — _layout.tsx should be assembly, not implementation. The fallback UI deserves to live in components/.

---

🟡 Should fix in this phase (small, do them with the rebase)

S1. jest.config.js has a trailing-indent quirk

  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1',
    '^expo-sqlite$': '<rootDir>/src/__mocks__/expo-sqlite.ts',
  },
  };

The closing }; is indented two spaces. Cosmetic but it tells me the file wasn't formatted. Run Prettier (or expo's equivalent) on it.

S2. The unused mock file

src/__mocks__/expo-sqlite.ts exists and jest.config.js maps expo-sqlite to it — but the actual meals.test.ts uses a different in-memory implementation declared inline (createInMemoryDb). The shared mock is never exercised by the current test.

So we have two slightly different in-memory SQLite implementations: one in __mocks__/ (registered globally) and one inline in meals.test.ts. They will drift.

Pick one. My recommendation: keep the inline one for now (it's scoped to the test that needs it), delete src/__mocks__/expo-sqlite.ts, and remove the moduleNameMapper entry that points to it. We can promote a shared mock later when a second test file actually needs it. YAGNI.

S3. Sentry release string format

release: `${Constants.expoConfig?.version}+${Constants.expoConfig?.ios?.buildNumber ?? '0'}`,

Sentry's iOS SDK conventions use version@build or bundleId@version+build, not version+build. The current format will work but won't match what the native SDK auto-derives, which means your JS errors and native crashes will be tagged with different release strings and Sentry won't group them. Subtle but it bites.

Fix: Either drop the explicit release: and let Sentry auto-derive (preferred — that's what uploadSourceMaps: true in the plugin assumes), or set it to the canonical format org.frostymug.frostfitness@${version}+${build}.

S4. Sentry won't initialize without EXPO_PUBLIC_SENTRY_DSN

Sentry.init({ dsn: process.env.EXPO_PUBLIC_SENTRY_DSN, ... });

If the env var is missing, Sentry.init is called with dsn: undefined. The SDK no-ops in that case, which is the right behavior — but there is no log warning anyone. Andy could ship a release build with no DSN and find out two weeks later when crashes aren't appearing.

Fix: Add a if (!process.env.EXPO_PUBLIC_SENTRY_DSN && !__DEV__) { console.warn('[Sentry] DSN missing in non-dev build'); } next to the init. Belt and suspenders — also document the env var in CLAUDE.md and README.md.

S5. tracesSampleRate: 0.2 should be conditional

20% trace sampling in development is wasteful and may slow down dev builds. Set it to 1.0 in dev (full sampling for debugging) and 0.2 in production:

tracesSampleRate: __DEV__ ? 1.0 : 0.2,

---

🟢 Nits (polish, optional, don't block merge)

N1. Test file uses Math.random() in makeItem()

id: `item-${Math.random().toString(36).slice(2)}`,

A test helper using the same insecure UID pattern we're about to remove in Phase 4 is mildly ironic. Not a real bug — test IDs don't need to be cryptographic — but a let counter = 0; id: \item-${++counter}\`` would be more deterministic and would make test failures easier to debug.

N2. The mock SQL parser is impressive but fragile

The regex-based SQL interpreter in meals.test.ts works because meals.ts uses a narrow, predictable set of SQL patterns. The moment someone writes a query with JOIN, GROUP BY, parameterized IN (?,?,?), or ORDER BY, the mock will silently return wrong results.

This is fine for now — it's the right tool for today's tests. But put a comment at the top of createInMemoryDb() listing the SQL features it supports, and a // TODO: when we add X, switch to better-sqlite3 so the next person knows the limits.

N3. JESSE-NOTES.md baseline is still placeholder

Crash-free rate (last 7 days): ___%
Session count: ___

Task 1.4 acceptance was "baseline metric documented with date." It's not documented yet — it's a template waiting to be filled in. Either fill it in (you can read the current Sentry dashboard) or push back on me that you don't have Sentry access yet. Don't merge with the blank.

What I'm impressed by

A short list, because positive feedback is also calibration:

  • getDb mocking strategy. Mocking @/db/database so getDb() returns an in-memory shim is exactly the right level of abstraction. Not too deep (mocking SQLite native bindings), not too shallow (mocking individual function calls). This is what good unit test architecture looks like.
  • 14 tests, not 8. Plan said "at least 8." You wrote 14 with thoughtful boundary cases (cross-date isolation, empty state, recompute after update). That's the right instinct.
  • The __DEV__ gating on Force Crash / Test Sentry. Production users will never see those buttons. Good defensive instinct.
  • The error fallback doesn't show stack traces. You explicitly tested for it. Excellent.
  • JESSE-NOTES.md scope log. You caught the Math.random() UID issue and logged it as "scheduled Phase 4" instead of fixing it. That is exactly the discipline I asked for in rule #3 of the plan. Resist scope creep, write it down, move on. Well done.
