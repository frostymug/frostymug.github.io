Technology Choices by Layer

This is the practical stack menu for Season Two. The recommended column is what I would reach for first; the alternatives are real options, but only worth taking when the product pressure justifies the extra moving parts.

Layer Recommended Choice Good Alternatives Pros Cons / Watchouts Frost Fitness Read
Mobile client React Native / Expo Native iOS/Android, Flutter Already in place; one app codebase; strong ecosystem; fast iteration. Native edge cases still happen; performance-sensitive screens need care. Keep it. This is the right client layer for the current team and product.
Edge gateway Cloudflare Worker AWS Lambda, Vercel Functions, API Gateway + Lambda Low latency; cheap; already deployed; great fit for auth checks, rate limits, validation, and AI proxying. Not ideal for heavyweight business logic or long-running jobs. Promote it from proxy to gateway. Do not replace it when the backend arrives.
Backend language TypeScript Go, Python, Kotlin, Java Same language family as app, Worker, tests, and tooling; easier hiring and agent handoffs. Runtime failures are still possible; needs disciplined typing and tests. Use TypeScript until a specific workload proves another language is better.
Backend runtime Node.js Bun, Deno, Cloudflare Containers Mature; boring in the good way; broad package support; compatible with most TypeScript backend tools. Can get messy without clear module boundaries; async code needs consistency. Start here. Bun or Deno are interesting, but Node is the reliable Season Two choice.
API framework Hono or Fastify NestJS, Express Hono is lightweight and Worker-friendly; Fastify is mature, fast, and plugin-rich. NestJS adds structure but also ceremony; Express is familiar but less modern. Hono if we stay close to Cloudflare; Fastify if we want a conventional Node backend.
Database Postgres Cloudflare D1, Supabase Postgres, Neon, PlanetScale Durable source of truth; proven; great for users, subscriptions, history, plans, and admin data. Needs migrations, backups, connection handling, and ownership discipline. Use Postgres when durable server-side product state becomes real.
ORM / query layer Drizzle or Prisma Kysely, raw SQL Drizzle is lean and type-safe; Prisma is polished and widely understood. Prisma can feel heavy; Drizzle asks for more SQL comfort. Drizzle if we want architectural sharpness; Prisma if developer ergonomics matter more.
Auth Clerk or Auth0 Supabase Auth, custom auth later Outsources login, sessions, password resets, MFA, and social identity. Vendor coupling; pricing and user data boundaries need review. Buy auth early. Custom auth creates avoidable production pain.
Billing Stripe RevenueCat, Paddle Best-in-class subscriptions, invoices, webhooks, customer portal, and entitlement integration. Webhooks and entitlement state must be handled carefully. Stripe for web/backend billing; consider RevenueCat if mobile app-store subscriptions dominate.
Jobs / queues Cloudflare Queues BullMQ + Redis, Temporal Good for lightweight async work; keeps us Cloudflare-native a little longer. Complex workflows can outgrow it; Temporal is powerful but a commitment. Start simple. Add Temporal only when workflows become the product, not before.
Object storage Cloudflare R2 AWS S3, Supabase Storage Cheap, S3-compatible, and naturally close to the Worker layer. Access control and lifecycle policies need explicit design. Use R2 for generated assets, exports, and large user files if those appear.
Analytics / events PostHog Segment, Amplitude, internal event table Product analytics, funnels, feature flags, and event capture in one place. Privacy boundaries matter, especially around health-adjacent data. Use a privacy-conscious setup. Track product behavior, not sensitive nutrition details.
Hosting Fly.io, Render, or Railway AWS ECS/Fargate, Cloudflare Containers Fast to ship a real backend without building a full platform team. Each has scaling, region, and operational tradeoffs. Pick boring managed hosting first. AWS can wait until there is AWS-shaped pain.

Working Recommendation

Stay Cloudflare-native as long as possible. When user accounts, billing, durable workout history, support workflows, and usage accounting need one authoritative home, add a TypeScript backend with Postgres behind the Worker gateway.

Do not reach for Java, Go, Python, or Kotlin just to feel grown-up. Those are specialization moves. For Frost Fitness, TypeScript keeps the team in one language family until the product has a specific reason to split the stack.
