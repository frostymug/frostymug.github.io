Hotfix Spec — Camera Capture Silently Fails (Phase 4 / Task 4.3 regression)

Author: Arya (Architect) Date: 2026-06-06 Severity: P0 — core flow broken. Photo capture does not save. Owner: Jesse (implementation) Base: master @ 6357b56 (post PR #13) Approval status: APPROVED to implement — hotfix forward (not a revert).

---

Symptom

User takes a photo (or picks from library) → meal is not saved, no error is shown. "The photo never comes back off the shelf."

Root Cause

Task 4.3 changed pendingImage.set() from a synchronous in-memory base64 write to an async file copy (FileSystem.copyAsync into cacheDirectory).

In app/log/camera.tsx, that call is now awaited before navigation, with no try/catch:

await pendingImage.set(asset.uri);   // can reject
router.push({ pathname: '/log/confirm', params: { imageUri: asset.uri, ... } });

If pendingImage.set() rejects for any reason (copy fails on a picker/camera URI, transient native FS error, deprecated legacy FS quirk on expo-file-system@19), the await throws, router.push never runs, and the user sees nothing. The image is dropped before it ever reaches the analyze/confirm screen.

The old code could not exhibit this — it was synchronous and navigation was unconditional:

if (asset.base64) pendingImage.set(asset.base64);  // sync, cannot reject
router.push(...);                                   // always ran

Why tests didn't catch it

always-resolving stubs, so copyAsync can never fail in the suite.

Result: green tests, broken capture. Mock-hides-reality gap.

  • src/__tests__/pendingImage.test.ts mocks expo-file-system/legacy with
  • There is no test covering camera.tsx navigation when set() rejects.

---

Design Principle For The Fix

The temp-file copy is an optimization, not a load-bearing dependency. Capture and analysis must work even if the cache copy fails. We keep the 4.3 wins (no stale-image leak, temp-file storage instead of in-memory base64) but make them best-effort: on any failure, fall back to the original picker URI. Navigation to confirm must be unconditional once we have a valid asset.

---

Changes

1. src/services/pendingImage.ts — set() must never reject; degrade to source URI

Wrap the copy in try/catch. On failure (or no cacheDirectory), store the original sourceUri and return it. set() should always resolve to a usable URI.

async set(sourceUri: string): Promise<string> {
  const previousUri = _uri;
  const cacheDirectory = FileSystem.cacheDirectory;
  let nextUri = sourceUri;

  if (cacheDirectory) {
    const target = `${cacheDirectory}pending-meal-${uid()}.jpg`;
    try {
      await FileSystem.copyAsync({ from: sourceUri, to: target });
      nextUri = target;
    } catch (e) {
      // Best-effort cache copy. Fall back to the original picker URI so capture
      // still works. confirm.tsx reads whatever path it is handed.
      console.warn('[pendingImage] cache copy failed, using source URI', e);
      nextUri = sourceUri;
    }
  }

  _uri = nextUri;
  if (previousUri && previousUri !== _uri) await removeIfCached(previousUri);
  return nextUri;
}

2. app/log/camera.tsx — navigation must not depend on set() succeeding

Apply to both handleTakePhoto and handlePickFromLibrary. Since set() will no longer reject after change #1, this is belt-and-suspenders, but it is required so a future regression in set() can never again kill navigation silently.

if (!result.canceled && result.assets[0]) {
  const asset = result.assets[0];
  try {
    await pendingImage.set(asset.uri);
  } catch (e) {
    console.error('[camera] pendingImage.set failed', e);
    Sentry.captureException(e);
    // proceed anyway — confirm falls back to the imageUri param below
  }
  router.push({ pathname: '/log/confirm', params: { imageUri: asset.uri, source: 'camera' } });
}

Add import * as Sentry from '@sentry/react-native'; to camera.tsx (already a project dependency; used in settings.tsx).

3. app/log/confirm.tsx — already correct, verify only

confirm.tsx reads pendingImage.get() and FileSystem.readAsStringAsync(...). With the fallback in place it may now receive the original picker URI instead of a cache path. Verify readAsStringAsync works against that URI; it should, since the picker URI is a readable file:// path. If read-from-source proves unreliable on-device, add a secondary fallback: if reading pendingImageUri throws, retry reading the imageUri route param. Do NOT add this speculatively — only if the device test in the verification section shows it is needed.
