Frost Fitness Monitoring Plan

Prepared: 2026-06-05 Owner: Cassian Scope: production app, Cloudflare Worker, Sentry, Expo/EAS, App Store Connect, CI/CD, and vendor status.

This plan gives Andy two practical paths to a comprehensive health picture for Frost Fitness. Plan A uses Cassian as the aggregator, pulling read-only status from each platform and reporting the operational truth. Plan B evaluates a consolidated monitoring product for a single shared dashboard.

Plan A: Cassian-Assembled Monitoring

This is the fastest and most controlled path. Cassian builds a small monitoring routine that pulls read-only status from each platform, normalizes it, and posts a single report on a schedule or on demand.

Coverage

AreaSourceWhat Cassian Pulls

TestFlight and App StoreApp Store Connect APIBuild processing state, beta review state, TestFlight feedback availability, crash submissions, and app review status.

Access Needed

  • Cloudflare: read-only API token with account analytics, Workers scripts/routes read, Workers observability/log read, and zone read for the Frost Fitness Worker account.
  • Sentry: org/project token with read access to projects, releases, issues, events, alerts, and monitors.
  • App Store Connect: API issuer ID, key ID, and .p8 private key. Minimum role should be Developer; App Manager is better for TestFlight, app status, and review visibility.
  • Expo/EAS: Expo token with project, build, submission, update, and environment read access.
  • GitHub: token or installed app access if repository workflow and deployment visibility requires more than the existing local GitHub auth.
  • Delivery target: Telegram by default; optional Notion, Slack, email, or a static HTML status page if Andy wants a second surface.

Report Shape

  • Green/yellow/red summary per system.
  • Latest production app build, TestFlight build, Worker deploy, and GitHub release commit.
  • Crash/error delta since the last check.
  • Worker request count, error count, latency, and route health.
  • Active vendor incidents that could affect production or release operations.
  • Concrete blockers and links to exact dashboards or logs.

Operational Read

Plan A is low cost, auditable, and easy to tune to Andy's actual decision loop. Its weakness is that it is a reporting layer, not a full live observability product. If we build it cleanly, the same collectors can feed a dashboard later.

Plan B: Consolidated Monitoring Tool

This path gives the team a shared command center. The practical candidates are Sentry-centered monitoring, Datadog, and Grafana Cloud.

Option 1: Sentry-Centered Monitoring

Sentry is the preferred near-term dashboard because Frost Fitness already uses it, Expo can integrate with Sentry for React Native crash reporting, and Worker telemetry can be routed into Sentry through structured events or OpenTelemetry-style export paths.

Covers WellDoes Not Cover Cleanly
React Native / Expo crash reports, releases, source maps, error trends, uptime monitors, alerts, and Worker error visibility.App Store Connect build/review lifecycle, rich Cloudflare account analytics, and Expo build status without custom automation.

Verdict: best least-moving-parts option. Use Sentry as the first operational center, then add a small Apple/Expo status bridge.

Option 2: Datadog

Datadog is the grown-up consolidated observability platform. It is strongest if Aspen Frost wants one monitoring plane across many apps, Workers, APIs, logs, synthetic checks, and alert routes.

Covers WellWeaknesses
Cloudflare metrics and logs, synthetic health checks, dashboards, alerting, service maps, broad integrations, and multi-product scaling.Higher cost, heavier setup, Sentry forwarding needs integration work, and App Store Connect still needs custom ingestion.

Verdict: best if the empire grows into multiple production apps and services. Too heavy for the immediate Frost Fitness gate unless we expect fast scale.

Option 3: Grafana Cloud

Grafana Cloud is the middle ground for engineering dashboards. It can work well with Cloudflare metrics, Prometheus-style collectors, logs, alerts, and custom panels.

Covers WellWeaknesses
Custom dashboards, Cloudflare metrics through exporters or integrations, alerting, and flexible ingestion.More DIY glue, less polished mobile-release workflow support, and custom Apple/Expo panels still required.

Verdict: good for engineering dashboards; less good as the primary mobile app health command center.

Recommended Path

  1. Start Plan A immediately with read-only access for Cloudflare, Sentry, App Store Connect, Expo/EAS, and GitHub.
  2. Make Sentry the first consolidated dashboard for crashes, releases, Worker errors, uptime checks, and alerting.
  3. Add a Cassian Apple/Expo bridge for build status, TestFlight state, app review state, and EAS build/submission status.
  4. Trial Datadog only when there are at least two or three production apps/services that need one observability plane.
  5. Use Grafana Cloud as the lower-cost engineering-dashboard alternative if Datadog is too heavy.

Safety Standard

A monitoring plan is not complete until it catches the failures we already know matter: app crashes, missing source maps, Worker upstream failures, bad deploys, build processing failures, vendor incidents, and TestFlight/App Store review blockers. Anything less is decoration. This plan starts with those failure modes and expands from there.
