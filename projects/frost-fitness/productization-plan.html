Frost Fitness — Productization Plan

Author: Arya (Architect) Date: 2026-05-25 For: Jesse (Engineer, Sonnet 4.6) Reviewed by: Arya before each phase merges Status: Active execution plan — Phase 2 complete as of 2026-06-05

---

Read This First (Jesse)

This document is your execution plan. It is scoped to take Frost Fitness from TestFlight to public App Store launch. Nothing more. New features (calisthenics, YouTube integration) are out of scope — that is a separate roadmap after launch readiness lands.

Rules of engagement:

  • Work top-to-bottom by phase. Each phase has a gate. Do not start the next phase until the current one is reviewed and merged.
  • One PR per task. Each task here is sized to be a single, reviewable PR. If a task feels like it needs more than ~400 lines of changed code, stop and ask me.
  • No scope creep. If you find a bug or refactor opportunity outside the scoped task, write it in projects/frost-fitness/JESSE-NOTES.md and keep moving. We triage later.
  • When in doubt, ask. Surface the question to Andy or me. Do not invent answers to ambiguity — especially around product behavior. The cost of asking is 30 seconds. The cost of guessing wrong is a rewrite.
  • Verify before you claim done. Every task has an acceptance checklist. Run through it before you open the PR. Paste the results into the PR description.
  • You own the test for your change. Phase 1 sets up the test harness. From Phase 2 onward, every change ships with at least one test unless I explicitly waive it in review.

Read these before you start, in this order:

  • The repo's README.md and CLAUDE.md — orientation
  • ARCHITECTURE-REVIEW.md (this folder) — the diagnosis this plan is treating
  • This document, end to end, before writing any code

Strategy Overview

The architecture review identified seven weaknesses. Five of them carried notes like "acceptable for personal use, blocker for productization." Andy has now told us we are productizing — App Store launch. That re-prioritizes everything:

#IssueOld priorityApp Store priorityPhase
1API keys in client bundleHigh if productizing🔴 Blocker2
2No error boundariesMedium🔴 Blocker1
3Fragile migrationsLow / High before multi-user🟡 Important4
4pendingImage globalLow🟢 Polish4
5Math.random() UIDsLow / High before sync🟡 Important4
6Hardcoded "Andy" defaultsProductization prereq🔴 Blocker3
7No test coverageMedium🔴 Blocker for agent-built code1

Plus App Store non-negotiables that aren't in the original review:

  • Privacy policy + Terms of Service
  • App Store Connect metadata (screenshots, descriptions, age rating)
  • HealthKit usage strings + Apple's HealthKit review (a separate, stricter review)
  • Account deletion flow (Apple requires it if there is any concept of user data)
  • Crash-free rate ≥ 99% (Sentry actually wired up and reporting)
  • Cold-start performance acceptable on the oldest supported device

The plan: 5 phases, ~3–4 weeks of Jesse-time with my reviews. Phases gate on each other.

Phase 1: Foundations (safety net before we change anything else)
  ↓
Phase 2: Server-side proxy (the security blocker)
  ↓
Phase 3: Multi-user productization (onboarding, defaults, account)
  ↓
Phase 4: Reliability hardening (the remaining technical debts)
  ↓
Phase 5: App Store submission (compliance, metadata, screenshots, review)

Why this order: Phase 1 first because every change after that needs the safety net. Phase 2 second because we cannot ship a binary with our Anthropic key in it — that is a write-credential leak the moment one person extracts an IPA. Phase 3 turns "Andy's app" into "an app." Phase 4 cleans the remaining debt. Phase 5 ships.

Phase 1 — Foundations (Safety Net)

Goal: Before we change behavior, install the things that will catch us when we break it. Duration estimate: 3–4 days of Jesse-time. Exit gate: All four tasks merged, CI green, error boundary verified on a forced crash.

Why: You are an AI agent. Agent-written code without tests is where silent regressions accumulate. Before we touch anything else, we need a way to catch them.

What to do:

  • Create src/__tests__/.
  • Write tests for the highest-value targets — the core data-layer functions:
  • Mock the SQLite layer with an in-memory shim — do not call real SQLite in tests. If expo-sqlite does not expose a node-compatible mode, write a thin in-memory mock implementing the methods we actually call.
  • Add an npm script: npm test.
  • Add a CI workflow (.github/workflows/test.yml) that runs npm test on every PR.

Phase 2 — Server-Side Proxy (The Security Blocker)

Goal: Eliminate the bundled API keys. The single most important pre-launch task. Duration estimate: 4–5 days of Jesse-time. Exit gate: App functions identically with all keys removed from the client bundle. Verified by building a release IPA and grepping for the key prefixes — they must not appear.

Task 2.1 — Architecture decision: where the proxy lives

Decision (made by me, Arya): Use Cloudflare Workers.

Rationale:

  • We have no other backend yet, so we don't want to stand up a server.
  • Cloudflare Workers: free tier comfortably covers us, sub-50ms cold start, simple wrangler deploy, secrets management built in, streaming support (we need it for Claude tool-use loops).
  • AWS Lambda: more ops overhead, cold starts can be rough.
  • Vercel Functions: fine, but we're not in a Next.js project and Vercel's pricing model is less predictable at scale.

If Andy has a strong preference for a different provider, that decision overrides mine. Otherwise: Cloudflare Workers.

Repo layout: New folder proxy/ at the repo root. Separate package.json, separate deploy. Documented in proxy/README.md.

Task 2.2 — Build the proxy

What to do:

- POST /v1/vision — proxies to Gemini 2.5 Flash vision endpoint - POST /v1/chat — proxies to Claude Sonnet 4.6 messages endpoint (with streaming support; the client tool-use loop requires it)

  • Scaffold a Cloudflare Worker in proxy/ via wrangler init.
  • Implement two routes:
  • Auth between client and proxy: client sends a shared bearer token (stored as a secret in the Worker, embedded as a public token in the client).

⚠️ Important nuance — read carefully: A shared client token still lives in the bundle. It is not a security primitive by itself. Its purpose is to give us a kill-switch (rotate the token in the Worker → all old clients are locked out) and a rate-limiting handle. The real security comes from the per-IP and per-token rate limits, not the token itself. This is the right pragmatic call for a v1 single-user → small-cohort launch. We revisit with proper user auth (Sign in with Apple) when we have actual user accounts. Document this trade-off in proxy/README.md.

  • Add rate limiting in the Worker: max N requests per minute per IP, max M requests per day per token. Use Cloudflare's built-in rate limit primitives or a simple KV-backed counter. Conservative limits (e.g., 30/min, 1000/day) — we tune later.
  • Add request logging (no payloads — just timestamps, route, status, latency) for ops visibility.
  • Forward Gemini and Claude error responses transparently — don't swallow them. The client needs the real error to handle correctly.

Acceptance:

  • [x] wrangler dev runs locally
  • [x] Both routes proxy correctly to upstream — verified with a curl test against a deployed staging Worker
  • [x] Auth rejects requests without the bearer token (401)
  • [x] Rate limits return 429 with Retry-After when exceeded
  • [x] Streaming works end-to-end for /v1/chat
  • [x] Logs in Cloudflare dashboard show the test traffic

Task 2.5 — Cassian: CI/CD pipeline for the Worker

Owner: Cassian. Goal: Make the new server-side component boring to ship, easy to verify, and fast to roll back.

The app now has two independently deployable surfaces:

  • The Expo/iOS client, shipped through EAS/TestFlight/App Store.
  • The Cloudflare Worker proxy, shipped through Wrangler to Cloudflare.

That means Phase 2 is not complete when Jesse has code working locally. Phase 2 is complete when Cassian has a repeatable deploy train with gates, secrets, logs, and rollback.

Recommended pipeline shape: GitHub Actions with separate workflows for client checks and Worker checks/deploys. Keep the Worker in proxy/ with its own package.json, wrangler.toml, tests, and README.

Worker environments:

  • local — wrangler dev, using local .dev.vars only. Never committed.
  • staging — automatic deploy from PR preview or merge to develop/staging if that branch exists. If the repo only uses master, use manual workflow_dispatch for staging until branching is formalized.
  • production — deploy only from master after checks pass. Require Cassian approval until the trust ramp says otherwise.

Required GitHub secrets:

  • CLOUDFLARE_API_TOKEN — scoped to deploy this Worker only.
  • CLOUDFLARE_ACCOUNT_ID.
  • WORKER_CLIENT_TOKEN_STAGING and WORKER_CLIENT_TOKEN_PRODUCTION — the shared client bearer tokens.
  • ANTHROPIC_API_KEY_STAGING and ANTHROPIC_API_KEY_PRODUCTION.
  • GEMINI_API_KEY_STAGING and GEMINI_API_KEY_PRODUCTION.
  • Existing EAS/App Store secrets remain owned by the client release workflow, not the Worker workflow.

Important secret rule: upstream AI provider keys must never be committed, logged, echoed in workflow output, or exposed through EXPO_PUBLIC_*. The only public-ish token is EXPO_PUBLIC_PROXY_TOKEN, and that is intentionally just a rate-limit/kill-switch handle, not real authentication.

Checks Cassian should enforce before Worker deploy:

- unauthenticated request returns 401 - authenticated invalid payload returns a controlled 4xx - /v1/vision reaches Gemini with a minimal test fixture if cost/rate limits allow - /v1/chat verifies streaming headers/body shape without logging prompt payloads

  • Install dependencies in proxy/ with a lockfile-backed install (npm ci, pnpm install --frozen-lockfile, or repo-standard equivalent).
  • Run Worker unit tests.
  • Run lint/typecheck if Jesse adds TypeScript or lint config.
  • Run a secret leak scan against the Worker and client bundle patterns: sk-ant, AIza, ANTHROPIC_API_KEY, GEMINI_API_KEY.
  • Run a smoke test against staging after deploy:
  • Block production deploy if staging smoke fails.

Production deploy policy:

  • Deploy production Worker from GitHub Actions, not laptops.
  • Use Cloudflare environments if possible (wrangler deploy --env production).
  • Production deploy requires all checks green plus Cassian approval.
  • Arya reviews architecture/security-affecting PRs before merge.
  • Jesse owns implementation fixes. Cassian owns deploy reliability and may veto release.

Rollback plan:

  • Worker rollback: use Cloudflare deployment rollback or wrangler rollback to the last known-good version.
  • Token rollback: rotate WORKER_CLIENT_TOKEN_* only if the deployed client can be updated or the goal is to intentionally lock out old clients.
  • Provider key rollback: rotate provider keys in Cloudflare secrets and redeploy without changing client code.
  • Client rollback has no true instant path once submitted to Apple; TestFlight can ship a new build, App Store production needs a new review. This is why Worker staging smoke tests are mandatory.

Observability Cassian should wire up:

  • Cloudflare Worker logs with route, status, latency, request id, and environment.
  • No request payload logging. Meal photos, chat text, and health-related context stay out of logs.
  • Alert on elevated 5xx rate, repeated 401/429 spikes, and upstream provider failures.
  • Andy gets dashboard access; Arya and Cassian get enough access to debug failures without seeing secrets.

Acceptance:

  • [x] .github/workflows/worker-ci.yml runs on PRs touching proxy/** and shared config
  • [x] .github/workflows/worker-deploy.yml can deploy staging and production with environment-specific secrets
  • [x] Staging deploy is smoke-tested automatically
  • [x] Production deploy is gated by green checks and Cassian approval
  • [x] Rollback procedure is documented in proxy/README.md
  • [x] Secret leak scan is part of CI and passes
  • [x] Cloudflare logs/alerts are configured without payload logging
  • [x] Cassian signs off that Worker deploys are repeatable from CI

Phase 2 gate: Arya reviews all PRs. Cassian signs off on CI/CD and rollback. Andy rotates keys. Current TestFlight build works through the production Worker. Phase 3 begins.

Goal: Make this an app a stranger can install. Remove "Andy"-specific assumptions. Add flows Apple requires for App Store approval. Duration estimate: 4–5 days. Exit gate: A fresh install on a new device walks a new user from "first launch" to "first meal logged" without any reference to Andy and without any hardcoded personal values.

Why: Architecture review item #6. DEFAULT_GOALS hardcodes displayName: 'Andy' and targetWeightKg: 83.9. Cannot ship.

Task 3.2 — Account deletion flow

Why: Apple App Store guideline 5.1.1(v): if your app supports account creation, it must support account deletion. We technically don't have "accounts" (no sign-in), but reviewers interpret this broadly — any persistent user data should be deletable from within the app. We do this defensively.

What to do:

  • In Settings, add "Delete all my data" with a confirmation flow ("This will permanently delete all your meals, weights, and settings. This cannot be undone.").
  • On confirm: DELETE FROM each user-data table in a single transaction. Reset to fresh-install state. Trigger onboarding on next launch.
  • Log this action in Sentry (as a breadcrumb, not an error).

Acceptance:

  • [ ] Delete flow works, app returns to onboarding state
  • [ ] Confirmation prevents accidental deletion (two taps minimum)
  • [ ] Test covers the deletion logic against the in-memory SQLite mock

Phase 4 — Reliability Hardening

Goal: Pay down the remaining debts from the architecture review that we don't want haunting us in production. Duration estimate: 2–3 days.

Task 4.1 — Cryptographic UIDs

What to do: Replace every Math.random().toString(36).slice(2) + Date.now().toString(36) with crypto.randomUUID(). (RN 0.74+ has it natively; if Expo's runtime version does not expose it, use expo-crypto's randomUUID.)

Acceptance:

  • [ ] grep -r 'Math.random' src/ returns no UID-generation hits
  • [ ] All tests still pass
  • [ ] One unit test verifies UID format is a valid UUID v4

Task 4.2 — Migration robustness

What to do:

  • Wrap each migration in its own withTransactionAsync.
  • Update db_version inside the same transaction as the migration DDL.
  • If any step throws, the transaction rolls back, db_version stays at the old value, and the next launch retries.
  • Add a test that simulates a failing migration and verifies db_version is unchanged.

Acceptance:

  • [ ] Per-migration transactional behavior verified by test
  • [ ] Documented in code comments

Task 4.3 — pendingImage cleanup

What to do: Always pendingImage.clear() on confirm.tsx mount. Move to a temp file path (use expo-file-system's cacheDirectory) rather than in-memory base64. Prefer the temp file path — more robust on backgrounding. Document the choice.

Acceptance:

  • [ ] Backgrounding the app mid-analysis no longer pollutes the next capture
  • [ ] Manual test documented in PR description

Task 4.4 — Cold start performance baseline

What to do:

  • Measure cold-start time on an iPhone XR (oldest device we will claim to support — confirm with Andy if that is right).
  • Document in JESSE-NOTES.md. Goal: < 2 seconds to first interactive screen.
  • If it is worse, identify the largest contributors (likely SQLite migration, Sentry init, or bundle size) and propose a fix for me to review before implementing.

Acceptance:

  • [ ] Baseline number documented
  • [ ] If > 2s, a proposal exists

Out of Scope (Explicitly)

These are not part of this plan. If they come up: write them down, do not do them.

  • New features (calisthenics, YouTube video integration) — deferred to post-launch roadmap
  • iCloud / CloudKit sync — explicit defer. Per Open Question #2 in the review: we are NOT solving cross-device sync for this launch. iOS device backup covers data preservation acceptably for v1. Users will be told (in onboarding/settings) that data is on-device. We revisit sync after we have actual user feedback.

  • Onboarding A/B testing infrastructure — premature
  • Analytics beyond Sentry — premature, and Apple privacy labels get more complicated for every SDK we add
