Hotfix Spec — "No data" on Confirm Meal (photo lost between camera and confirm)

Author: Arya (Architect) Date: 2026-06-06 Severity: P0 — core flow still broken after PR #14. Owner: Jesse (implementation) Base: master @ e5403a9 (post PR #14) Approval status: APPROVED to implement — Fix B now, Fix A as structural follow-up.

---

Symptom (after PR #14)

PR #14 fixed the first bug: navigation to Confirm Meal now succeeds. But the user takes a photo, reaches the Confirm Meal screen, and gets the alert:

> No data — Something went wrong - no photo or query found.

Where That String Comes From

app/log/confirm.tsx shows that alert only when promise is null — i.e. both query is empty AND pendingImage.get() returned null:

useEffect(() => {
  const pendingImageUri = pendingImage.get();
  pendingImage.clear();                 // <-- cleared at mount, before read completes
  let promise = null;
  if (query) promise = visionService.lookupByName(query);
  else if (pendingImageUri) promise = FileSystem.readAsStringAsync(...)...;
  if (!promise) { Alert.alert('No data', ...); return; }   // <-- this fires
  ...
}, []);

So the image is being lost between camera setting it and confirm reading it. pendingImage.set() always sets _uri to at least the source URI and resolves, so the shelf is non-null right after camera. For confirm to read null, the shelf must be cleared between camera and confirm's surviving mount.

Root Cause — two compounding defects

Defect 1 (the trigger): Phase 3 onboarding gate re-fires on every navigation

app/_layout.tsx:

useEffect(() => {
  getDb().then(() => isOnboardingComplete()).then((complete) => {
    const onOnboarding = pathname === '/onboarding';
    if (!complete && !onOnboarding) router.replace('/onboarding');
    if (complete && onOnboarding) router.replace('/(tabs)');
  });
}, [pathname, router]);          // <-- depends on pathname → runs on EVERY nav

Navigating camera → /log/confirm changes pathname, re-running this effect and its async DB round-trip. This churns the modal navigation stack (remount). Note: isOnboardingComplete() is stable-true for an existing user, so this is NOT redirecting to onboarding — the damage is the remount churn, not a wrong redirect. This bug also causes subtle remount churn on every navigation app-wide, not just the camera flow.

Defect 2 (the fragility PR #13/4.3 introduced): confirm clears the shelf at mount

4.3 moved pendingImage.clear() to the top of confirm's effect. So: confirm mounts → reads URI → clears shelf → begins file read. If the layout churn (Defect 1) remounts confirm, the second mount reads an already-empty shelf → null → "No data." Before 4.3, clear() ran only AFTER analysis, so a remount mid-flow still found the image. The patch did not create the remount; it made the remount fatal.

---

The Fix

Fix B — REQUIRED NOW (highest leverage, robust to the race)

confirm.tsx already receives imageUri as a route param, which survives remounts (route params are serialized into the URL). Treat pendingImage as a fast-path cache and fall back to the param. One-line core change:

useEffect(() => {
  const pendingImageUri = pendingImage.get() ?? imageUri ?? null;  // fall back to route param
  // Do NOT clear at the top. Clear/remove only after we have consumed it (see below).
  ...
  if (query) { promise = visionService.lookupByName(query); }
  else if (pendingImageUri) { promise = FileSystem.readAsStringAsync(pendingImageUri, ...)... ; }
  ...
}, []);

Additional hardening in the same effect:

top. Capture the URI, kick off analysis, and only pendingImage.remove(pendingImageUri) in .finally (already there) — drop the mount-time clear() entirely, or guard the effect against double-invocation with a useRef(false) ran-once flag.

read works against it on-device. It is a readable file:// URI from the picker, so it should. If readAsStringAsync ever throws on it, the existing .catch already shows "Analysis failed" rather than silently dying.

  • Move the clear/remove to after consumption. Do not pendingImage.clear() at mount
  • Because imageUri (the original picker URI) may now be the source read, confirm the
