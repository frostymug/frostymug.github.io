Frost Fitness Worker — Rollback Procedures

Owner: Cassian (SRE) Last updated: 2026-06-02

---

When to Roll Back

Roll back the Worker if, after a production deploy:

  • Smoke tests fail automatically (the deploy workflow handles this)
  • Error rate in Cloudflare dashboard spikes above baseline
  • 5xx rate exceeds ~1% sustained over 5 minutes
  • Upstream provider auth failures (Worker started seeing 401s from Anthropic/Gemini)

Do NOT roll back for elevated 429s — that means rate limiting is working.

---

Scenario 1: Bad Worker Code

The fastest path. Reverts the Worker binary to the previous Cloudflare deployment.

npx wrangler rollback --env production

Recovery time: < 60 seconds.

Verify with a manual smoke run:

SMOKE_ENV=production \
PROXY_URL=<production-url> \
PROXY_TOKEN=<production-token> \
node proxy/scripts/smoke.js

---

Scenario 2: Compromised or Leaked Client Token

Use this when WORKER_CLIENT_TOKEN_PRODUCTION has been exposed (e.g., leaked in logs, screenshot, etc.).

``bash npx wrangler secret put WORKER_CLIENT_TOKEN --env production ``

  • Generate a new token (any secure random string, 32+ chars)
  • Update the secret in the Worker:
  • Update WORKER_CLIENT_TOKEN_PRODUCTION in GitHub Secrets
  • Update EXPO_PUBLIC_PROXY_TOKEN in the iOS client config and ship a new TestFlight build

Effect: All existing clients using the old token are immediately locked out. This is intentional — it is the kill-switch. Coordinate with Andy on client update timing before rotating.

---

Scenario 3: Compromised Upstream Provider Key

Use this when an Anthropic or Gemini key stored in the Worker has been exposed.

``bash npx wrangler secret put ANTHROPIC_API_KEY --env production # or npx wrangler secret put GEMINI_API_KEY --env production ``

``bash npx wrangler deploy --env production ``

  • Andy rotates the key in the Anthropic/Google console (Cassian does not hold these credentials)
  • Update the Worker secret:
  • Redeploy the Worker (no code change needed — secrets update takes effect on next request after redeploy):
  • Run smoke tests to confirm upstream calls succeed with new key

No client change needed for this scenario.

---

Scenario 4: Client Rollback (iOS App)

There is no fast rollback path for a submitted iOS app.

Once a build is live on the App Store, the only path is a new submission (another Apple review cycle). This is why Worker staging smoke tests are mandatory — the Worker and the client are behaviorally coupled.

For TestFlight: a new build can be promoted in minutes. If production is on fire and the issue is client-side, prioritize a new TestFlight build as the fastest path.
