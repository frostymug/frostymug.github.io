Ice Age Trail App Deployment Spike

Status: Draft Prepared by: Aspen Date: 2026-06-18 Purpose: Andy-only learning deployment, not an official Ice Age Trail Alliance app.

Goal

Prove whether Lisa Siewert's static ArcGIS prototype can be deployed from the public GitHub repo using an Andy/Aspen-controlled ArcGIS Location Platform API key, hosted somewhere we control, with enough restriction and usage visibility to stay inside the free tier.

This is a technical learning spike. It should answer whether the existing project can be made visible on the web without buying the wrong Esri license, relying on Lisa's university resources, or implying Alliance ownership.

Source Repo

Lisa's repo: https://github.com/ecolville/GEOG778_Capstone

Current evidence:

  • The app is a static web app: index.html, app.js, style.css, segments.json, PDFs, and supporting HTML/docs.
  • No deployed public URL has been found.
  • The webinar demo appears to run locally from 127.0.0.1.
  • The app expects window.ARCGIS_API_KEY from config.local.js, which is intentionally not committed.

Deployment Hypothesis

The app can probably run as a static deployment if we provide a production-safe API key config and the referenced ArcGIS services are publicly reachable with the right authentication/usage terms.

Recommended first path:

  • Fork or clone Lisa's repo under an Aspen/Andy-controlled namespace.
  • Create an ArcGIS Location Platform API key under Andy's account.
  • Deploy the static app to a controlled host such as Cloudflare Pages, GitHub Pages, or Netlify.
  • Restrict the API key to the deployed app's referrer/domain.
  • Verify map load, layer load, segment interactions, POI behavior, parking layer, activity links, and mobile basics.
  • Confirm Esri usage metrics show requests against Andy's key and remain inside the free tier during testing.

Non-Goals

  • Do not present this as the official Ice Age Trail Alliance app.
  • Do not use ArcGIS for Personal Use as the assumed path for this deployment.
  • Do not ask Lisa or the Alliance for credentials for this spike.
  • Do not add public contribution/photo upload features.
  • Do not redesign the app during this spike unless a minimal change is required to make deployment work.
  • Do not commit secrets or unrestricted API keys.

Success Criteria

The spike succeeds if Andy can open a private/low-visibility deployed URL and confirm:

  • The map renders in a browser without local files.
  • ArcGIS basemap and FeatureLayer requests succeed using Andy's API key or an explicitly documented no-key/public-layer path.
  • Core UI works: sidebar, segment selection, POIs, parking, activity links, and focus mode.
  • The API key is restricted to the deployment domain/referrer.
  • Esri usage telemetry is visible in Andy's account.
  • The repo includes enough setup/deploy notes that a future engineer can repeat the deployment.

The spike should also produce a clear failure answer if deployment is blocked, including the exact blocker: API key scope, private layers, licensing, CORS/referrer restriction, missing assets, hosting behavior, or code assumptions about localhost.

Preferred Hosting Options

Option 1: Cloudflare Pages

Pros: already in Andy/Aspen orbit, easy static deploys, custom access controls possible later. Cons: needs repo connection or CLI deployment setup.

Option 2: GitHub Pages

Pros: simplest if the fork is public and the app is static. Cons: public by default, weaker control over visibility unless paired with private repo features and plan constraints.

Option 3: Netlify

Pros: simple static deployment and environment handling. Cons: another platform to manage.

Default recommendation: use Cloudflare Pages if low-visibility access control matters; use GitHub Pages if speed and simplicity matter most.

Config Strategy

Current repo pattern expects a gitignored config.local.js that sets window.ARCGIS_API_KEY.

For a static deployment, evaluate the smallest safe option:

  • Keep config.local.js for local development.
  • Add a committed config.example.js with placeholder values.
  • Add deployment documentation explaining how the production config is generated or injected.
  • Avoid committing the real API key, even if referrer-restricted.

Possible deployment approaches:

  • Generate config.js during build/deploy from an environment variable.
  • Use a Cloudflare Pages build command to write a tiny config file from a protected variable.
  • If GitHub Pages is used, consider whether the key must be public in the built artifact and whether referrer restriction makes that acceptable for a learning-only demo.

Important: browser-based ArcGIS API keys are inherently visible to users. The real protection is restricting allowed referrers, limiting scopes, and monitoring usage.
