Ice Age Trail App Kickoff Brief

Status: Draft Prepared by: Aspen Date: 2026-06-18 Sources: Andy/Lisa call transcript from 2026-06-10; Lisa's public repo: https://github.com/ecolville/GEOG778_Capstone; YouTube webinar, 2/23/26 Integrating GIS & Geoscience: A Capstone Project on the Geology of the Ice Age Trail: https://youtu.be/F7w8FFOu0qQ

Executive Read

This is not a blank-slate app idea. Lisa already has a working static ArcGIS JavaScript prototype with real educational content, geologic layers, trail segment interactions, POIs, and activity PDFs. The kickoff question is how to turn a promising capstone into something maintainable, legally clean, operationally stable, and expandable enough for the Ice Age Trail Alliance to consider adopting.

The most important product shift from the call: Lisa started with a geology/educator app, but user demand points toward a broader natural-history guide. Hikers ask things like "what am I looking at?" and "why is this burned?" The long-term project may need to support geology, ecology, landscape history, trail interpretation, and educator resources without becoming a sprawling encyclopedia.

Current Prototype

Repo: ecolville/GEOG778_Capstone

The repo is a static web app:

  • index.html loads ArcGIS JavaScript API and app UI.
  • app.js creates the map, layers, sidebar tabs, trail-segment click behavior, POI layer, parking layer, and focus mode.
  • style.css handles the map/sidebar/modal UI.
  • segments.json contains segment-specific content: overview, bedrock, glacial story, POIs, concepts, and activities.
  • PDF activity sheets live at repo root and under docs/.
  • config.local.js is expected to provide window.ARCGIS_API_KEY, but is not committed.

Observed external dependencies in code:

  • ArcGIS JS API from https://js.arcgis.com/4.29/
  • ArcGIS stylesheet from https://js.arcgis.com/4.25/esri/themes/light/main.css
  • ArcGIS feature services for bedrock geology, surficial geology, POIs, IAT trail segments, and parking areas.
  • YouTube concept links inside segments.json.
  • Ice Age Trail Alliance links in about-this-map.html.

Deployment evidence:

  • No public app URL was found in the repo.
  • The likely GitHub Pages URL, https://ecolville.github.io/GEOG778_Capstone/, returns 404, and GitHub Pages metadata was not available for the repo during inspection.
  • Andy's review of Lisa's webinar found the app running from 127.0.0.1, which supports the working assumption that the prototype has not been publicly deployed.

What Lisa Wants

Lisa's stated goal is to make the app better structured so it can expand statewide and support more points of interest, more segment information, and possibly more educational content if the Alliance greenlights the project.

Concrete asks and concerns from the call:

  • Move the project away from university-controlled resources where ownership/licensing may be ambiguous.
  • Find a hosting/API-key setup that does not depend on Lisa's personal or university accounts.
  • Figure out whether ArcGIS hosting/API costs are really a blocker and whether the Alliance's existing ArcGIS setup can carry the project.
  • Replace or outgrow the current segments.json approach with a better content/data model.
  • Create a way for people to submit photos with location metadata and notes.
  • Make the app more visually useful by adding POI photos.
  • Preserve Lisa's role and authorship while making it easier for others to help.
  • Keep her time burden low; she has interest and expertise, but not spare execution capacity.

Audience Hypotheses

Primary likely audiences:

  • Teachers preparing classroom or field-trip activities tied to the Ice Age Trail.
  • Hikers and trail volunteers who want to understand what they are seeing.
  • Ice Age Trail Alliance staff/volunteers evaluating whether this can become an official or semi-official resource.
  • Lisa as content expert/steward, needing a sane way to add and review content.

Important tension: educator depth versus hiker simplicity. The current app leans educator/geology. The pull from real users is broader and more field-facing. Product strategy should decide whether this becomes one unified experience or two modes: educator planning and trail interpretation.

Scope Already Present

The prototype includes custom content for a small number of segments, including New Hope, Kettlebowl, Ringle, and Dells of the Eau Claire. Lisa said the broader mapped coverage is only four counties because that was what she could manage as a grad student, and even within that region not every segment has custom content.

The current region is around Marathon, Lincoln, Portage, and nearby north-central Wisconsin context from the capstone effort.

Major Risks

Ownership and Licensing

Lisa used university resources and ArcGIS licenses during creation. She is worried that university-system rules could create ambiguity around ownership. Before serious adoption, the project needs a clean position on code, data, content, photos, and hosting accounts.

ArcGIS Dependency and Cost

The app depends on ArcGIS JavaScript and ArcGIS-hosted FeatureLayers. Lisa found pricing that looked expensive when approaching ArcGIS as an individual educator. The Alliance already uses ArcGIS Online for its current interactive map, so the best path may be Alliance-owned hosting and API keys rather than a separate paid setup.

Esri's ArcGIS for Personal Use license is useful for individual learning and personal noncommercial projects, but it is not a clean path for an Alliance-facing project because Esri excludes use for the benefit of third parties, including nonprofit entities. It also appears technically mismatched to Lisa's current prototype: Esri's API-key documentation says API key authentication is not available in ArcGIS Online subscriptions purchased through ArcGIS for Personal Use, while Lisa's code expects window.ARCGIS_API_KEY. A private Andy-only learning deployment may still be possible through a separate ArcGIS Location Platform account or a different authentication approach, but Personal Use should not be treated as the production deployment answer.

There is no evidence yet of a durable public hosting setup for the application itself. Treat deployment as an open architecture/operations work item, not as an already-solved piece of the project.

A separate Deployment Spike now captures the Andy-only learning path: deploy Lisa's static app from the repo using an Andy/Aspen ArcGIS Location Platform API key, controlled hosting, referrer restrictions, and free-tier usage visibility.

Content Architecture

segments.json was a quick way to get content into the app for graduation. It will become painful as content expands statewide, adds POIs/photos, and needs review workflows. The project needs a structured content model before large-scale content entry begins.

Contributor Photos

Photos are valuable, but they introduce metadata, permissions, moderation, storage, and privacy questions. A submission workflow should capture location, notes, contributor permission, and review status. Do not simply tell people to email photos or drop them into a personal cloud album as the long-term plan.

Alliance Adoption

The Alliance has enthusiasm around the idea, but no formal greenlight is captured here. The kickoff should separate exploratory prototype work from official Alliance commitments.

Early Direction Options

Option A: Stabilize Current Static App

Keep the current static ArcGIS app, improve documentation, configuration, API-key handling, deployment, and content hygiene. Lowest effort and likely best first technical step.

Option B: Static App Plus Structured Content Pipeline

Keep the front end static, but move content authoring into a structured source such as Airtable, Google Sheets, Git-backed Markdown/JSON, or a small CMS. Generate app-ready JSON at build time. Good middle ground if Lisa/volunteers need easy editing without building a full backend.

Option C: Full Web App with Database and Submission Workflow

Build a hosted app with authentication, database-backed content, contributor uploads, moderation, and admin tools. Powerful, but likely too much before Alliance buy-in and operational ownership are clear.

Option D: Alliance ArcGIS-Centered Implementation

Explore whether the Alliance can host layers, API keys, and data/content workflows inside its existing ArcGIS Online environment. This may solve ownership and cost but could constrain UX and editorial workflow.

Recommended Kickoff Stance

Start with discovery and architecture framing, not coding.

The first technical target should be a "stewardable prototype" plan: keep the current prototype working, document its dependencies, map every external service and data source, define the target content model, and identify the smallest migration that removes the scariest account/API-key risk.

Do not jump straight to a full database app. The greenlight, ownership model, contributor moderation responsibility, and hosting owner are not clear enough yet.

  • Review dependency/hosting findings, especially ArcGIS API key and layer ownership.
  • Review content/data model recommendation.
  • Decide whether to pilot photo/POI contribution intake.
  • Decide what, if anything, to ask from the Alliance before more build work.
  • Pick the next concrete deliverable Andy/Aspen will provide.

Aspen Recommendation

Treat this as a real project, but keep it on the low-stakes side of the safety ramp until public submissions, official Alliance branding, and long-term hosting are involved. The most valuable near-term move is not building a shiny replacement. It is giving Lisa and the Alliance confidence that the existing prototype can become durable without creating legal, cost, or volunteer-maintenance headaches.
