Ice Age Trail Architecture Options After Lisa Check-in

Date: 2026-07-11 Prepared by: Aspen Source: Andy/Lisa check-in on 2026-07-10, preserved in CALL-NOTES-GEMINI-2026-07-10.md Status: Draft for Andy to send or adapt for Lisa

Executive Summary

The main architecture decision is not really front-end hosting anymore. The Cloudflare deployment spike proved the existing static app can be hosted under Andy/Aspen control. The real decision is where the project data layers live, who controls them, and how Lisa can keep expanding them safely while a student contributes new bedrock geology data.

My recommendation is a two-track path:

  • Use ArcGIS as the editing system for now, because Lisa and the student workflow are already there. Let Lisa move/copy the important project layers from UW-Stevens Point to her Minnesota State account as the near-term safer home, and have her document the steps.
  • Add an independent backup/snapshot pipeline, not a full custom backend. Periodically export or copy the ArcGIS layers to a neutral, versioned location so the project is protected if an institution account changes, disappears, or becomes politically messy.

Do not build a custom geospatial backend yet. That would be too much architecture for the actual risk. The risk is data sovereignty and continuity, not query performance.

What Changed In The Lisa Conversation

  • Lisa's priority is the data layers more than the app shell.
  • The current app can be deployed independently, but its real runtime dependency is still ArcGIS-hosted data.
  • The current geology/POI layers are tied to UW-Stevens Point infrastructure, which Lisa does not fully trust long term.
  • Lisa has a Minnesota State account that may be a more stable near-term destination.
  • A student will likely need to enter data through the university ArcGIS account for supervision/funding compliance.
  • The bedrock geology layer is the immediate funded work; glacial geology and POIs remain important but separate layers.
  • segments.json is a separate non-georeferenced content layer linked by segment name, and it is not yet a clean long-term content model.

Recommended Path

Phase 1: Protect The Data Without Rebuilding The World

Lisa should proceed with the Minnesota State migration/copy if she is comfortable with the rights situation there. She should document:

  • Which layers were moved or copied.
  • Original source URLs and destination URLs.
  • Item owners and organization names.
  • Sharing settings.
  • Editing permissions.
  • Any terms-of-use or attribution text.
  • Whether the data was copied, exported/imported, or transferred.

Andy/Aspen should treat that documentation as the source for a lightweight migration playbook.

Phase 2: Add A Siphon/Backup Pipeline

Build or document a small repeatable process that exports the important ArcGIS layers on a schedule or on demand into a neutral, versioned backup location.

Start with the Lisa-owned educational layers:

  • Bedrock geology.
  • Glacial/surficial geology.
  • Geological POIs.
  • Segment supplemental data, currently segments.json.

For official IATA trail and parking layers, do not copy/rehost without explicit permission. For those, it may be enough to record dependencies and keep them live from the official source.

Phase 3: Decide The Future Steward

After the student workflow is clearer, pick the long-term owner:

  • If this stays Lisa-led educational research/content: Minnesota State ArcGIS may be acceptable as the working source, with independent backups.
  • If this becomes an official Ice Age Trail Alliance app: IATA should own or explicitly bless the production ArcGIS organization/layers.
  • If this remains an Andy/Lisa prototype: Andy/Aspen can host the front end and possibly prototype copies, but the docs should say clearly that Aspen is not the authoritative data owner.

My Recommendation To Lisa

I would tell Lisa:

> The app itself can be hosted almost anywhere. The strategic choice is where the data lives. Since your immediate goal is developing the bedrock geology layer with a student, I would keep ArcGIS as the editing environment for now. Move or copy the important layers from UWSP to your Minnesota State account if that is the safer institutional home, and document exactly how that migration works. In parallel, Andy/Aspen can help design a lightweight backup/export process so the project is not dependent on any one university account. I would not build custom GIS infrastructure yet. Once the data layers mature and IATA's role is clearer, we can decide whether IATA should become the official production steward.

Pros And Cons To Send Lisa

Recommended Near-Term Architecture: Minnesota State ArcGIS + Independent Backup

Pros:

  • Keeps Lisa and the student in familiar ArcGIS tooling.
  • Avoids a distracting rebuild while the bedrock layer is the priority.
  • Reduces dependence on UWSP if that employment/account situation is unstable.
  • Gives Andy/Aspen a clear way to protect the data through periodic exports.
  • Leaves the future IATA official path open.

Cons:

  • Minnesota State is still an institution account, not a neutral permanent home.
  • Student editing rights and funding/supervision rules still need confirmation.
  • Any copied layers can drift unless the source-of-truth workflow is explicit.
  • Permissions and terms still need cleanup before public/official launch.

Longer-Term Official Architecture: IATA ArcGIS Stewardship

Pros:

  • Best fit if the app becomes official.
  • Cleanest ownership and continuity story for public users.
  • Easier to define staff/admin/editor roles.
  • Official trail/parking data can stay authoritative.

Cons:

  • Slower and more bureaucratic.
  • Requires IATA buy-in and account/admin details.
  • May complicate Lisa's role if board/payment conflict remains unresolved.
  • Premature before the educational data model and student workflow mature.

Deferred Architecture: Custom Backend

Pros:

  • Most independent.
  • Could eventually support advanced features, custom editing, public contributions, offline packs, or non-Esri maps.

Cons:

  • Not justified yet.
  • Expensive to build and maintain.
  • Creates security, backup, admin, and migration problems immediately.
  • Distracts from the actual next milestone: durable data layers.

Questions For Lisa

  • Which exact ArcGIS layers are Lisa-created educational layers versus IATA/third-party official layers?
  • Can the student edit in Minnesota State ArcGIS, or must all funded work happen under UWSP?
  • Does Minnesota State allow Lisa to share/export/republish this data for a public-facing educational app?
  • Are there photos, text, PDFs, or external links in the POI/segment content with separate copyright constraints?
  • Does Lisa want the Minnesota State account to be the working source of truth, or just a backup destination?
  • Is IATA interested in owning the production data layer infrastructure eventually, or only reviewing/using the app?
  • How often should data be backed up or siphoned during the student project: weekly, after each editing session, or before each app release?

Andy's Concrete Next Steps

  • Send Lisa the options summary above, especially the recommendation to keep ArcGIS editing but add independent backup.
  • Ask Lisa to document her UWSP to Minnesota State migration steps as she performs them.
  • Ask for the exact source and destination layer URLs once she has them.
  • Have Arya inspect those URLs and produce a layer inventory: owner, sharing, schema, editability, terms, and app dependency.
  • Decide whether Aspen/Jesse should build a tiny ArcGIS export/snapshot script or just document a manual export process first.

Important Caveat

These notes come from Gemini's meeting summary, not a verbatim transcript. The direction is clear enough for planning, but ownership, copyright, and institutional policy claims should be confirmed with Lisa before anyone treats them as hard requirements.
