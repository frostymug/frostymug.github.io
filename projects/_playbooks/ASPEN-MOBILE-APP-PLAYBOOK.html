Aspen Frost Mobile App Playbook

Owner: Arya (Architect) | Created 2026-06-28 | Status: Active reference Generalized from the shipped Frost Fitness app (Expo/EAS, live on the App Store via TestFlight as of 2026-06-11).

Purpose

This is the reusable "how Aspen Frost builds and ships a mobile app" playbook. It exists so that no future mobile project — Frost Rocks included — re-asks questions we already answered in production with Frost Fitness. When a new app starts, an architect should be able to copy this stack wholesale and spend their thinking budget on what is different about the new app, not on re-deciding the platform, the proxy, the CI/CD, or the release process.

Source of truth: the frost-fitness repo (frostymug/frost-fitness) and the shared project docs at projects/frost-fitness/ (ARCHITECTURE.md, PRODUCTIZATION-PLAN.md, CI-CD.md, RELEASE-PROCESS.md, STATUS.md). When this playbook and those disagree, the repo wins and this doc should be updated.

The AI Proxy Pattern (use whenever an app calls an AI provider)

The single most important productization lesson from Frost Fitness: API keys must never ship in the client bundle. TestFlight/App Store IPAs can be extracted and reverse-engineered; a leaked Anthropic/Gemini key has account-level write access.

The pattern (lives in proxy/, a Cloudflare Worker):

  • Client holds only EXPO_PUBLIC_PROXY_URL + EXPO_PUBLIC_PROXY_TOKEN (a shared bearer token).
  • Worker holds the real provider keys as Worker secrets (ANTHROPIC_KEY, GEMINI_KEY, etc.).
  • Worker enforces model + max_tokens caps server-side so the client can't escalate to an expensive model.
  • Worker rate-limits by IP and by token (CF [[ratelimits]] bindings, no KV needed).
  • Worker logs {route, status, latencyMs, ts, reqId} only — never payloads (no photos, no user text).
  • The shared token is a kill-switch: rotating it instantly locks out all clients. It is not per-user auth — layer Sign in with Apple / CF Access on top later if a product needs real accounts.

Cost instrumentation: log token counts + latency + success/failure per call (Frost Fitness uses an ai_call_log SQLite table). Cost spikes should be visible before the credit-card bill, not after.

Offline / Local-First Contract

Default posture for a field/travel app: capture never depends on the network.

  • All user input (photo, note, location, timestamp, tags) writes to local SQLite immediately and is never blocked on connectivity.
  • Anything requiring a network call (AI enrichment, sync) is queued and marked needs-context, then drained when connectivity returns — silently in the background, or at minimum on next app open.
  • The UI says so plainly ("Saved. I'll add context when you're back online") — a queued state, not a spinner or an error.
  • Durability caveat: local-only means the device is the only copy. Cheap mitigations without a backend: save photos to the device camera roll (survives in iCloud Photos) and offer a one-tap export. Real backup (iCloud/CloudKit/backend sync) is a separate, larger decision — defer until a product earns it.

CI/CD & Release (owned by Cassian / SRE)

The pipeline already exists and is generalizable. New apps copy these into the repo:

  • .github/workflows/test.yml — app test suite on PR.
  • .github/workflows/worker-ci.yml — proxy PR CI: install, typecheck, lint, tests, secret scan, Wrangler dry run.
  • .github/workflows/worker-deploy.yml — staging/production Worker deploy, gated by a GitHub Environment (worker-production) requiring Cassian approval, with post-deploy smoke tests + auto-rollback.
  • proxy/scripts/smoke.js — post-deploy smoke tests (auth, routes, streaming headers, rate limits).

Release / rollback runbook (RELEASE-PROCESS.md):

  • Bad Worker code: wrangler rollback --env production — < 60s recovery.
  • Leaked client token: rotate Worker secret + GitHub Secret + ship new client build. Kill-switch effect is intentional.
  • Leaked provider key: Andy rotates in the provider console (SRE does not hold these), update Worker secret, redeploy. No client change.
  • iOS client rollback: there is NO fast rollback for a submitted App Store build — only a new review cycle. TestFlight builds can be promoted in minutes, so a new TestFlight build is the fastest client-side fix. This is why Worker staging smoke tests are mandatory: client and Worker are behaviorally coupled.
  • Do not roll back on elevated 429s — that's rate limiting working as intended.

Build & Submit Mechanics (the un-skippable details)

`` eas secret:create --scope project --name EXPO_PUBLIC_PROXY_URL --value "https://<app>-proxy.aspenfrost.com" eas secret:create --scope project --name EXPO_PUBLIC_PROXY_TOKEN --value "<token>" eas secret:create --scope project --name EXPO_PUBLIC_SENTRY_DSN --value "https://..." ``

  • EAS profiles (eas.json): development (dev client, internal), preview (internal release build), production (store build). appVersionSource: remote, autoIncrement: true.
  • Bundle ID / package: com.<appname>.app pattern.
  • Build number: App Store rejects duplicate build numbers. autoIncrement handles it via EAS; if building manually, bump buildNumber in app.json before every production build.
  • Permissions: declare every device permission with a real infoPlist usage string (camera, photo library, location, health, etc.) — Apple rejects vague or missing strings. Copy Frost Fitness's wording as a template.
  • ITSAppUsesNonExemptEncryption: false in infoPlist to skip the export-compliance prompt (true for our apps — standard HTTPS only).
  • Privacy + Terms URLs: required for App Store. Host on docs.aspenfrost.com/<app>/privacy and /terms (already the pattern).
  • EAS secrets for builds (not .env.local, which is dev-only):
  • Submit: eas build --profile production --platform ios then eas submit (with ascAppId in eas.json). Promote to TestFlight testers in App Store Connect.

Testing Doctrine — Runtime Compatibility Is a First-Class Boundary

The hardest-won lesson from Frost Fitness (the production photo-submit failure): Node/Jest is not the Expo/Hermes runtime. Tests that pass in Jest can hide missing or partial APIs in the actual app runtime.

  • Any code touching runtime-provided APIs (crypto — esp. getRandomValues present while randomUUID absent — FileSystem, camera/media, SQLite adapters, route params, browser-shaped globals like Blob/URL/atob/fetch) needs tests that model the deployed shape of the API: full API exists, partial exists, missing, and fallback.
  • Prefer real local utilities in screen tests; mock only at the external-service / device boundary.
  • Every production crash/error becomes a regression test that recreates the exact failing capability.
  • For UID/primary-key generation: prefer crypto.randomUUID() → fall back to v4 from crypto.getRandomValues → only then Math.random(). Never trust Math.random() for keys that might ever sync.

iOS-First Testing Strategy

Manual TestFlight testing is useful, but it is the final confidence check, not the test strategy. Aspen mobile apps should build confidence in layers, with the cheapest reliable check running first and native iOS automation added as soon as macOS infrastructure exists.

Constraint: iOS Simulator requires macOS + Xcode. Headless Ubuntu cannot run native iOS simulator tests. Ubuntu can still run Jest/RNTL, Expo lint/typecheck, proxy tests, and optional Expo Web Playwright smoke tests.

Required baseline on every mobile app

  • npm test -- --ci using jest-expo.
  • npx tsc --noEmit for TypeScript safety.
  • npx expo lint for Expo/React Native lint coverage.
  • React Native Testing Library coverage for core screen flows, loading states, empty states, permission-denied states, and local persistence boundaries.
  • Stable testID props on controls/screens that will become native E2E targets.
  • PR handoffs include an evidence section: commands run, results, and screenshots or simulator notes when UI is touched.

Native E2E default

Use Maestro as the default native E2E tool unless Arya identifies a reason to choose Detox. Maestro is the preferred first tool because the flows are black-box, readable, and portable across iOS Simulator and Android Emulator. Detox remains available for cases that need deeper React Native synchronization.

Rollout ladder

  • Linux-safe confidence now: strengthen Jest/RNTL, lint, typecheck, and proxy CI. This is immediately runnable from Ubuntu and should be required on PRs.
  • iOS preparation now: add simulator-capable EAS profiles and stable testIDs. Write Maestro flows, but do not claim native iOS coverage until they run on macOS.
  • iOS simulator automation next: add a macOS runner (hosted GitHub Actions, Bitrise/Codemagic, or self-hosted Mac mini) and run Maestro against iOS Simulator on PRs/nightly depending on cost.
  • Real-device smoke later: add BrowserStack, Sauce Labs, or AWS Device Farm only for release candidates or device-specific risk. This supplements simulator tests; it does not replace them.

Current repo baseline, July 2026

  • frost-fitness: Jest/RNTL exists with screen and service tests; GitHub Actions currently runs Jest, but should also run typecheck and Expo lint.
  • frost-rocks: Jest exists with logic/database/export coverage; GitHub Actions already runs typecheck, Expo lint, and Jest.
  • Neither app currently has Maestro, Detox, Playwright, native iOS simulator automation, real-device automation, or simulator-specific EAS build profiles.

Productization Ladder (personal → product)

Frost Fitness shipped as a personal app first, then climbed. Know which rung you're on; it changes what's a blocker vs. acceptable debt:

  • Personal / single-user (TestFlight): Local SQLite, proxy for AI keys, Sentry. Hardcoded personal defaults OK. No onboarding needed.
  • Small beta cohort: Add error boundaries, onboarding/first-run defaults (strip hardcoded personal values), basic test coverage.
  • Multi-user product: Per-user auth (Sign in with Apple), migration hardening (each migration in its own transaction), sync/backup strategy, UID hardening before any cross-device merge.
  • App Store public: Privacy/terms URLs, App Store positioning copy, screenshot package, real reviewer-facing onboarding.

The Frost Fitness PRODUCTIZATION-PLAN.md is the worked example of climbing this ladder; reuse its phase structure.

New-App Checklist (copy/paste for the next mobile project)

  • [ ] npx create-expo-app with expo-router + TypeScript; copy tsconfig, eslint, jest config from Frost Fitness.
  • [ ] Set app.json: name, slug, bundleIdentifier/package = com.<app>.app, owner: aspen-frost, icons, permission infoPlist strings, ITSAppUsesNonExemptEncryption: false.
  • [ ] Create EAS project (eas init), copy eas.json profiles, set ascAppId after App Store Connect app creation.
  • [ ] If AI: scaffold proxy/ from Frost Fitness's Worker; set model/token caps; set Worker secrets; wire client to EXPO_PUBLIC_PROXY_URL/_TOKEN.
  • [ ] Copy CI/CD workflows (test.yml, worker-ci.yml, worker-deploy.yml) + proxy/scripts/smoke.js; set GitHub Secrets + worker-production environment gate (Cassian).
  • [ ] App CI runs npx tsc --noEmit, npx expo lint, and npm test -- --ci on PRs.
  • [ ] Add React Native Testing Library coverage for the first golden path and all permission-denied states.
  • [ ] Add stable testIDs for controls/screens that will be driven by Maestro.
  • [ ] Add an iOS simulator EAS build profile and Maestro flows before claiming automated native iOS coverage.
  • [ ] Wire Sentry: DSN as EAS secret, root error boundary.
  • [ ] Set src/theme/tokens.ts as the single design-token source.
  • [ ] Publish privacy + terms to docs.aspenfrost.com/<app>/.
  • [ ] EAS secrets for proxy URL/token + Sentry DSN.
  • [ ] First development build → device; then preview; then production → TestFlight.
