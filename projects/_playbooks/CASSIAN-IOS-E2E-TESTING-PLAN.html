Cassian Plan: iOS-First Mobile Testing Upgrade

Owner: Cassian (SRE) | Requester: Andy | Prepared by: Jesse | Date: 2026-07-09 | Status: Ready for implementation planning

Goal

Move Aspen mobile apps from manual TestFlight-only confidence to a layered, iOS-first testing system. The immediate target is stronger PR confidence from Ubuntu-run checks, plus a clear path to real iOS Simulator E2E once macOS infrastructure is available.

Scope

Initial target repos:

  • frost-fitness
  • frost-rocks

Do not make Android the strategic focus. Android can remain useful as a cheap native signal, but the implementation plan should optimize for iOS confidence.

Current Baseline

frost-fitness:

  • Expo SDK 54 / React Native app.
  • EAS iOS build profiles exist for development, preview, and production.
  • Jest, jest-expo, and React Native Testing Library are installed.
  • 11 tests exist, including screen/component tests for camera, confirm meal, and error boundary behavior.
  • GitHub Actions test.yml currently runs npm test, but not typecheck or Expo lint.

frost-rocks:

  • Expo SDK 54 / React Native app.
  • EAS iOS build profiles exist for development, preview, and production.
  • Jest, jest-expo, and React Native Testing Library are installed.
  • 6 tests exist, mostly logic/database/export/enrichment coverage.
  • GitHub Actions test.yml runs npx tsc --noEmit, npx expo lint, and npm test -- --ci.

Current gaps across both repos:

  • No Maestro flows.
  • No Detox setup.
  • No Playwright setup.
  • No native iOS simulator automation.
  • No iOS simulator-specific EAS build profile.
  • No real-device automation.
  • No required golden-path E2E smoke suite.

Strategy

Use three layers:

  • Ubuntu-safe PR checks now: typecheck, lint, Jest/RNTL, proxy CI.
  • iOS-readiness now: stable testIDs, simulator build profiles, Maestro flow definitions.
  • Native iOS execution next: macOS runner executes Maestro against iOS Simulator.

Maestro is the default native E2E tool unless Arya identifies a hard requirement for Detox. Maestro is easier to read, easier to hand off, and good enough for black-box release smoke flows.

Phase 0: Confirm Repo Ownership and Branching

Tasks:

  • Confirm canonical repo locations/remotes for frost-fitness and frost-rocks.
  • Create one implementation branch per repo, for example sre/ios-testing-foundation.
  • Confirm which GitHub Actions runners are available and whether macOS minutes are acceptable.
  • Confirm whether Cassian should implement directly or open infra PRs for Arya/Jesse review.

Acceptance checks:

  • Branches exist.
  • Repo remotes and default branches are documented in the PR descriptions.
  • No production deploy workflows are touched without explicit Cassian approval.

Phase 1: Standardize Ubuntu-Safe PR Checks

Tasks for frost-fitness:

- npm ci - npx tsc --noEmit - npx expo lint - npm test -- --ci

  • Update .github/workflows/test.yml to run:
  • Add a typecheck script to package.json if useful for consistency.
  • Keep the workflow on ubuntu-latest.

Tasks for frost-rocks:

  • Verify current workflow already runs typecheck, Expo lint, and Jest.
  • Normalize script names if needed, but avoid churn if the workflow is already clear.

Acceptance checks:

  • PR CI fails on TypeScript errors.
  • PR CI fails on lint errors.
  • PR CI fails on Jest failures.
  • Local evidence is captured in the PR: commands run and results.

Phase 2: Strengthen React Native Testing Library Coverage

Tasks:

  • Define 5-10 golden paths per app with Arya/Product before writing broad tests.
  • Add RNTL tests for the highest-risk user flows first.
  • Prefer real local utilities in screen tests; mock only external service/device boundaries.
  • Add regression tests for runtime compatibility failures, especially partial/missing Expo/Hermes APIs.

Suggested frost-fitness golden paths:

  • First app load reaches the primary dashboard without crashing.
  • Camera permission denied shows a recoverable state.
  • Photo/library selection failure shows a recoverable state.
  • Confirm meal screen can save a valid meal.
  • Confirm meal screen handles missing/invalid meal data.
  • Fasting state can start/stop or render current state correctly.
  • Local database write/read path works for meal or nutrition targets.

Suggested frost-rocks golden paths:

  • First app load reaches the main experience without crashing.
  • New observation/capture path can save locally.
  • Location permission denied shows a recoverable state.
  • Photo/media permission denied shows a recoverable state if media is part of the flow.
  • Export/share flow handles success and failure states.
  • Enrichment queue behavior is covered for offline or deferred context.

Acceptance checks:

  • Each app has at least five golden-path or high-risk screen/service tests.
  • Permission-denied states are covered for every iOS permission declared in app.json.
  • Tests run in CI and locally with npm test -- --ci.

Phase 3: Add E2E Selectors Without Changing UX

Tasks:

- screen-dashboard - button-start-capture - button-save-meal - input-observation-note - state-permission-denied-camera

  • Add stable testID props to primary screens, navigation actions, form fields, submit buttons, permission recovery actions, and export/share actions.
  • Use predictable names, for example:
  • Avoid using visible text as the only future E2E selector.
  • Do not alter visual UI or accessibility labels unless Arya approves.

Acceptance checks:

  • Future Maestro flows can target stable IDs for the golden paths.
  • RNTL tests can also query critical elements reliably.
  • No UI copy changes are introduced just to satisfy tests.

Phase 4: Prepare Maestro, But Do Not Claim iOS Coverage Yet

Tasks:

  • Add Maestro as documented project tooling, likely with .maestro/ flows per repo.
  • Write initial flows for the top three golden paths per app.
  • Add README notes explaining that these flows require macOS/Xcode for iOS Simulator execution.
  • Keep flows black-box and product-level: launch app, navigate, tap/type, assert visible states.

Suggested flow files:

  • .maestro/launch-smoke.yml
  • .maestro/permission-denied-camera.yml
  • .maestro/save-primary-record.yml
  • .maestro/export-or-share.yml

Acceptance checks:

  • Maestro flows are syntactically valid.
  • Flow documentation says where they run and what app build they expect.
  • PR does not claim native iOS automation until a macOS run proves it.

Phase 5: Add iOS Simulator Build Profile

Tasks:

  • Add an EAS build profile for simulator artifacts, for example simulator or development-simulator.
  • Set ios.simulator: true for that profile.
  • Keep device/TestFlight profiles unchanged.
  • Document the command:
eas build --profile simulator --platform ios

Acceptance checks:

  • eas.json has a simulator-specific profile.
  • Existing development/preview/production profiles still target physical devices/TestFlight as before.
  • A simulator artifact can be produced by EAS Build, even if Ubuntu cannot run it.

Phase 6: Choose and Configure macOS Execution

Decision needed from Andy/Cassian:

  • Hosted GitHub Actions macOS runner: fastest to start, recurring minutes cost.
  • Bitrise/Codemagic: mobile-specialized CI, easier simulator ergonomics.
  • Self-hosted Mac mini: best long-term control, more setup/maintenance.

Cassian recommendation to prepare:

  • Start with hosted macOS CI or Bitrise/Codemagic for proof of value.
  • Move to self-hosted Mac mini only if mobile app volume justifies it.

Tasks after runner choice:

  • Install Xcode-compatible dependencies.
  • Install Maestro CLI.
  • Download or build the iOS simulator app artifact.
  • Boot a known simulator device, e.g. latest stable iPhone SE or standard iPhone model.
  • Run the top Maestro smoke flow.
  • Upload logs/screenshots/videos as artifacts.

Acceptance checks:

  • A CI job runs at least one Maestro flow against iOS Simulator.
  • CI artifacts include enough evidence to debug a failure.
  • The job is either PR-gated or scheduled nightly, with the cost tradeoff documented.

Phase 7: Release-Candidate Real Device Smoke Later

Do this only after simulator automation proves useful.

Tasks:

  • Evaluate BrowserStack, Sauce Labs, and AWS Device Farm for iOS real-device smoke tests.
  • Pick one provider only if device-specific failures are recurring or releases justify the cost.
  • Run a minimal suite against release candidates, not every PR.

Acceptance checks:

  • Release-candidate job runs on at least one real iPhone target.
  • Results are attached to the release/TestFlight handoff.
  • Simulator CI remains the primary automated native signal.

PR/Handoff Evidence Standard

Every Cassian PR for this work should include:

  • What changed.
  • Which repo/app it affects.
  • Commands run and results.
  • CI link or local output summary.
  • Whether native iOS coverage was actually executed.
  • If native iOS was not executed, say plainly: Not run: requires macOS/Xcode runner.

Definition of Done

Minimum done for the foundation PRs:

  • frost-fitness and frost-rocks both run typecheck, Expo lint, and Jest in PR CI.
  • Each repo has documented golden paths.
  • Each repo has stable testIDs for the first native E2E targets.
  • Each repo has an iOS simulator EAS build profile.
  • Each repo has initial Maestro flow files and documentation.
  • The mobile playbook reflects the iOS-first testing strategy.

Full done for native iOS E2E:

  • A macOS runner executes at least one Maestro flow against iOS Simulator.
  • Logs/screenshots/videos are saved as CI artifacts.
  • The result is linked from the PR/release handoff.
