Software Development and Testing Principles - Aspen Frost

> Companywide engineering doctrine for how Aspen Frost builds software. Project architecture docs can add stricter local rules, but should not weaken these principles.

Started 2026-06-06 by Arya after the Frost Fitness Expo/Hermes crypto.randomUUID production failure.

---

Why This Exists

Aspen Frost is building software with agent teams. That gives us speed, but speed without institutional memory turns every production bug into a recurring villain.

This document captures durable engineering lessons that should apply across projects, agents, and repos. It is not a process checklist. It is the judgment layer underneath the process.

When a project teaches us something broadly useful, capture it here and then add any project-specific version to that project's architecture, testing, or release docs.

---

Principle 1: Test the Runtime, Not the Fantasy

Jest, Node, local mocks, browser devtools, simulators, and CI containers are useful approximations. They are not automatically the runtime our users actually touch.

A test suite is incomplete when it proves behavior only in a convenient fake universe. For code that depends on runtime-provided APIs, tests must model the deployed runtime's capabilities, including partial availability and missing APIs.

What Triggered This

Frost Fitness hit a production failure after photo submission on 2026-06-06. The app reached Confirm Meal, then failed because crypto.randomUUID was available in the Node/Jest test environment but absent in the Expo/Hermes app runtime.

The bug survived because the suite tested two comfortable things:

  • The UID utility worked in Jest's Node-shaped runtime.
  • The Confirm Meal screen worked while mocking the UID utility entirely.

Neither test proved the real screen could call the real utility in the real app runtime.

The Rule

Any code touching platform, browser-shaped, device, filesystem, storage, navigation, or runtime-global APIs must have compatibility coverage that models the deployed runtime shape.

Examples include:

  • crypto, especially partial availability such as getRandomValues present while randomUUID is absent
  • filesystem/cache APIs and cleanup behavior
  • camera, media picker, image manipulation, and upload paths
  • local storage, SQLite, secure storage, and persistence adapters
  • route params, navigation lifecycle, background/foreground behavior
  • browser-shaped globals such as Blob, File, URL, atob, fetch, FormData, and streams
  • serverless edge runtimes, where Node APIs may be absent or subtly different

Required Testing Pattern

For platform-touching utilities, test more than the happy runtime:

  • Full API exists.
  • Partial API exists.
  • API is absent.
  • Fallback path behaves correctly.
  • Failure path fails clearly and recoverably.

For feature/screen tests:

  • Mock external services, network calls, secrets, permissions, and device-only integrations at the boundary.
  • Prefer real local utilities when they are deterministic and cheap to run.
  • Do not mock below the failure boundary unless the mock is the thing being tested.

For production bugs:

  • Every production screenshot, crash, or runtime error should become a regression test that recreates the exact missing capability or failing state.
  • The regression test should live as close as practical to the code that failed.
  • If the failure reveals a companywide lesson, update this document too.

PR Review Question

Every engineering PR should be able to answer:

> Does this rely on runtime APIs whose availability differs between local tests, CI, and production? If yes, where is the compatibility or regression coverage?

If the answer is "yes, but we did not test that shape," the PR is not ready. That is not bureaucracy. That is how we stop rewarding tests for being flattering.
