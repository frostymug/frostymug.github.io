eOS — Empire OS. Our own flavor of EOS (Entrepreneurial Operating System). This doc defines the long-term vision for the Aspen Frost agentic empire. Andy and Aspen only. For current org state, see Empire.

Vision

A full agentic product development pipeline — from idea to production — organized around three core functions: Design, Product, and Engineering. Andy acts as CPO, setting direction and staying in the loop until trust is established with each agent and flow.

This is simultaneously a real production system for building apps and an experiment in how agent teams should be structured.

Two Axes of Evolution

Axis 1 — Breadth

Stand up the three functions. Figure out how Andy works with each one.

  • What does a good session with each role look like?
  • What artifacts does each role produce?
  • How do handoffs between functions work?

Axis 2 — Depth (adversarial review)

Introduce productive tension within and between roles.

Architect vs. Engineer
Systems thinking vs. practical buildability
Designer vs. PM
User experience vs. scope and timeline
QA vs. Engineer
Quality bar vs. shipping pressure
Strategist vs. PM
Long-term vision vs. near-term roadmap

Human-in-the-Loop Trust Ramp

1
Andy + one agent at a time
Andy is the integration layer between functions. Building intuition for which decisions actually need him.
2
Agents hand off to each other
Andy reviews the handoffs, not the execution.
3
Adversarial pairs run autonomously
Andy reviews outcomes, not process.
4
Andy is CPO again
Setting direction and reviewing exceptions only. Full delegation.

Model Strategy

Different roles may use different models deliberately. A model optimized for code generation isn't necessarily best for strategic challenge or long-horizon architecture reasoning.

  • Strategist / thought partner — different reasoning style, slightly adversarial
  • Architect — strong at systems reasoning and constraint satisfaction
  • Engineer — strong at code generation and instruction-following
  • QA — strong at finding edge cases and failure modes

Checks & Balances

  • Adversarial review — QA and Engineer have explicitly different incentives
  • Cross-functional sign-off — Designer and PM must agree before Engineering starts
  • Red team agent — dedicated agent whose only job is finding problems with everyone else's output
  • Confidence routing — agents flag low-confidence decisions for human review
